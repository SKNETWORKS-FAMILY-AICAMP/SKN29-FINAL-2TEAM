## 스크럼 회의록

### 1. 도메인 변경

**변경 내용**

* 기존 디지털 마케팅(Digital Marketing) 도메인에서 개발 관리(Development) 도메인으로 변경

**변경 배경**

* 초기에는 시장 수요를 고려하여 디지털 마케팅 분야를 대상으로 선정
* 멘토링 및 팀 회의를 통해 기존 상용 플랫폼과의 차별성이 부족하다는 점을 확인
* 문제를 업무 분배 효율화와 특정 인원의 업무 과부하 방지로 구체화하면서, 팀의 경험과 도메인 이해도가 높은 개발 분야로 전환하기로 결정

**기대 효과**

* 도메인 전문성을 기반으로 보다 현실적인 시나리오와 기능 설계 가능
* 실제 업무 데이터를 활용한 검증 및 구현 용이

⸻

### 2. 플랫폼 방향 변경

**변경 내용**

* 기존 통합형 업무 플랫폼에서 커넥터(Connector) 기반 플랫폼으로 방향 수정

**변경 배경**

* 메일, 메신저, 일정 등을 모두 포함하는 신규 플랫폼은 기업 입장에서 기존 시스템을 교체해야 하는 부담이 큼
* 데이터 종속(Vendor Lock-in) 문제와 도입 비용 증가 우려 존재

**변경 방향**

* 기존 업무 환경은 그대로 유지
* 필요한 데이터만 API를 통해 연동하는 Connector First 구조로 변경
* 기업의 도입 부담을 최소화하고 기존 업무 프로세스를 유지하는 방향으로 설계

⸻

### 3. API 연동 및 데이터 분석

**연동 대상 선정**

프로젝트에서 활용할 핵심 업무 시스템을 선정하고 API 활용 가능 범위를 조사

**연동 대상**

* Google Drive / Google Docs : 프로젝트 문서 및 기획서
* Google Calendar : 일정 정보
* Jira : 프로젝트 및 태스크 관리
* Workday : HR(인사) 정보

**진행 내용**

* 각 API에서 조회(Input) 가능한 데이터 항목 조사
* 플랫폼에서 업데이트(Output) 가능한 데이터 범위 검토
* 서비스 시나리오에 필요한 데이터 활용 방안 정리

⸻

### 4. 시스템 아키텍처 재설계

**진행 내용**

* 변경된 서비스 시나리오에 맞춰 전체 시스템 아키텍처 재설계
* 커넥터 기반 데이터 흐름을 반영한 레이어 구조 설계

**목표**

* 커넥터를 통해 수집된 데이터를 효율적으로 처리
* AI 에이전트 간 데이터 전달 및 통신 구조 최적화
* 서비스 확장성과 유지보수성을 고려한 구조 설계

⸻

### 5. AI 에이전트 데이터 요구사항 정의

**진행 내용**

* 업무 분배 AI가 활용해야 하는 데이터와 판단 기준 구체화
* 에이전트별 입력 데이터와 활용 목적 정의

**주요 논의**

* 업무 분배에 필요한 핵심 데이터 선정
* 업무량 및 일정 판단에 필요한 지표 정의
* 담당자 추천을 위한 데이터 활용 기준 정리
* AI가 활용하는 데이터의 근거와 신뢰성 확보 방안 논의

⸻

### 금일 진행 결과

* 개발(Development) 도메인으로 변경
* 커넥터 기반 플랫폼 구조 확정
* Google Drive, Google Calendar, Jira, Workday API 활용 범위 조사
* 서비스 시나리오에 맞는 아키텍처 재설계
* AI 에이전트의 데이터 요구사항 및 판단 기준 정의
