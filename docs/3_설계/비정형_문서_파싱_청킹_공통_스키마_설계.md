# 비정형 문서 파싱·청킹 공통 스키마 설계

> 문서 상태: 설계 확정안  
> 작성일: 2026-07-22  
> 적용 대상: Google Drive·Docs, PDF, DOCX, PPTX, 스캔 문서  
> 목적: 파일 형식별 파서 출력을 공통 구조로 정규화하고, 전체 문서의 의미 정보와 관계를 Project Knowledge Model로 통합한 뒤 Task를 추출한다. Embedding·Vector 검색은 문서 간 유사 정보와 과거 Jira 근거를 찾는 보조 수단으로 사용한다.

---

## 1. 설계 배경

PDF, DOCX, PPTX, Google Docs는 문서 구조와 위치 표현 방식이 서로 다르다.

- PDF는 페이지와 좌표 중심이다.
- DOCX와 Google Docs는 제목·문단·표의 계층 중심이다.
- PPTX는 슬라이드와 도형·노트 중심이다.
- 스캔 문서는 OCR 결과와 좌표·인식 신뢰도가 필요하다.

파서의 원본 출력을 바로 청킹하면 파일 형식마다 다음 로직이 달라진다.

- 제목과 본문 구분
- 표 보존 방식
- 위치와 Citation 생성
- 청크 경계 결정
- 변경 청크 비교
- Vector DB 필터 메타데이터
- RAG 검색 결과를 원문으로 되돌리는 방식

따라서 파서와 청커 사이에 **Canonical Document Schema**를 두고 모든 형식의 출력을 같은 `ContentBlock` 구조로 변환한다.

---

## 2. 핵심 설계 결정

1. 하나의 거대한 스키마로 모든 단계를 처리하지 않고 단계별 계약을 분리한다.
2. 파서의 공통 출력 단위는 `ParsedDocument`와 `ContentBlock`이다.
3. 청커의 출력 단위는 `ChunkRecord`다.
4. Vector DB 저장 단위는 `VectorIndexRecord`다.
5. 문서 한 revision의 처리 상태는 `DocumentIndex`가 불변 Manifest 역할을 한다.
6. 파일 형식별 위치 차이는 공통 `SourceLocator`와 `extensions`로 흡수한다.
7. 검색용 텍스트와 표·목록의 구조화 데이터는 분리해 함께 보존한다.
8. ACL, tenant, project, 보안 등급은 파싱 이후 모든 단계에서 상속한다.
9. Citation은 문서·revision·청크뿐 아니라 원본 블록과 위치까지 추적한다.
10. 청크 내용이 같아도 원본 revision과 위치가 다르면 별도의 Index 레코드를 만들되 Embedding 값은 재사용할 수 있다.
11. Task 추출의 주 입력은 Top-K 검색 결과가 아니라 프로젝트 범위의 `ProjectKnowledgeModel`이다.
12. Vector/RAG는 유사 표현 통합, 과거 Jira 업무, 추가 근거 검색을 위한 보조 경로이며 전체 범위의 누락 판정에 사용하지 않는다.

## 3. 전체 데이터 흐름

```text
Google Drive·Docs
        ↓
SourceDocumentRecord
        ↓ 파일 다운로드·원본 저장
Object Storage
        ↓ 형식별 Parser / OCR
ParsedDocument
        ↓
ContentBlock[]
        ├─ 전체 Block 의미 분류·구조화 추출
        ↓
KnowledgeItem[]
        ↓ 중복 통합·상충/최신 버전 판정·관계 구성
FeatureCluster[]
        ↓
ProjectKnowledgeModel
        ↓ LLM 업무 구조화 추출
NewTaskDraft
        ↓ PM 확인
NewTaskSnapshot

ContentBlock[]
        ↓ 공통 Chunking Policy (보조 검색 경로)
ChunkRecord[]
        ↓ Embedding
VectorIndexRecord[]
        ↓
Vector DB + DocumentIndex
        ↓ ACL·Project 필터 + Hybrid Search
RagEvidence[]
        └─ ProjectKnowledgeModel 통합·업무 근거 보완
```

### 3.1 계층별 책임

| 계층 | 입력 | 출력 | 책임 |
|---|---|---|---|
| Source Registry | Drive 폴더·파일 설정 | `SourceDocumentRecord` | 수집 범위·문서 역할·ACL·revision 확인 |
| Downloader | SourceDocumentRecord | 원본 파일 URI | 다운로드·무결성 확인·Object Storage 저장 |
| Parser Adapter | 원본 파일 | Parser별 Raw 결과 | PDF·DOCX·PPTX·Google Docs 해석 |
| Normalizer | Parser Raw 결과 | `ParsedDocument`, `ContentBlock[]` | 형식별 구조를 공통 블록으로 변환 |
| Semantic Extractor | ContentBlock[] 전체 | `KnowledgeItem[]` | 목표·범위·요구사항·결정·제약·일정·리스크·완료조건·담당 영역 구조화 |
| Knowledge Integrator | KnowledgeItem[] | `FeatureCluster[]`, `ProjectKnowledgeModel` | 중복 통합·상충 판정·전역 관계·원문 계보 구성 |
| Chunker | ContentBlock[] | `ChunkRecord[]` | 의미·구조 기반 청크 생성 |
| Embedder | ChunkRecord | Embedding | 검색용 벡터 생성 |
| Index Writer | Chunk+Embedding | `VectorIndexRecord`, `DocumentIndex` | Vector DB 저장·활성 revision 관리 |
| Retriever | 보완 검색 요청 | `RagEvidence[]` | ACL 필터·Hybrid Search·재순위화·Citation 구성 |
| Task Extractor | ProjectKnowledgeModel + RagEvidence[] 선택 | NewTaskDraft | 전체 프로젝트 지식 기준으로 업무·역할·일정·공수·근거 구조화 |

## 4. 공통 스키마 객체

```text
SourceDocumentRecord  1 ─── N  DocumentRevision
DocumentRevision      1 ─── N  ParsedDocument
ParsedDocument        1 ─── N  ContentBlock
ParsedDocument        1 ─── N  DocumentIndex
DocumentIndex         1 ─── N  ChunkRecord
ContentBlock          N ─── M  ChunkRecord (ChunkBlock 연결)
ChunkRecord           1 ─── N  VectorIndexRecord
ChunkRecord           1 ─── N  CitationRef
RAG 검색 결과         1 ─── N  RagEvidence
ContentBlock          1 ─── N  KnowledgeItem
KnowledgeItem         N ─── M  FeatureCluster
FeatureCluster        N ─── 1  ProjectKnowledgeModel
ProjectKnowledgeModel 1 ─── N  NewTaskDraft
```

| 객체 | 역할 | 저장 위치 |
|---|---|---|
| `SourceDocumentRecord` | 원천 파일·revision·권한·수집 범위 | PostgreSQL |
| `ParsedDocument` | 파싱 결과의 문서 단위 Envelope | PostgreSQL/Object Storage |
| `ContentBlock` | 형식과 무관한 최소 구조 블록 | PostgreSQL/Object Storage |
| `ChunkBlock` | 블록과 청크의 N:M 연결·순서·중첩 구분 | PostgreSQL |
| `ChunkRecord` | Embedding과 검색의 기본 단위 | PostgreSQL/Vector DB Metadata |
| `VectorIndexRecord` | Vector DB 저장 계약 | Vector DB |
| `DocumentIndex` | revision별 처리 결과와 활성 상태 Manifest | PostgreSQL |
| `CitationRef` | 청크·원문 위치 추적 | PostgreSQL/결과 객체 |
| `KnowledgeItem` | 원문에서 구조화한 목표·요구사항·결정·제약·일정·리스크·완료조건·담당 영역 | PostgreSQL |
| `FeatureCluster` | 여러 문서의 동일 기능·요구사항을 통합한 단위 | PostgreSQL |
| `ProjectKnowledgeModel` | 프로젝트 전체 범위·관계·근거를 고정한 Task 추출 기준 | PostgreSQL/Snapshot |
| `RagEvidence` | 유사 정보·과거 업무·추가 근거를 찾은 보조 검색 결과 | 실행 상태/결과 저장소 |

대용량 `ContentBlock` 본문과 파서 Raw 결과는 Object Storage에 JSON 또는 Parquet 형태로 보관할 수 있다. PostgreSQL에는 검색·감사·상태 관리에 필요한 식별자와 Manifest를 저장한다.

## 5. 공통 Envelope

모든 객체는 다음 필드를 공통으로 가진다.

| 필드 | 타입 | 필수성 | 설명 | 검증 규칙 |
|---|---|---:|---|---|
| `schemaName` | string | 필수 | 객체 스키마 이름 | 허용된 스키마 이름 |
| `schemaVersion` | string | 필수 | 객체 스키마 버전 | Semantic Version 권장 |
| `tenantId` | string | 필수 | 고객 데이터 경계 | 요청 tenant와 일치 |
| `documentId` | string | 필수 | 내부 문서 고유 ID | Tenant 내 중복 불가 |
| `sourceSystem` | enum | 필수 | 원천 시스템 | `GOOGLE_DRIVE/MANUAL_UPLOAD` 등 |
| `sourceFileId` | string | 필수 | Drive fileId 또는 업로드 ID | 원천 내 중복 불가 |
| `revisionId` | string | 필수 | 원천 파일 revision | 동일 파일의 revision 식별 가능 |
| `projectId` | string | 필수 | 프로젝트 필터 | 요청 프로젝트와 일치 |
| `documentRole` | enum | 필수 | 문서 역할 | 허용된 역할 코드 |
| `securityLevel` | enum | 필수 | 문서 보안 등급 | Tenant 정책에 존재 |
| `aclPrincipals` | array | 필수 | 접근 가능한 사용자·그룹 | 빈 배열이면 검색 차단 |
| `sourceModifiedAt` | datetime | 필수 | 원천 수정 시각 | 미래 시각 금지 |
| `createdAt` | datetime | 필수 | 객체 생성 시각 | ISO 8601 |
| `traceId` | string | 필수 | 처리 실행 추적 ID | 실행 로그와 연결 |

`tenantId`, `projectId`, `securityLevel`, `aclPrincipals`는 Vector 검색 전에 필터링해야 하므로 자유형 `metadata` 안에만 넣지 않고 명시적 필드로 둔다.

## 6. SourceDocumentRecord

원천 문서의 수집 대상 여부, revision, 권한과 원본 보관 위치를 관리한다.

| 필드 | 타입 | 필수성 | 설명 |
|---|---|---:|---|
| `documentId` | string | 필수 | 플랫폼 내부 문서 ID |
| `sourceFileId` | string | 필수 | Google Drive fileId |
| `revisionId` | string | 필수 | Drive revision 또는 수정 버전 |
| `parentFolderIds` | array | 필수 | 수집 대상 폴더 경로 |
| `fileName` | string | 필수 | 원천 파일명 |
| `mimeType` | string | 필수 | 원천 MIME 타입 |
| `fileSizeBytes` | integer/null | 선택 | 파일 크기 |
| `contentChecksum` | string | 필수 | 다운로드 원본 해시 |
| `ownerIds` | array | 선택 | 원천 소유자 ID |
| `documentRole` | enum | 필수 | 기획서·요구사항·명세서 등 |
| `projectId` | string | 필수 | 연결 프로젝트 |
| `departmentId` | string/null | 선택 | 관련 조직 |
| `securityLevel` | enum | 필수 | 공개 범위·보안 등급 |
| `aclPrincipals` | array | 필수 | 사용자·그룹·도메인 ACL |
| `webViewLink` | string/null | 선택 | 원문 보기 링크 |
| `objectStorageUri` | string | 필수 | 원본 보관 위치 |
| `sourceCreatedAt` | datetime | 선택 | 원천 생성 시각 |
| `sourceModifiedAt` | datetime | 필수 | 원천 수정 시각 |
| `discoveredAt` | datetime | 필수 | 변경 감지 시각 |
| `deleted` | boolean | 필수 | 원천 삭제 여부 |
| `accessRevoked` | boolean | 필수 | 접근 권한 상실 여부 |

Google Drive가 안정적인 revision ID를 제공하는 형식은 원천 revision을 사용한다. Google Docs export처럼 동일한 revision 식별자를 얻기 어려운 경우에는 `modifiedTime + export contentChecksum`으로 내부 `revisionId`를 생성하고 생성 방식을 함께 기록한다.

객체 계약의 `SourceDocumentRecord`는 처리 편의를 위해 revision 정보를 함께 전달한다. 관계형 DB에서는 변하지 않는 파일 정보는 `source_document`, revision·checksum·원본 URI·수정시각은 `document_revision`으로 분리해 1:N으로 저장한다.

### 6.1 documentRole 권장 값

```text
PROJECT_PLAN
REQUIREMENTS
TECHNICAL_SPEC
MEETING_MINUTES
RESUME_PROFILE
SCHEDULE
RISK_REGISTER
REFERENCE
OTHER
```

문서 역할은 Parser가 추측해 자동 확정하지 않는다. Source Registry 설정을 우선하고, 미설정 시 AI 분류 결과를 후보값으로 제공해 사용자 확인을 받는다.

## 7. ParsedDocument

`ParsedDocument`는 하나의 파일 revision을 공통 블록 구조로 변환한 결과의 Envelope다.

| 필드 | 타입 | 필수성 | 설명 | 검증 규칙 |
|---|---|---:|---|---|
| `parsedDocumentId` | string | 필수 | 파싱 결과 ID | revision·parserVersion 조합 내 중복 불가 |
| `parserType` | enum | 필수 | 사용 Parser | `PDF/DOCX/PPTX/GOOGLE_DOCS/OCR` |
| `parserVersion` | string | 필수 | Parser 버전 | 불변 버전 |
| `normalizerVersion` | string | 필수 | Parser Raw 결과를 공통 블록으로 변환한 규칙 버전 | 불변 버전 |
| `parseStatus` | enum | 필수 | 파싱 상태 | `SUCCESS/PARTIAL_RESULT/BLOCKED` |
| `detectedLanguage` | string/null | 선택 | 주 언어 | BCP 47 코드 권장 |
| `pageCount` | integer/null | 선택 | PDF·스캔 페이지 수 | 0 이상 |
| `slideCount` | integer/null | 선택 | PPTX 슬라이드 수 | 0 이상 |
| `blockCount` | integer | 필수 | 생성 블록 수 | 실제 블록 수와 일치 |
| `textLength` | integer | 필수 | 정규화 텍스트 길이 | 0 이상 |
| `documentTextHash` | string | 필수 | 전체 정규화 텍스트 해시 | sha256 권장 |
| `structureUri` | string | 필수 | ContentBlock JSON 저장 위치 | Object Storage 접근 가능 |
| `rawParserResultUri` | string/null | 선택 | Parser 원본 결과 | 보존 정책 적용 |
| `warnings` | array | 필수 | 파싱 경고 | 없으면 빈 배열 |
| `errors` | array | 필수 | 파싱 오류 | 없으면 빈 배열 |
| `parsedAt` | datetime | 필수 | 파싱 완료 시각 | sourceModifiedAt 이후 |

### 7.1 parseStatus 판정

| 조건 | 상태 |
|---|---|
| 본문과 위치 정보가 정상 생성됨 | `SUCCESS` |
| 일부 페이지·표·이미지 또는 낮은 OCR 신뢰도 구간이 존재함 | `PARTIAL_RESULT` |
| 지원하지 않는 형식, 암호화, 다운로드 실패, 본문 0자 | `BLOCKED` |

## 8. ContentBlock 공통 스키마

`ContentBlock`은 문서의 의미와 구조를 보존하는 최소 단위다. 청크 크기와는 다르며 하나 이상의 블록이 하나의 청크가 될 수 있다.

| 필드 | 타입 | 필수성 | 설명 | 검증 규칙 |
|---|---|---:|---|---|
| `blockId` | string | 필수 | revision 내 블록 ID | 중복 불가 |
| `parentBlockId` | string/null | 선택 | 상위 Section·표·목록 | 존재하는 blockId 참조 |
| `sequence` | integer | 필수 | 문서 읽기 순서 | 0 이상, 중복 금지 |
| `blockType` | enum | 필수 | 블록 유형 | 허용값 확인 |
| `text` | string | 필수 | 검색·청킹 가능한 정규화 텍스트 | 공백 블록은 제외 또는 명시적 구조 블록 처리 |
| `structuredContent` | object/null | 선택 | 표·목록·키값 구조 | blockType별 Schema 검증 |
| `headingLevel` | integer/null | 선택 | 제목 깊이 | 제목일 때 1 이상 |
| `headingPath` | array | 필수 | 상위 제목 경로 | 없으면 빈 배열 |
| `listLevel` | integer/null | 선택 | 목록 깊이 | 목록일 때 0 이상 |
| `sourceLocator` | object | 필수 | 원본 위치 | 형식별 필수 위치 존재 |
| `extractionMethod` | enum | 필수 | 추출 방식 | `NATIVE/OCR/AI_VISION/MANUAL` |
| `extractionConfidence` | decimal/null | 조건부 | OCR·Vision 신뢰도 | 0~1 |
| `language` | string/null | 선택 | 블록 언어 | BCP 47 권장 |
| `contentHash` | string | 필수 | 블록 내용 해시 | 정규화 규칙 버전 포함 |
| `excludeFromEmbedding` | boolean | 필수 | Embedding 제외 여부 | 제외 사유와 일치 |
| `excludeReason` | string/null | 선택 | 제외 사유 | 제외 시 필수 |
| `extensions` | object | 선택 | 형식별 보조 데이터 | Namespace 사용 |

### 8.2 Embedding 제외 기본 규칙

다음 블록은 원문 보존은 하되 기본적으로 Embedding 대상에서 제외한다.

- 여러 페이지에 동일하게 반복되는 Header·Footer
- 빈 문단과 장식용 문자열
- 페이지 번호만 존재하는 블록
- 목차의 점선·리더 문자만 있는 블록
- OCR 신뢰도가 Tenant 임계값보다 낮고 사람이 확인하지 않은 블록
- 접근 권한 또는 보안 정책상 인덱싱할 수 없는 블록

## 10. 표와 목록 공통 표현

### 10.1 표

표는 검색용 텍스트와 구조화 데이터를 함께 보존한다.

```json
{
  "blockType": "TABLE",
  "text": "담당 역할 | 시작일 | 종료일\n백엔드 | 2026-08-03 | 2026-08-14",
  "structuredContent": {
    "type": "TABLE",
    "headers": ["담당 역할", "시작일", "종료일"],
    "rows": [
      ["백엔드", "2026-08-03", "2026-08-14"]
    ],
    "mergedCells": [],
    "headerRowCount": 1
  }
}
```

큰 표는 행 묶음 단위로 여러 청크로 나누되 모든 청크에 표 제목과 헤더를 반복한다.

### 10.2 목록

```json
  "blockType": "LIST_ITEM",
  "text": "Jira 기존 업무의 잔여 공수를 수집한다.",
  "listLevel": 1,
  "parentBlockId": "BLOCK-LIST-01",
  "headingPath": ["개발 범위", "데이터 연동"]
}
```

## 13. 공통 청킹 규칙

### 13.1 기본 정책

- 기본 목표 크기: 500~800 토큰
- 기본 중첩: 80~120 토큰
- 제목·문단·표·슬라이드 경계를 토큰 수보다 우선한다.
- 서로 다른 `documentId`, `revisionId`, ACL, 보안 등급의 블록을 하나의 청크로 합치지 않는다.
- 제목 경로는 본문 앞에 검색용 Context로 추가할 수 있지만 원문과 구분해 저장한다.
- 요구사항 번호, 일정, 역할, 공수 단위를 자르는 분할을 피한다.

이 값은 초기 기본값이며 문서 역할별 `ChunkingPolicy`로 버전 관리한다.

### 13.2 블록 유형별 규칙

| 블록 유형 | 청킹 규칙 |
|---|---|
| TITLE/HEADING | 바로 다음 본문과 함께 포함, 제목 단독 청크 최소화 |
| PARAGRAPH | 문단 경계 우선, 너무 길면 문장 단위 분할 |
| LIST_ITEM | 동일 상위 목록과 함께 묶고 listLevel 유지 |
| TABLE | 작은 표는 전체 보존, 큰 표는 헤더+행 그룹으로 분할 |
| CODE_BLOCK | 가능한 한 코드 블록 전체 보존, 너무 크면 함수·구문 경계 분할 |
| PPTX | 기본적으로 슬라이드 단위, 제목·본문·노트를 함께 고려 |
| OCR | 낮은 신뢰도 영역이 다른 정상 블록의 신뢰도를 오염시키지 않도록 분리 |

### 13.3 청크 생성 의사코드

```text
for each ParsedDocument revision:
    blocks = sortBySequence(valid ContentBlocks)
    current = new chunk

    for block in blocks:
        if block.excludeFromEmbedding:
            continue

        if securityOrAclChanged(current, block):
            flush(current)

        if isHardBoundary(block):
            flush(current)

        if wouldBreakTableOrRequirement(block):
            applyTypeSpecificChunking(block)
        else if tokenLimitExceeded(current, block):
            flushWithOverlap(current)

        append(current, block)

    flush(current)
    validateChunkCoverage(blocks, chunks)
```

### 13.4 청크 검증

- 모든 Embedding 대상 블록은 최소 하나의 청크에 포함되어야 한다.
- `blockIds` 순서는 원문 `sequence` 순서와 같아야 한다.
- 한 청크 안에서 제목 경로가 충돌하면 청크를 분리한다.
- 중첩 구간을 제외한 본문이 빈 청크는 저장하지 않는다.
- 표 헤더만 있고 데이터 행이 없는 청크는 만들지 않는다.
- 최대 토큰을 초과한 청크는 `qualityFlags`에 사유를 남기고 재분할한다.

| `documentRole` | enum | 필수 | 문서 역할 |
| `accessDecision` | enum | 필수 | 접근 허용 결과 |
| `retrievedAt` | datetime | 필수 | 검색 시각 |

RAG 절차는 다음 순서를 따른다.

```text
검색 질의 생성
→ tenant·project·security·ACL 선필터
→ Vector 검색
→ 키워드·요구사항 ID 검색 결합
→ 중복 제거
→ 재순위화
→ 인접 청크 구성
→ Citation 생성
→ RagEvidence
→ Project Knowledge Model 통합 또는 Task 근거 보완
```

## 19. 증분 인덱싱 규칙

### 19.1 변경 감지

```text
Drive Changes API pageToken
→ fileId·revision·modifiedAt·ACL 비교
→ 신규 / 수정 / 삭제 / 권한 상실 분류
```

### 19.2 신규 파일

1. 새 `documentId`를 생성한다.
2. `SourceDocumentRecord`를 저장한다.
3. 원본을 Object Storage에 저장한다.
4. 전체 파싱·정규화·청킹·Embedding을 수행한다.
5. `DocumentIndex.isActiveRevision=true`로 활성화한다.

### 19.3 수정 파일

1. 같은 `documentId`에 새 `revisionId`를 생성한다.
2. 새 revision을 파싱하고 ContentBlock·ChunkRecord를 생성한다.
3. `contentHash + embeddingVersion`으로 Embedding 캐시를 조회한다.
4. 변경된 청크만 새 Embedding을 생성한다.
5. 새 revision의 모든 필수 청크가 준비된 후 활성 revision을 전환한다.
6. 이전 revision은 `isActive=false`로 보존한다.

새 revision이 완성되기 전에 이전 활성 revision을 제거하지 않는다. 부분 인덱싱 상태가 검색 결과에 섞이는 것을 방지한다.

### 19.4 삭제·권한 상실

| 변경 | 처리 |
|---|---|
| 원천 삭제 | `DELETED`, Vector 레코드 비활성화, 보존기간 후 삭제 |
| 권한 상실 | `ACCESS_REVOKED`, 즉시 검색 제외 |
| ACL 변경 | Vector Metadata 갱신 또는 새 Index 레코드 생성 |
| 폴더 수집 범위 제외 | 비활성화 후 정책에 따라 보존·삭제 |

## 20. 파싱·청킹 Feature Readiness

### 20.1 단계별 필수 데이터

| 기능 | 필수 데이터 | 상태 판정 |
|---|---|---|
| 다운로드 | sourceFileId, revisionId, MIME, 읽기 권한 | 누락 시 `BLOCKED` |
| 원본 보존 | objectStorageUri, checksum | 실패 시 파싱 `BLOCKED` |
| 파싱 | 지원 MIME, Parser, 원본 파일 | 일부 페이지 실패는 `PARTIAL_RESULT` |
| 정규화 | blockId, sequence, blockType, text, locator | 위치 없는 본문은 Citation 기능 제한 |
| 청킹 | 유효 ContentBlock, ChunkingPolicy | 유효 블록 0개면 `BLOCKED` |
| Embedding | chunkText, contentHash, 모델·버전 | 일부 실패는 문서 역할과 실패 비율에 따라 부분 결과 |
| Vector 저장 | vector, metadata, ACL, dimension | ACL 누락은 `BLOCKED` |
| RAG 검색 | 활성 revision, 접근 권한, 검색 가능한 청크 | 검색 결과 0건은 빈 결과, 검색 실패와 구분 |
| 의미 추출·지식 모델 구성 | 전체 활성 ContentBlock, KnowledgeItem Schema, Citation | 핵심 문서 미처리 또는 Citation 없는 항목은 `BLOCKED/PARTIAL_RESULT` |
| Task 추출 | ProjectKnowledgeModel, Citation, Task Schema | 지식 모델 또는 Citation 없는 추출은 `BLOCKED` |

### 20.2 품질 기준 제안

| 지표 | 초기 기준 | 처리 |
|---|---:|---|
| 본문 추출 문자 수 | 1자 이상 | 0자면 `BLOCKED` |
| Embedding 완료율 | 100% 권장 | 핵심 청크 실패 시 `PARTIAL_RESULT/BLOCKED` |
| 블록→청크 포함률 | 100% | 누락 시 `BLOCKED` |
| Citation 위치 연결률 | 100% | 누락 필드 추출 금지 |
| OCR 평균 신뢰도 | Tenant 설정 | 낮은 구간 분리·경고 |
| ACL 포함률 | 100% | 누락 시 검색 금지 |

단순 실패 비율만으로 상태를 결정하지 않는다. 요구사항·일정·역할 등 Task 추출에 중요한 핵심 청크가 실패했는지도 함께 판단한다.

## 22. 데이터베이스 저장 구조 제안

### 22.1 PostgreSQL

> 본 문서의 물리 테이블명이 문서 처리 계층의 **기준(SoT)** 이다. Canonical ER Catalog의 Entity(DOCUMENT_BLOCK 등)는 논리 뷰이며, 매핑은 ER Catalog 2.1을 따른다.

```text
source_document
```

`content_block_manifest`에는 블록 ID·타입·순서·위치·해시를 저장하고, 전체 본문 JSON은 Object Storage URI로 분리할 수 있다.

```text
/tenant/{tenantId}/document/{documentId}/revision/{revisionId}/
  original/{fileName}
  parsed/raw-parser-result.json
  parsed/content-blocks.json
  chunks/chunks.json
```

```text
id: vectorRecordId
```

| 변경 | 필요한 처리 |
|---|---|
| Parser 버전 변경 | 재파싱·재청킹·재임베딩 검토 |
| Normalizer 버전 변경 | 재정규화 이후 영향 블록 재청킹 |
| Chunker/Policy 변경 | 재청킹·재임베딩 |
| Embedding 모델 변경 | 기존 청크 재임베딩 |
| Metadata 필터 필드 변경 | Vector Metadata 마이그레이션 |
| ACL 변경 | 즉시 검색 권한 갱신, 본문 재임베딩 불필요 |

기존 추천과 Citation은 당시 `DocumentIndex`와 버전을 계속 참조한다. 새 버전의 인덱스가 생겼다고 과거 근거를 덮어쓰지 않는다.

## 25. 보안·개인정보 규칙

- ACL 확인 전 Embedding 검색 결과를 반환하지 않는다.
- ACL은 원본 Drive 권한과 프로젝트 권한의 교집합으로 계산한다.
- 보안 등급이 다른 블록을 한 청크로 합치지 않는다.
- 문서 원문과 파서 Raw 결과는 보존기간 정책을 적용한다.
- Vector DB Metadata에 불필요한 개인정보를 저장하지 않는다.
- 이력서·프로필 문서는 별도 `documentRole`과 접근 정책을 적용한다.
- 로그에는 전체 `chunkText`를 기본 기록하지 않고 ID·해시·처리 상태를 남긴다.
- 권한 상실 이벤트는 재임베딩보다 검색 차단을 우선 실행한다.
