# AI 프로젝트 운영 코파일럿 업무 분배 최종안

> 적용 도메인: 개발 프로젝트  
> 목적: 내부 People DB·Jira·프로젝트 문서 데이터를 이용해 담당자를 추천하고 PM이 최종 결정한다.  
> 기준 문서: 기획서 v5의 데이터 소스·부하 계산·단계별 구현 범위를 따른다.

---

## 1. 서비스 정의

> 내부 People DB의 인력정보와 Jira의 프로젝트 현황을 연결해 신규 업무의 담당자와 대안을 근거와 함께 추천하는 AI 프로젝트 운영 코파일럿이다.

핵심 원칙:

- AI는 담당자를 자동 확정하지 않는다.
- 원본에 없는 Skill·경력·가용시간을 만들지 않는다.
- 필수 데이터가 없으면 관련 결과를 출력하지 않는다.
- 선택 데이터가 부족하면 제한 사항과 낮은 신뢰도를 표시한다.
- 검증을 통과한 결과만 PM에게 전달한다.

```text
데이터 연결 → 업무·부하 분석 → 후보 추천 → 검증 → PM 승인
```

## 2. 시스템별 역할

| 시스템 | 원본 데이터 | 사용 목적 |
|---|---|---|
| 내부 People DB | 구성원, 조직, 직무, Skill, 근무시간, FTE, 휴가·휴직 | 자격·적합도·가용성 판단 |
| Jira | 기존 프로젝트, Task, 담당자, 상태, 일정, 공수, 완료 이력 | 현재 부하와 과거 경험 판단 |
| 기획서 | 신규 프로젝트 목적, 요구사항, 기능, 일정 | 신규 Task 생성 |
| 우리 플랫폼 | 부하율, 추천, 근거, 검증, 승인 이력 | 분석과 의사결정 지원 |

1차 구현의 조직·인력 데이터는 PostgreSQL People DB에서 관리한다. People Repository가 이를 Canonical 객체와 Snapshot으로 변환하며, 실제 고객사 HR Connector는 후속 확장으로 분리한다.

```text
내부 People DB 테이블·회사별 Jira 필드
→ People Repository·Jira Connector Adapter
→ PersonSnapshot·TaskSnapshot
```

## 3. 사용자 흐름

### 3.1 Jira 최초 연결

우리 플랫폼을 처음 사용하는 것과 Jira를 처음 사용하는 것은 다르다. 따라서 연결 직후 Jira 데이터 상태를 먼저 확인한다.

```text
Jira 연결
→ 접근 가능한 프로젝트 선택
→ 필드·사용자·이슈 현황 진단
→ 내부 HR 구성원과 Jira 계정 매핑
→ 분석 모드 결정
→ 초기 동기화
```

| 분석 모드 | Jira 상태 | 활용 방식 |
|---|---|---|
| 이력 보유 | 활성 업무와 완료 이력 존재 | 현재 부하 + 유사 업무 경험 |
| 현재 업무 | 활성 업무만 존재 | 현재 부하 중심 |
| 신규 프로젝트 | 관련 프로젝트·Task 없음 | 내부 People DB + 프로젝트 문서 중심 |

최초 진단 항목:

- 활성·완료 이슈와 이력 기간
- 담당자·상태·시작일·마감일
- 예상·잔여 공수 사용 여부
- 내부 HR–Jira 사용자 매핑률
- 마지막 업데이트 시각

### 3.2 신규 프로젝트 등록

신규 프로젝트는 Jira에 없으므로 기획서에서 시작한다.

```text
신규 프로젝트 등록·기획서 연결
→ Task·역할·일정 추출
→ 기존 Jira에서 팀원별 현재 업무 조회
→ 내부 People DB에서 사람·조직·가용성 조회
→ 신규 Task 반영 후 예상 부하 계산
→ 담당자·대안 추천 및 검증
→ PM 승인과 DecisionRecord 저장
→ 후속 단계에서 Jira에 Epic·Task 반영
```

승인 전에는 내부 Draft로만 관리한다. 1차 구현에서는 PM의 결정 기록까지 저장하며, Jira 이슈와 프로젝트 생성은 후속 확장 기능으로 분리한다.

## 4. 업무 분배 판단 구조

기존 A~F의 판단 기준은 유지하고 실제 구현은 4개 모듈로 묶는다.

| 기존 판단 | 실행 모듈 | 주요 판단 |
|---|---|---|
| A. 수행 자격 | 1. 필수 제약조건 검사 | 역할·Skill·권한·부재 |
| B. 업무 적합성 | 2. 업무 적합도 평가 | 직무·Skill·유사 경험 |
| C. 업무량·부재 | 3. 기간별 부하 계산 | 기존 업무·휴가·가용시간 |
| D. 배정 가능성 | 3. 기간별 부하 계산 | 신규 Task 반영 후 수행 가능성 |
| E. 프로젝트 충돌 | 4. 추천 결과 검증 | 일정·프로젝트 충돌 경고 |
| F. 집중·병목 | 4. 추천 결과 검증 | 과부하·특정인 집중 검사 |
| G. 최종 결정 | PM 승인 | 승인·대안 선택·반려 |

### 4.1 필수 제약조건

- 필수 역할·Skill·자격·권한
- 조직 제한이 있는 경우 소속 조건
- 프로젝트 기간의 휴가·휴직 중복

```text
MATCH: 충족 근거 있음
MISMATCH: 불충족 근거 있음 → 후보 제외
UNKNOWN: 데이터 부족 → 조건부 후보, PM 확인
```

데이터가 없다는 이유만으로 Skill이 없다고 판단하지 않는다.

### 4.2 적합도와 부하

적합도는 역할·Skill·유사 업무 이력을 비교하고 원본 근거를 함께 반환한다.

부하는 기존 업무, 잔여 공수, 마감일, 휴가, 신규 Task 기간과 공수를 일·주 단위로 계산한다.

### 4.3 검증

- 휴가와 배정 기간이 겹치지 않는가?
- 가용시간보다 업무량이 크지 않은가?
- 필수조건 위반 후보가 추천되지 않았는가?
- 추천 근거가 원본에 존재하는가?
- 특정인에게 업무가 과도하게 집중되지 않았는가?
- 프로젝트 일정이 충돌하지 않는가?

```text
PASS: 필수 위반 없음
CONDITIONAL_PASS: 데이터 부족 또는 경고 존재
REJECT: 필수조건 위반 또는 근거 오류
```

`REJECT`는 PM 승인과 Jira 반영 단계로 전달하지 않는다.

## 5. 필수·선택 데이터

| 구분 | 의미 | 누락 시 처리 |
|---|---|---|
| 필수 | 핵심 추천에 반드시 필요 | 관련 결과 출력 중단 |
| 조건부 필수 | 특정 계산·기능에 필요 | 해당 기능 중단 또는 대체 모드 |
| 선택 | 추천 근거 보강 | 결과 출력, 신뢰도 하향 |

결과 상태:

```text
SUCCESS: 필수 데이터 충족
PARTIAL_RESULT: 선택 데이터 일부 누락
BLOCKED: 필수 데이터 누락
```

### 5.1 시스템별 분류

| 시스템 | 필수 | 조건부 필수 | 선택 |
|---|---|---|---|
| 내부 People DB | 구성원 ID, 이름, 조직, 직무, 재직 상태, Jira 매핑 정보 | Skill과 숙련도, 근무시간·FTE, 휴가·휴직 기간, 위치·시간대 | 직급, Job Profile, 근속연수, 자격증 |
| 기존 Jira | Issue Key, 프로젝트, 담당자, 상태, 제목, 업데이트 시각 | 시작일, 마감일, 예상·잔여 공수, Story Point, Sprint·Velocity | 설명, Priority, Component, Label, 완료 이력, Worklog, 의존관계 |
| 신규 프로젝트 | 프로젝트명·목적·기간·우선순위·기획서, Task 제목·설명·역할·기간·예상 공수 | 필수 Skill·권한, 조직 제한, 선행 Task, 특정 인력 조건 | Component, Label, 산출물, 검토자, 코드 저장소, 위험도 |

휴가 또는 Jira 업무가 0건인 것과 조회 실패는 구분한다.

```text
조회 성공 + 0건 → 없음
조회 실패 → 확인 불가
```

### 5.2 기능별 차단 기준

| 기능 | 반드시 필요한 데이터 |
|---|---|
| 후보 생성 | 구성원 ID·역할·조직, 신규 Task |
| 현재 업무량 | Jira 사용자 매핑·담당자·상태 |
| 정량 부하율 | 근무시간·Task 기간·잔여 공수 |
| 일정 충돌 | 시작일·마감일·휴가 기간 |
| Skill 적합성 | 요구 Skill·보유 Skill |
| 유사 업무 판단 | 신규 Task 설명·과거 완료 Task |
| Jira 반영 | 대상 프로젝트·Issue Type·제목·Jira accountId·PM 승인 |

선택 데이터는 추천의 근거 범위, 설명력, 후보 구분력과 신뢰도를 높인다.

## 6. 부하 계산

```text
총 근무용량
= 기간 내 근무일 × 일 기준 근무시간 × FTE

유효 가용용량
= 총 근무용량 - 휴가·부재시간 - 고정 비프로젝트 일정시간

현재 할당량
= 해당 기간에 배분된 기존 업무 잔여 공수

잔여 가용시간
= max(0, 유효 가용용량 - 현재 할당량)

현재 부하율
= 현재 할당량 / 유효 가용용량 × 100

배정 후 부하율
= (현재 할당량 + 신규 Task 공수) / 유효 가용용량 × 100
```

내부 People DB에 개인별 근무시간이 없으면 회사 근무정책을 CapacityPolicy로 적용한다. 유효 가용용량이 0이면 퍼센트 계산을 수행하지 않고 해당 기간 배정을 BLOCKED 처리한다. 배정 후 부하율이 100%를 초과하면 과부하 위험으로 검증한다.

| 모드 | 조건 | 출력 |
|---|---|---|
| HOURS | 근무시간·기간·공수 존재 | 현재·배정 후 부하율 |
| POINTS | Story Point·Sprint Capacity·Velocity 존재 | Capacity 사용률 |
| PROXY (후속) | 공수 데이터 부족 | 업무량 지수 + 낮은 신뢰도 |

Story Point를 근거 없이 시간으로 바꾸지 않는다. PROXY 결과를 퍼센트 부하율로 표시하지 않는다. **PROXY 모드는 8/6 MVP 범위에서 제외하고 후속 확장으로 둔다. MVP는 HOURS를 기본, POINTS를 보조로 사용하며 공수 데이터가 부족하면 BLOCKED/PARTIAL_RESULT로 처리한다.**

## 7. 데이터 접근 구현 원칙

### 7.1 Jira

- 최초 연결 시 `/rest/api/3/field`로 사이트별 필드를 확인한다.
- 활성 이슈를 필요한 필드만 일괄 조회한 후 사람·프로젝트별로 내부 그룹화한다.
- `nextPageToken`으로 페이지네이션한다.
- 필요한 이슈만 설명·Worklog 등 상세 데이터를 추가 조회한다.
- 현재 KAN은 칸반 보드이므로 Sprint·Story Point를 기본 전제로 사용하지 않는다.
- 검증 통과와 PM 승인 이후에만 Epic·Task·담당자·일정을 반영한다.

```http
GET  /rest/api/3/field
POST /rest/api/3/search/jql
GET  /rest/api/3/issue/{key}
POST /rest/api/3/issue
PUT  /rest/api/3/issue/{key}
```

### 7.2 내부 People DB

1차 구현은 PostgreSQL People DB와 비식별 Seed 데이터를 사용한다. People Repository가 운영 테이블을 Canonical 객체로 정규화한 뒤 PersonSnapshot 등 분석 Snapshot을 생성한다. Graph DB나 OrgGraph는 사용하지 않으며 조직 계층은 관계형 테이블의 `parent_organization_id`와 `manager_person_id`로 표현한다.

주요 테이블:

- `organization`: 조직과 상하위 관계
- `person`: 직원·직무·재직·관리자·위치·시간대
- `responsibility_level`: 책임수준 기준
- `skill`, `person_skill`: Skill과 숙련도
- `work_schedule`: FTE와 근무시간
- `person_absence`: 휴가·휴직·부재
- `identity_link`: Jira·Google 외부 계정 매핑

모든 레코드에 updated_at과 source_version을 두고, 변경 시 새로운 PersonSnapshot을 생성한다. 실제 HR/Workday Connector는 후속 확장으로 분리하며 Snapshot 이후 추천·검증 로직은 동일하게 유지한다.

## 8. 에이전트와 출력

| 구성 | 출력 |
|---|---|
| 업무 추출 | Task·역할·일정·원문 근거 |
| 업무부하 판단 | 부하율 또는 업무량 지수 |
| 업무 분배 | 추천 후보·대안·추천 근거 |
| 검증 | PASS·CONDITIONAL_PASS·REJECT |
| PM | 승인·대안 선택·반려 |

수식은 결정론적 코드로 계산하고 LLM은 문서 해석과 설명에 사용한다.

최종 추천 결과에는 다음을 포함한다.

- 추천 담당자와 최소 1명의 대안
- 적합도와 부하 결과
- 사용한 원본 근거
- 누락 데이터와 가정
- 신뢰도와 검증 상태

## 10. 최종 정리

> 1차 구현에서는 내부 People DB가 사람·조직·가용성 정보의 원본이고, Jira는 기존 프로젝트 현황, Google Drive 프로젝트 문서는 신규 프로젝트 정보의 원본이다. 문서는 Project Knowledge Model로 통합한 뒤 내부 Draft와 PM 업무 분해 확인을 거쳐 추천하며, Jira 반영은 후속 단계로 분리한다. 필수 데이터가 없으면 관련 기능을 차단하고, 선택 데이터가 부족하면 제한 사항과 신뢰도를 표시한다. AI는 근거가 있는 담당자와 대안을 추천하고 최종 결정은 PM이 수행한다.
