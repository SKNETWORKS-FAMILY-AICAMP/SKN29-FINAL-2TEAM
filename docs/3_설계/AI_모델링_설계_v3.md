# AI 기반 프로젝트 업무 추출·분배 모델링 기획 (v3)

> 변경 이력 (v3, 2026-07-24)
> - **실행 순서 교정**: v5 정합 — `원천 검증 → Canonical 정규화 → Identity Mapping → AnalysisSnapshot 고정 → Feature Readiness → Agent·계산`. (v2는 Readiness→Snapshot 순서로 어긋나 있었음)
> - **문서 구조화 모델 재편**: "Parent 청크" 개념을 v5 Canonical Model 용어로 정리 — `DocumentBlock`(원문 구조 보존·추출 기준) + `ChildChunk`(검색 보조) + `KnowledgeItem`/`FeatureCluster`/`ProjectKnowledgeModel`.
> - **추출 주입력 명시**: 프로젝트 전체 Task는 `ProjectKnowledgeModel` 기준으로 추출하고 RAG는 유사 통합·과거 업무·근거 보완 보조로 한정.
> - 기준 문서: 기획서 v5, Canonical Entity-Relationship Catalog, Data Flow.
>
> 변경 이력 (v2)
> - 용어 통일(추출/분배 Agent), 팀장 확인 게이트 2단계, Feature Readiness·Snapshot·Task 출처·검증 Agent·업무량 결합 신설.

## 1. 모델링 목표

본 시스템은 대규모 프로젝트가 팀에 배정되었을 때 프로젝트 관련 문서, 기존 Jira 업무 이력 및 인사 데이터를 종합하여 프로젝트를 실행 가능한 업무 단위로 분해하고 적합한 담당자를 추천하는 것을 목표로 한다.

AI가 생성한 결과는 업무 추출·분배 초안으로 제공하며, 팀장의 검토와 승인을 거쳐 최종 반영한다. 팀장 확인은 두 번 수행한다.

1. **1차 확인 — 업무 추출 직후**: AI가 분해한 업무 구조를 승인·수정·반려한다. 1차 확인 전에는 담당자 추천을 실행하지 않는다.
2. **2차 확인 — 업무 분배·검증 후**: 담당자 추천안과 검증 결과를 보고 최종 승인·수정·반려한다.

문서 수정 시에는 재분배 기능으로 변경 내용을 다시 분석하고, 영향을 받는 업무와 담당자 배정안을 갱신한다. 재분배 결과도 동일하게 두 번의 확인을 거친다.

## 2. 활용 데이터

### 2.1 프로젝트 관련 비정형 문서 (Google Drive)

프로젝트 기획서, 요구사항 정의서, 회의록, 프로젝트 계획서, 기술 설계서, 일정·마일스톤 자료, 의사결정 기록, 리스크·제약사항 문서. 이들에서 목표·범위·요구사항·일정·제약조건·완료 조건·담당 영역을 추출한다.

### 2.2 Jira 업무 데이터

과거 업무 내용·유형, 업무별 담당자, 처리 상태·결과, 유사 프로젝트 배정 사례, 담당자별 수행 이력. 신규 업무와 과거 Jira 업무의 유사성을 분해·추천 근거로 활용한다.

### 2.3 인사 데이터 (내부 People DB)

소속 팀, 직무·역할, 보유 기술, 프로젝트 경험, 업무 가능 여부·가용성. 1차 구현의 인사 데이터는 내부 PostgreSQL People DB에 합성 데이터로 구성하며, 추출·분배 로직은 People Repository와 PersonSnapshot을 통해서만 접근한다. 물리 테이블명에는 mock을 쓰지 않고 합성 여부는 `data_origin`/`dataset_version`으로 구분한다(기획서 v5 4.2).

## 3. 전체 처리 흐름

```
프로젝트 문서 수집
→ 구조 보존 파싱 (DocumentBlock 생성)
→ 검색 보조 청킹 (ChildChunk) · 임베딩 · Vector 색인
→ 의미 유형 추출 (KnowledgeItem)
→ 중복 통합·클러스터링 (FeatureCluster → ProjectKnowledgeModel)
→ 원천 검증 → Canonical 정규화 → Identity Mapping
→ 분석 시점 Snapshot 고정 (AnalysisSnapshot)
→ 데이터 준비 상태 확인 (Feature Readiness)
→ 프로젝트 업무 추출 (ProjectKnowledgeModel 기준, RAG 보조)
→ 팀장 업무 확인 (1차: 승인·수정·재추출)
→ 업무량 계산 결과(WorkloadResult) 결합
→ 업무 분배 (담당자 추천)
→ 검증 Agent (PASS / CONDITIONAL_PASS / REJECT)
→ 팀장 최종 확인 (2차: 승인·수정·반려)
```

문서 수정 후 재분배: 수정 문서 재수집 → 재파싱·변경 Block 재처리 → KnowledgeItem·ProjectKnowledgeModel 영향 분석 → 새 Snapshot → 영향 업무 재추출 → 1차 확인 → 담당자 재계산 → 검증 → 2차 확인.

> **핵심 순서 규칙**: Snapshot을 먼저 고정하고 그 위에서 Feature Readiness를 판정한다. Readiness는 원천 레코드를 직접 검사하지 않고 고정된 Snapshot을 기준으로 실행 가능 여부를 결정한다.

## 4. 문서 파싱

Google Drive 문서는 파일 형식별 파서로 구조화한다. 파서는 의미를 판단하지 않고 물리적 구성요소만 추출한다: 문서 제목, 장·절 제목과 단계, 문단, 글머리표·번호 목록, 표의 행·열, 체크박스, 링크·강조, 텍스트 순서·위치.

파싱 결과는 고유 식별자를 가진 **DocumentBlock**으로 구성한다. DocumentBlock은 원문 구조와 의미를 보존하는 최소 단위이자 정보 추출의 기준이다.

```json
{
  "document_id": "doc-001",
  "blocks": [
    { "block_id": "b001", "block_type": "heading", "level": 1, "text": "회원 인증", "source_locator": {"page": 1} },
    { "block_id": "b002", "block_type": "paragraph", "text": "카카오와 구글 로그인을 지원해야 한다." },
    { "block_id": "b003", "block_type": "list_item", "text": "기존 이메일 계정과 연결할 수 있어야 한다." }
  ]
}
```

파싱 단계에서 원문 순서·구조를 유지하여 이후 AI 분석 결과를 원문과 연결한다. (형식별 파서·SourceLocator·표/목록 보존 상세는 「비정형 문서 파싱·청킹 공통 스키마 설계」 문서를 따른다.)

## 5. 문서 및 의미 유형 분류

문서 유형은 회사가 사전 정의한 카테고리 중 LLM이 선택하며 새 유형을 임의 생성하지 않는다. (예: 기획서, 요구사항 문서, 회의록, 프로젝트 계획서, 기술 설계서, 일정 문서, 의사결정 기록, 리스크 문서, 기타, 분류 불가)

하나의 문서에 여러 성격이 섞일 수 있으므로 문서 전체 유형과 청크 수준의 의미 유형(`semantic_type`)을 별도 분류한다. semantic_type 허용값: objective, scope, requirement, decision, constraint, milestone, dependency, risk, acceptance_criteria, role_or_ownership.

## 6. 청킹 전략과 DocumentBlock·ChildChunk 구조

파싱 문서를 고정 글자 수로 단순 분할하면 하나의 업무에 속한 요구사항·담당 영역·일정·완료 조건이 분리될 수 있다. 따라서 물리적 구조와 업무상 의미를 함께 고려한다.

### 6.1 두 계층 분리

| 계층 | 역할 | 사용처 |
|---|---|---|
| **DocumentBlock** | 원문 구조·의미 보존 단위 | 의미 추출·Task 근거의 **기준**. 전체 프로젝트 분석은 활성 Block 전체를 대상으로 한다 |
| **ChildChunk** | Block을 검색 가능한 크기로 나눈 파생 단위 | 임베딩·Vector 검색 **보조**. 기본 500~800 토큰, 중첩 80~120 토큰 |

검색은 작은 ChildChunk를 대상으로 하고, 실제 분석과 근거 제시는 해당 Block의 전체 문맥을 활용한다. 이 구조로 검색 정확도를 확보하면서 하나의 업무와 관련된 조건이 분리 해석되는 문제를 줄인다.

> v2의 "Parent 청크"는 v3에서 **DocumentBlock**(원문 기준)으로 대체한다. 프로젝트 전체 범위의 통합 문맥은 별도 청크가 아니라 아래 9장의 **ProjectKnowledgeModel**이 담당한다.

### 6.2 의미적 그룹화

LLM은 파싱된 Block을 분석하여 업무상 의미가 같은 Block들을 하나의 의미 단위로 묶는다. 원문을 새로 작성하지 않고 Block 식별자를 그룹화한다.

```json
{ "topic": "인증 API 개발", "block_ids": ["b010", "b011", "b012"],
  "semantic_types": ["requirement", "role_or_ownership", "milestone"] }
```

## 7. 메타데이터 생성

문서·Block·청크에 검색·의미 분류·원문 추적·변경 감지용 메타데이터를 부여한다.

- **문서**: 프로젝트 식별자, 문서 식별자, 문서명, 문서 유형, 상태, 버전(revision), 생성·수정일, 원본 링크, 접근 권한(ACL), 콘텐츠 해시
- **Block/청크**: Block·ChildChunk 식별자, 섹션 경로, 의미 유형, 업무 영역, 기능, 관련 역할·기술, 담당자·조직 언급, 일정·마감일, 이전 버전 식별자, 콘텐츠 해시

업무 영역(Business Scope)은 사전 정의 분류에서 LLM이 선택하고, Feature는 원문에서 추출해 표준 명칭으로 정규화한다. 근거가 부족하면 미분류·빈 값으로 둔다.

## 8. 임베딩 및 벡터 검색 (보조 경로)

검색용 ChildChunk에 프로젝트·의미 정보를 덧붙여 임베딩한다. 원문과 임베딩용 텍스트는 분리 관리한다(원문=근거 확인용, 임베딩 텍스트=검색용). Agent는 프로젝트 식별자·접근 권한으로 검색 범위를 제한한다.

벡터 검색의 용도는 **보조**로 한정한다: 여러 문서의 유사 요구사항 통합, 같은 기능의 다른 표현 검색, 관련 기술·일정·제약 탐색, 신규 업무와 유사한 과거 Jira 업무 검색. **RAG는 전체 문서 누락 여부를 판정하거나 프로젝트 전체 Task 구조를 단독 생성하지 않는다.**

## 9. 프로젝트 정보 클러스터링 → ProjectKnowledgeModel

프로젝트 내용은 여러 문서에 반복·분산될 수 있다. 문서별로 각각 업무화하면 중복 업무가 생긴다. 따라서 업무 추출 전에 의미 정보를 통합한다.

```
활성 DocumentBlock 전체
→ 의미 유형 추출 → KnowledgeItem[]
→ 중복 통합·상충/최신 버전 판정 → FeatureCluster[]
→ ProjectKnowledgeModel (프로젝트 전체 범위·관계·근거를 고정)
```

- **KnowledgeItem**: Block에서 구조화한 단일 지식 항목(목표·요구사항·결정·제약·일정·리스크·완료조건·담당 영역). 각 항목은 원문 Block 근거를 가진다.
- **FeatureCluster**: 여러 문서의 동일 기능·요구사항을 통합한 묶음.
- **ProjectKnowledgeModel**: 중복을 통합하고 상충·최신 버전을 해결한 프로젝트 단위 구조. **업무 추출의 기준 입력**이다.

```json
{ "cluster": "소셜 로그인", "business_scope": "회원",
  "source_knowledge_item_ids": ["ki-req-01", "ki-meeting-03", "ki-design-02"] }
```

## 10. 분석 시점 Snapshot 고정과 Identity Mapping

### 10.1 Snapshot

추출·분배 실행 직전에 입력 데이터를 불변 **AnalysisSnapshot**으로 고정한다(기획서 v5 11장).

- 대상: PersonSnapshot(인사), ExistingTaskSnapshot(Jira), DocumentIndex/ProjectKnowledgeModel(문서), 회사 정책, 팀장 추가 요구사항.
- 분석이 시작되면 입력 Snapshot은 변경하지 않는다. 원천이 바뀌어도 기존 결과는 당시 `snapshot_as_of`를 유지한다.
- 모든 결과에 Snapshot 식별자·정책 버전·계산 버전을 기록하여 재현·감사를 보장한다.
- 재분배 시 새 Snapshot과 새 결과를 생성한다.

### 10.2 Identity Mapping

People DB·Jira·Google 계정을 내부 personId로 연결한다(기획서 v5 6장). 우선순위: 확정 IdentityLink → employee_id와 사전 등록 외부 ID → 검증된 회사 이메일 완전 일치 → 복합 일치 후보 → 수동 확인. 이름 단독·LLM 추정만으로 자동 확정하지 않는다. 매핑 실패는 흐름을 중단하지 않고 PARTIAL_RESULT로 처리하되, 정책상 Jira 업무량이 필수인 기능은 BLOCKED.

## 11. 데이터 준비 상태 확인 (Feature Readiness)

**Snapshot 고정 이후**, 고정된 Snapshot을 기준으로 각 기능의 실행 가능 여부를 판정한다(기획서 v5 10장). 원천 레코드를 직접 검사하지 않는다.

| 상태 | 의미 | 처리 |
|---|---|---|
| SUCCESS | 필수 데이터 충족 | 전체 기능 실행 |
| PARTIAL_RESULT | 선택 데이터·일부 매핑 누락 | 제한·가정·낮은 신뢰도 표시 후 가능 범위 실행 |
| BLOCKED | 필수 데이터 누락 | 해당 기능 중단, 누락 항목·해결 방법 안내 |

판정 예시: 문서 본문 없음·권한 없음 → BLOCKED / 후보 필수 Skill 데이터 없음 → 부적합이 아니라 UNKNOWN 조건부 후보 / Jira 계정 매핑 실패 인원 → 제외하지 않고 PARTIAL_RESULT 기록. 모든 결과에 missingData·limitations·assumptions·confidence를 포함한다.

## 12. 업무 추출 Agent

업무 추출 Agent는 **ProjectKnowledgeModel**을 기준으로 프로젝트 전체를 실행 가능한 업무 구조로 분해한다. 필요 시에만 RAG로 유사 표현·과거 Jira 업무·추가 근거를 보완한다.

업무 계층: `프로젝트 → 업무 영역 → 업무 패키지 → 실행 가능한 세부 업무`. 세부 업무 분리 기준: 담당 역할 상이, 산출물 상이, 독립 완료 판단 가능, 선행·후행 의존, 기술 역량 상이, 별도 검토·테스트 필요.

결과 필드: 업무명·설명, 상위 업무 영역, 필요 역할·기술, 선행·후행 업무, 완료 조건, 일정·우선순위, 원문 Block/KnowledgeItem 근거. 모든 업무는 하나 이상의 근거와 연결한다.

### 12.1 Task 출처 구분

| 유형 | 의미 | 처리 |
|---|---|---|
| EXTRACTED | 문서 명시 업무 구조화 | 원문 Citation 필수 |
| GENERATED | 상위 요구를 하위 업무로 분해 | 상위 근거·생성 이유 필수 |
| AI_SUGGESTED_MISSING_TASK | 문서엔 없으나 의존성·품질상 필요 | 자동 확정 금지, 1차 확인에서 별도 배지·승인 필수 |
| USER_ADDED | 팀장 직접 추가 | 추가자·사유 기록 |

## 13. 팀장 업무 확인 (1차 게이트)

업무 추출 직후 팀장이 분해 결과를 확인한다. **1차 확인 전에는 담당자 추천을 실행하지 않는다.**

| 조치 | 처리 |
|---|---|
| 승인 | 초안을 불변 NewTaskSnapshot으로 고정 |
| 수정 | 업무명·역할·공수·일정·우선순위 수정, 이력 저장 후 확정 |
| 반려 | 문서·분해 조건 보완 후 재추출 |

## 14. 업무량 계산 결합 (WorkloadResult)

가용성·부하는 LLM이 판단하지 않는다. 규칙·수식 기반 **업무량 계산 모듈**이 결정론적으로 계산하고, 업무 분배 Agent는 그 출력(WorkloadResult)을 입력으로만 사용한다(기획서 v5 13.1).

```
총 근무용량   = 기간 내 근무일 × 일 기준 근무시간 × FTE
유효 가용용량 = 총 근무용량 − 휴가·부재시간 − 고정 비프로젝트 일정시간
현재 할당량   = 해당 기간에 배분된 기존 업무 잔여시간
잔여 가용시간 = max(0, 유효 가용용량 − 현재 할당량)
현재 부하율   = 현재 할당량 / 유효 가용용량 × 100
배정 후 부하율 = (현재 할당량 + 신규 배정시간) / 유효 가용용량 × 100
```

유효 가용용량이 0이면 부하율을 계산하지 않고 해당 기간 배정을 BLOCKED. 시간과 Story Point는 직접 합산하지 않으며 계산 방식별 Readiness를 먼저 판정한다. LLM의 역할은 계산 결과를 팀장이 이해할 설명으로 변환하는 것까지다.

## 15. 업무 분배 Agent

업무 분배 Agent는 **확정 Task(1차 확인 완료)** 와 **WorkloadResult** 를 입력으로 받아 담당자를 추천한다. 고려 항목: 직무·역할 적합도, 요구-보유 기술 일치도, 과거 유사 Jira 업무 경험, 동일 프로젝트·시스템 경험, 현재 가용성·부하율(WorkloadResult 기반), 프로젝트 전체 편중도.

업무별 최고 점수만 개별 선택하면 특정 인력 집중이 생기므로 프로젝트 전체 업무와 전체 후보자를 함께 고려해 배정안을 구성한다. 결과: 추천 담당자, 최소 1명의 대안, 추천 점수, 직무·기술 근거, 유사 업무 근거, 예상 부하율, 누락 데이터·가정·신뢰도.

Agent는 계산식을 직접 수행하거나 계산값을 임의 변경하지 않는다. 적격성·적합도·가용성·위험 지표는 규칙·수식 코드가 생성한다.

## 16. 검증 Agent

분배 초안은 팀장에게 전달되기 전 검증 Agent를 통과해야 한다. 규칙 기반으로 수행하며 LLM에 최종 판정을 맡기지 않는다(기획서 v5 14.3).

검사: 역할·필수 Skill·책임수준, 조직 제한·권한, 휴가·휴직 중복, 일정 충돌·최대 부하율, 추천 근거 존재, 원본 출처 접근 가능, 데이터 누락·가정·신뢰도.

| 결과 | 후속 |
|---|---|
| PASS | 팀장 2차 확인 전달 |
| CONDITIONAL_PASS | 제한 표시 후 2차 확인 전달 |
| REJECT | 승인 대상 제외, 데이터·조건 수정 후 재분배 |

REJECT는 Jira에 반영하지 않는다.

## 17. 팀장 최종 확인 (2차 게이트)

| 조치 | 처리 |
|---|---|
| 승인 | DecisionRecord 저장 (Jira 반영은 후속 단계) |
| 수정 | 담당자·기간·공수 변경 후 검증 Agent로 직접 재전달·재검증 |
| 반려 | 사유별 회귀: 분해 오류 → 업무 추출, 조건 변경 → 업무 분배, 데이터 오류 → 데이터 수집 |

## 18. 재분배 처리

문서 수정 시 변경 문서를 재파싱·재청킹하고 새 Snapshot을 생성한다. 문서·Block·revision의 버전과 콘텐츠 해시를 비교해 신규·수정·삭제·변경 없음을 식별하고, 변경 Block이 KnowledgeItem·ProjectKnowledgeModel·기존 업무의 근거인지 추적해 영향 업무를 식별한다.

업무 변경: 추가·수정·삭제·분할·병합·변경 없음. 재추출 업무는 팀장 1차 확인을 다시 거친다. 역할·필요 기술이 달라지면 담당자 적합도를 재계산하고, 기존 업무를 그대로 수행 가능하면 연속성을 고려해 기존 담당자를 유지하는 방향으로 추천한다. 재분배 결과도 검증 Agent와 2차 확인을 거친다.

## 19. 모델링 원칙

1. 문서를 고정 길이로 단순 분할하지 않고 구조와 의미를 함께 고려한다.
2. 검색은 ChildChunk를 대상으로 하고 실제 분석·근거는 DocumentBlock 문맥을 활용한다.
3. 프로젝트 전체 Task는 ProjectKnowledgeModel을 기준으로 추출하며 RAG는 통합·과거 업무·근거 보완의 보조다.
4. 업무 영역은 사전 정의 분류를 쓰고 기능은 원문에서 추출해 정규화한다.
5. 모든 업무·추천은 원본 문서 또는 기존 데이터 근거와 연결한다.
6. 시간·가용성·부하율·필수조건은 규칙·수식으로 계산하며 LLM에 최종 판정을 맡기지 않는다 (Bounded LLM).
7. 데이터가 부족해도 가능한 범위의 결과를 제공하고 누락·제한·가정·신뢰도를 표시한다.
8. **Snapshot을 먼저 고정하고 그 위에서 Feature Readiness를 판정한다.** 추출·분배 입력은 불변 Snapshot으로 재현·감사가 가능하게 한다.
9. 문서 변경 시 전체 재생성 대신 변경 영향 중심으로 재분배한다.
10. AI는 추출·분배 초안을 생성하며, 업무 추출 직후(1차)와 최종 반영 전(2차) 두 번의 팀장 확인을 거친다.
