# AI 프로젝트 운영 코파일럿 업무 분배 필수 데이터 정의서

> 문서 상태: 확정안  
> 작성일: 2026-07-22  
> 기준 문서: `기획서_v5_AI프로젝트운영코파일럿.md`, `업무_분배_로직_및_API_데이터_최종안.md`  
> 적용 범위: 후보 생성, 업무량·가용성 계산, 담당자 추천, 필수 제약조건 검증, PM 검토

---

## 1. 목적

이 문서는 업무 분배에 필요한 데이터를 원천 시스템의 필드 나열이 아니라 **추천 시점의 정규화 객체와 기능별 실행 조건**으로 정의한다.

핵심 원칙은 다음과 같다.

1. 업무 분배 Agent는 HR 테이블, Jira 응답, Google 문서와 Calendar 이벤트를 직접 사용하지 않는다.
2. 원천 데이터는 검증·정규화되어 Snapshot 또는 정책 객체로 변환된 후 사용한다.
3. Google Drive·Docs는 문서 기반 Task 추출 기능의 조건부 필수 데이터다.
4. Google Calendar는 일정 충돌과 가용시간 보정 기능의 조건부 필수 데이터다.
5. PM이 확정한 `NewTaskSnapshot`은 업무 분배의 필수 입력이다.
6. 데이터가 없다는 사실과 조회 실패를 구분한다.
7. Skill 데이터가 없으면 불충족으로 단정하지 않고 `UNKNOWN`으로 처리한다.
8. 모든 추천 입력은 동일한 `snapshotAsOf` 기준으로 고정한다.

## 2. 데이터 필수성 정의

| 구분 | 의미 | 누락 시 기본 처리 |
|---|---|---|
| 필수 | 해당 기능을 실행하기 위해 항상 필요한 데이터 | `BLOCKED` |
| 조건부 필수 | 특정 계산 모드·제약조건·Connector 기능을 사용할 때 필요한 데이터 | 해당 기능 `BLOCKED` 또는 대체 모드 전환 |
| 선택 | 추천 근거·설명력·정확도를 보강하는 데이터 | `PARTIAL_RESULT`, 신뢰도 하향 또는 영향 없음 |
| 파생 | 필수·조건부 입력을 이용해 규칙·수식 코드가 계산하는 데이터 | 입력 데이터의 Readiness를 상속 |

### 2.1 결과 상태

| 상태 | 의미 | 사용자 처리 |
|---|---|---|
| `SUCCESS` | 요청한 기능에 필요한 데이터와 정책이 준비됨 | 정상 실행 |
| `PARTIAL_RESULT` | 일부 후보·선택 데이터·보조 기능을 확인할 수 없음 | 제한 사항과 신뢰도를 표시하고 PM 검토 |
| `BLOCKED` | 필수 데이터 또는 필수 정책이 없어 요청 기능을 실행할 수 없음 | 데이터·정책·사용자 입력 보완 |

### 2.2 필드 상태

필드 값에는 값 자체와 함께 다음 상태를 관리한다.

| 필드 상태 | 의미 | 예시 |
|---|---|---|
| `PRESENT` | 유효한 값이 존재함 | 잔여 공수 16시간 |
| `EMPTY_CONFIRMED` | 조회에는 성공했으며 값이 실제로 없음 | 휴가 일정 0건 |
| `UNAVAILABLE` | 권한·연결·매핑 문제로 확인할 수 없음 | Jira accountId 매핑 실패 |
| `INVALID` | 형식·범위·참조 규칙에 맞지 않음 | 종료일이 시작일보다 빠름 |
| `STALE` | 허용된 최신성 기준을 초과함 | Jira 마지막 동기화가 TTL 초과 |

`EMPTY_CONFIRMED`와 `UNAVAILABLE`은 동일하게 처리하지 않는다.

## 3. 업무 분배 입력 구조

```text
내부 People DB ───→ PersonSnapshot ──────────┐
                                              │
Jira Connector ──→ ExistingTaskSnapshot ─────┤
                                              │
Google Calendar ─→ CalendarEventSnapshot ────┤
                                              ├─→ WorkloadResult
회사 정책 ───────→ CapacityPolicy ───────────┤
                                              │
Google Drive·Docs ─→ DocumentIndex ─→ ProjectKnowledgeModel
                                      ↓       │
                                  Task 추출   │
                                      ↓       │
                               NewTaskDraft   │
                                      ↓       │
                                  PM 확인     │
                                      ↓       │
                               NewTaskSnapshot┘

확정 NewTaskSnapshot + PersonSnapshot + WorkloadResult
→ 후보 생성
→ 적합도·가용성·충돌 분석
→ RecommendationResult
→ ValidationResult
```

업무 분배의 직접 입력은 다음과 같다.

- `PersonSnapshot`
- `ExistingTaskSnapshot`
- `NewTaskSnapshot`
- `CapacityPolicy`
- 필요 시 `CalendarEventSnapshot`
- `IdentityLink`

`DocumentIndex`는 Task 추출과 Citation 생성에 사용되는 지원 객체이며, PM 확인이 끝난 이후의 업무 분배 계산에는 직접 사용하지 않는다. 단, 추천 근거를 원문까지 추적하기 위한 `sourceRefs`는 유지한다.

## 4. 공통 Snapshot 필드

모든 Snapshot은 다음 공통 필드를 가진다.

| 필드 | 타입 | 필수성 | 업무 의미 | 검증 규칙 | 누락 시 |
|---|---|---:|---|---|---|
| `snapshotId` | string | 필수 | Snapshot 고유 ID | Tenant 내 중복 불가 | `BLOCKED` |
| `tenantId` | string | 필수 | 고객사·조직 데이터 경계 | 요청 tenant와 일치 | `BLOCKED` |
| `schemaVersion` | string | 필수 | 내부 객체 스키마 버전 | 지원 버전인지 확인 | `BLOCKED` |
| `mappingVersion` | string | 필수 | 원천→내부 필드 매핑 버전 | 사용한 ConnectorProfile과 일치 | `BLOCKED` |
| `sourceVersion` | string | 필수 | 원천 데이터 또는 Seed 버전 | 원천별 버전 형식 확인 | `BLOCKED` |
| `sourceUpdatedAt` | datetime | 필수 | 원천 데이터 마지막 변경 시각 | ISO 8601, 미래 시각 금지 | `BLOCKED` 또는 `STALE` |
| `lastSyncedAt` | datetime | 필수 | 플랫폼 마지막 동기화 시각 | ISO 8601, 미래 시각 금지 | `BLOCKED` 또는 `STALE` |
| `snapshotAsOf` | datetime | 필수 | 추천 입력을 고정한 기준 시각 | 동일 추천 요청의 Snapshot 간 기준 일치 | `BLOCKED` |
| `createdAt` | datetime | 필수 | Snapshot 생성 시각 | `snapshotAsOf` 이후 생성 | `BLOCKED` |
| `sourceRefs` | array | 필수 | 원천 레코드·문서·이슈 근거 | 최소 1개, 접근 가능한 원천 ID | `BLOCKED` 또는 `PARTIAL_RESULT` |
| `dataReadinessStatus` | enum | 필수 | 객체 단위 준비 상태 | `SUCCESS/PARTIAL_RESULT/BLOCKED` | `BLOCKED` |
| `missingData` | array | 필수 | 누락·확인 불가 필드 목록 | 누락이 없으면 빈 배열 | `BLOCKED` |
| `limitations` | array | 필수 | 계산·설명의 제한 사항 | 제한이 없으면 빈 배열 | `BLOCKED` |
| `assumptions` | array | 필수 | 적용한 정책·Fallback | 가정이 없으면 빈 배열 | `BLOCKED` |
| `confidence` | enum/number | 필수 | 데이터 기반 신뢰도 | `HIGH/MEDIUM/LOW` 또는 0~1 | `BLOCKED` |

## 5. PersonSnapshot 데이터 정의

`PersonSnapshot`은 추천 시점의 직원·조직·직무·역량·근무 조건을 나타낸다.

| 필드 | 타입 | 필수성 | 업무 의미 | 원천 | 검증 규칙 | 누락 시 |
|---|---|---:|---|---|---|---|
| `personId` | string | 필수 | 플랫폼 내부 직원 식별자 | HR Repository | 중복 불가 | `BLOCKED` |
| `employeeId` | string | 필수 | 내부 직원 식별자 | People DB | 직원 테이블 참조 확인 | `BLOCKED` |
| `displayName` | string | 필수 | PM 화면 표시 이름 | People DB | 공백 불가 | `BLOCKED` |
| `employmentStatus` | enum | 필수 | 재직·휴직·퇴사 상태 | People DB | `ACTIVE/LEAVE/TERMINATED` 등 허용값 | `BLOCKED` |
| `primaryOrganizationId` | string | 필수 | 주 소속 조직 | People DB | 활성 조직 참조 확인 | `BLOCKED` |
| `jobRoleCode` | string | 필수 | 후보 생성에 사용하는 직무·역할 | People DB | 역할 기준정보 참조 | `BLOCKED` |
| `jobTitle` | string | 선택 | PM 설명용 직책명 | People DB | 문자열 | `PARTIAL_RESULT` 영향 낮음 |
| `responsibilityLevel` | string/number | 조건부 필수 | Task 책임수준 충족 여부 | People DB | 고객별 책임수준 체계에 존재 | 책임수준 검증 `BLOCKED` |
| `skills` | array | 조건부 필수 | Skill 적합성 판단 | People DB | Skill 코드·숙련도 범위·검증 시각 확인 | Skill 결과 `UNKNOWN` |
| `fte` | decimal | 조건부 필수 | 개인 근무용량 산정 | People DB | 0 초과, 기본 1.0 이하 | 회사 정책 또는 근무시간으로 대체 |
| `weeklyHours` | decimal | 조건부 필수 | HOURS 방식 기준 근무시간 | People DB | 0 초과 | `CapacityPolicy`로 대체, 둘 다 없으면 `BLOCKED` |
| `workScheduleId` | string | 조건부 필수 | 근무일·시간대·근무형태 참조 | People DB | 유효기간이 추천 기간과 중첩 | `CapacityPolicy`로 대체 |
| `absenceRanges` | array | 조건부 필수 | 휴가·휴직·부재 차감 | People DB | 시작≤종료, 직원 일치, 중복 병합 | 휴가 검증 정책에 따라 `BLOCKED/PARTIAL_RESULT` |
| `locationCode` | string | 선택 | 위치·근무지 제한 및 설명 | People DB | 위치 기준정보 참조 | 신뢰도 하향 가능 |
| `timezone` | string | 조건부 필수 | 다중 시간대 일정 계산 | People DB/정책 | IANA timezone | Tenant timezone Fallback |
| `organizationRelations` | array | 선택 | 겸직·보조 소속 | People DB | 유효기간 확인 | 주 소속만 사용 |
| `identityMappingStatus` | enum | 조건부 필수 | 외부 계정 매핑 상태 | Identity Mapping | 허용 상태 확인 | 외부 데이터 기능 영향 |
| `jiraAccountId` | string | 조건부 필수 | Jira 업무를 직원에게 귀속 | Identity Mapping | `MATCHED`일 때 단일 accountId | 해당 직원 Jira 부하 `UNAVAILABLE` |
| `googleUserId` | string | 조건부 필수 | Calendar 이벤트를 직원에게 귀속 | Identity Mapping | `MATCHED`일 때 단일 Google ID | Calendar 보정 제외 |

### 5.1 PersonSnapshot 핵심 실행 조건

- 후보 생성에는 `personId`, `employmentStatus`, `primaryOrganizationId`, `jobRoleCode`가 필수다.
- `employmentStatus != ACTIVE`인 직원은 기본 후보에서 제외한다.
- Skill 조건이 있는 Task에서 `skills`가 없으면 `MISMATCH`가 아니라 `UNKNOWN`이다.
- 정량 부하율 계산에는 `weeklyHours` 또는 적용 가능한 `CapacityPolicy`가 반드시 있어야 한다.
- Jira 매핑 실패 직원은 후보에서 자동 제외하지 않는다. 해당 직원의 Jira 업무량을 확인할 수 없음을 표시하고 고객 정책에 따라 `PARTIAL_RESULT` 또는 기능 `BLOCKED`로 처리한다.

## 6. ExistingTaskSnapshot 데이터 정의

`ExistingTaskSnapshot`은 Jira의 서로 다른 이슈 필드를 통일된 기존 업무 단위로 변환한 객체다.

| 필드 | 타입 | 필수성 | 업무 의미 | 원천 | 검증 규칙 | 누락 시 |
|---|---|---:|---|---|---|---|
| `existingTaskId` | string | 필수 | 내부 기존 업무 ID | Jira Adapter | 중복 불가 | `BLOCKED` |
| `issueKey` | string | 필수 | Jira 이슈 식별자 | Jira | 사이트 내 형식·중복 확인 | `BLOCKED` |
| `projectId` | string | 필수 | 기존 프로젝트 식별자 | Jira | 선택 프로젝트 범위에 포함 | `BLOCKED` |
| `summary` | string | 필수 | 업무명 및 근거 표시 | Jira | 공백 불가 | `BLOCKED` |
| `assigneeAccountId` | string/null | 조건부 필수 | Jira 담당자 계정 | Jira | 미배정 이슈는 null 허용 | 담당자별 부하에서 제외·제한 표시 |
| `assigneePersonId` | string/null | 조건부 필수 | 내부 직원 귀속 | Identity Mapping | 단일 personId와 매핑 | 해당 직원 부하 `UNAVAILABLE` |
| `status` | string | 필수 | Jira 상태 원문 | Jira | 사이트 상태 매핑 존재 | `BLOCKED` |
| `statusCategory` | enum | 필수 | 미착수·진행·완료 통합 상태 | Jira Adapter | `TODO/IN_PROGRESS/DONE` | `BLOCKED` |
| `priority` | string | 선택 | 업무 긴급도·위험 보정 | Jira | 사이트 Priority 매핑 | 근거 축소 |
| `startAt` | datetime/null | 조건부 필수 | 일정 중복 시작 시각 | Jira | 시작≤종료 | 일정 충돌 기능 `BLOCKED` |
| `dueAt` | datetime/null | 조건부 필수 | 일정 중복·마감 위험 | Jira | 시작≤마감 | 일정 충돌 기능 `BLOCKED` |
| `originalEstimateHours` | decimal/null | 조건부 필수 | 최초 예상 공수 | Jira | 0 이상, 단위 변환 기록 | 대체 공수 확인 |
| `remainingEstimateHours` | decimal/null | 조건부 필수 | 현재 할당량 계산 | Jira | 0 이상 | HOURS 방식 `BLOCKED` 또는 대체 모드 |
| `timeSpentHours` | decimal/null | 선택 | 공수 검증·진행 판단 | Jira | 0 이상 | 설명력 하향 |
| `storyPoints` | decimal/null | 조건부 필수 | POINTS 방식 부하 계산 | Jira | 0 이상 | POINTS 방식 `BLOCKED` |
| `sprintId` | string/null | 조건부 필수 | Sprint 범위 귀속 | Jira Software | Board·Sprint 참조 확인 | POINTS 방식 `BLOCKED` |
| `sprintStartAt` | datetime/null | 조건부 필수 | Sprint 기간 계산 | Jira Software | 시작≤종료 | POINTS 방식 `BLOCKED` |
| `sprintEndAt` | datetime/null | 조건부 필수 | Sprint 기간 계산 | Jira Software | 시작≤종료 | POINTS 방식 `BLOCKED` |
| `updatedAt` | datetime | 필수 | 이슈 최신성 판단 | Jira | 미래 시각 금지 | `BLOCKED/STALE` |
| `worklogUpdatedAt` | datetime/null | 선택 | 상세 공수 최신성 | Jira | 미래 시각 금지 | Worklog 설명 제한 |
| `dependencyIssueKeys` | array | 선택 | 선행·후행 업무 충돌 | Jira | 존재하는 이슈 참조 | 의존성 분석 제외 |

### 6.1 Jira 공수 모드

| 모드 | 조건부 필수 데이터 | 출력 | 누락 시 |
|---|---|---|---|
| `HOURS` | 기간, `remainingEstimateHours`, 근무용량 | 현재·배정 후 부하율 | `POINTS` 또는 `PROXY` 검토 |
| `POINTS` | `storyPoints`, Sprint, Velocity 또는 Sprint Capacity | Capacity 사용률 | `HOURS` 또는 `PROXY` 검토 |
| `PROXY` | 담당자, 상태, 우선순위, 갱신 시각, 활성 이슈 수 | 업무량 지수 | 낮은 신뢰도 명시 |

Story Point를 근거 없이 시간으로 변환하지 않는다. `PROXY` 결과를 퍼센트 부하율로 표시하지 않는다.

## 7. NewTaskSnapshot 데이터 정의

`NewTaskSnapshot`은 AI가 추출한 Draft를 PM이 승인·수정한 후 확정한 신규 업무 객체다.

| 필드 | 타입 | 필수성 | 업무 의미 | 원천 | 검증 규칙 | 누락 시 |
|---|---|---:|---|---|---|---|
| `newTaskId` | string | 필수 | 신규 업무 내부 ID | 플랫폼 | 프로젝트 내 중복 불가 | `BLOCKED` |
| `projectId` | string | 필수 | 신규 프로젝트 식별자 | 프로젝트 요청 | 요청 프로젝트와 일치 | `BLOCKED` |

### 7.1 Task 출처 처리

- `EXTRACTED`: 원문에 직접 존재하며 Citation이 있다.
- `GENERATED`: 원문 정보를 조합해 구조화했으며 사용한 근거를 연결한다.
- `AI_SUGGESTED_MISSING_TASK`: 원문에 직접 없는 누락 가능 업무로 별도 표시한다.
- `USER_ADDED`: PM이 직접 추가한 업무이며 사용자·시각·사유를 근거로 남긴다.
- `AI_SUGGESTED_MISSING_TASK`는 PM 승인 전 확정 Task로 사용하지 않는다.
- PM 수정값은 원문 추출값을 덮어쓰지 않고 변경 이력과 함께 새로운 확정값으로 저장한다.

## 8. CalendarEventSnapshot 데이터 정의

Google Calendar는 시스템 전체의 주요 데이터 소스이지만 모든 업무 분배 요청의 절대 필수 데이터는 아니다. Calendar 기반 일정 충돌 또는 가용시간 보정을 사용하는 경우에만 조건부 필수다.

Calendar 이벤트는 API상 구조화된 필드를 가지며, 제목·설명과 같은 자유 텍스트는 비정형 정보다. 업무 분배에는 가능한 한 자유 텍스트를 사용하지 않고 가용성 영향만 정규화한다.

| 필드 | 타입 | 필수성 | 업무 의미 | 검증 규칙 | 누락 시 |
|---|---|---:|---|---|---|
| `calendarEventId` | string | 조건부 필수 | 이벤트 고유 ID | Calendar 내 중복 불가 | Calendar 기능 `BLOCKED` |
| `personId` | string | 조건부 필수 | 이벤트 대상 직원 | Identity Mapping 단일 일치 | 해당 직원 Calendar `UNAVAILABLE` |
| `calendarId` | string | 조건부 필수 | 원천 캘린더 | 허용된 Calendar 범위 | Calendar 기능 `BLOCKED` |
| `eventType` | enum | 조건부 필수 | 회의·휴가·출장·근무 위치 구분 | `MEETING/OUT_OF_OFFICE/TRAVEL/WORK_LOCATION/OTHER` | 보정 방식 `BLOCKED` |
| `startAt` | datetime | 조건부 필수 | 이벤트 시작 | 시작≤종료 | 이벤트 제외 |
| `endAt` | datetime | 조건부 필수 | 이벤트 종료 | 시작≤종료 | 이벤트 제외 |
| `timezone` | string | 조건부 필수 | 일정 시간대 | IANA timezone | 사용자·Tenant timezone Fallback |
| `allDay` | boolean | 조건부 필수 | 종일 이벤트 여부 | boolean | 이벤트 해석 제한 |
| `transparency` | enum | 조건부 필수 | Busy/Free 상태 | `BUSY/FREE` | 기본 정책 적용 및 가정 표시 |
| `responseStatus` | enum | 선택/조건부 | 참석 여부 | `ACCEPTED/TENTATIVE/DECLINED/NEEDS_ACTION` | 회의 차감 신뢰도 하향 |
| `recurrenceId` | string/null | 선택 | 반복 일정 인스턴스 연결 | 반복 원본 참조 | 중복 위험 표시 |
| `availabilityImpactHours` | decimal | 파생 | 실제 가용시간 차감량 | 0 이상, 중복 구간 병합 | Workload 계산 제한 |
| `updatedAt` | datetime | 조건부 필수 | 이벤트 최신성 | 미래 시각 금지 | `STALE` |

### 8.1 개인정보 최소화

- 이벤트 제목과 설명은 기본 수집 대상이 아니다.
- 휴가 사유는 저장하지 않는다.
- 업무 분배에는 이벤트 유형, 기간, Busy/Free, 가용성 차감 시간만 사용한다.
- PM에게는 상세 회의명이 아니라 “회의 4시간”, “부재 1일”과 같이 집계된 결과를 표시한다.

| 문서 기반 Task 추출 | Drive 파일, 파싱 본문, 청크, ACL, Citation | OCR이 필요한 경우 OCR 결과 | 제목 계층·표 구조 | 추출 `BLOCKED` |
| PM 수동 Task 입력 | Task 필수 필드, 사용자 입력 근거 | 없음 | 첨부 문서 | Docs 없이 실행 가능 |
| 추천 근거 표시 | Snapshot sourceRefs, 계산식 버전 | 문서 추출 Task는 Citation | 보조 설명 | 근거 누락 추천 `REJECT` |
| 필수 제약조건 검증 | 적용된 필수조건과 후보 데이터 | Task에 선언된 조건별 데이터 | 선택 위험 지표 | 필수조건 확인 불가 시 `CONDITIONAL_PASS/BLOCKED` 정책 적용 |

## 13. 원천 데이터 우선순위와 충돌 처리

| 데이터 | 1순위 | 2순위/Fallback | 충돌 처리 |
|---|---|---|---|
| Task 확정값 | PM이 승인·수정한 값 | 문서 `EXTRACTED` 값 | PM 변경 이력과 원문 값을 모두 보존 |
| AI 누락 업무 제안 | PM 승인 후 확정값 | `AI_SUGGESTED_MISSING_TASK` | 승인 전 추천 입력 사용 금지 |
| 재직·조직·직무 | People DB | 운영자 데이터 보정 | 보정은 별도 버전·감사 로그 필수 |
| 직원 Skill | HR에서 검증된 Skill | 이력서·문서 추출 Skill은 미검증 근거 | 미검증 Skill만 있으면 `UNKNOWN/CONDITIONAL` |
| 개인 근무시간 | 개인 `workSchedule` | 조직·Tenant `CapacityPolicy` | Fallback 사용을 assumptions에 기록 |
| 휴가·휴직 | HR 공식 부재 데이터 | Calendar `OUT_OF_OFFICE` | 시간 구간 합집합으로 병합해 이중 차감 금지 |
| 회의·출장·근무 위치 | Calendar 정규화 이벤트 | 관리자 정책 입력 | 상세 제목 없이 시간·유형만 사용 |
| Jira 현재 할당량 | `remainingEstimateHours` | 유효한 원본 공수에서 계산 가능한 파생값 | 파생식·단위·가정을 기록 |
| Jira POINTS | Jira Story Point·Sprint | 시간 공수와 독립 모드 | 근거 없는 시간 변환 금지 |
| 외부 사용자 ID | 수동 확정 IdentityLink | 등록 ID→회사 이메일→복합 후보 | 복수 후보는 자동 확정 금지 |
| 시간대 | 개인 timezone | 근무지 정책→Tenant timezone | Fallback 단계 기록 |

### 13.1 휴가와 Calendar 중복 제거

```text
HR 휴가:      2026-08-03 09:00 ───────── 18:00
Calendar OOO: 2026-08-03 09:00 ───────── 18:00

잘못된 계산: 8시간 + 8시간 = 16시간 차감
올바른 계산: 시간 구간 합집합 = 8시간 차감
```

동일인의 중복 이벤트는 `personId + time range + event type + sourceRefs`를 기준으로 병합하고, 어떤 원천을 사용했는지 계산 근거에 남긴다.

## 14. 최신성 기본 기준

다음 값은 초기 구현의 제안 기본값이며 고객별 `FreshnessPolicy`로 변경할 수 있다.

| 데이터 | 제안 기본 기준 | 초과 시 |
|---|---|---|
| People DB | `sourceVersion` 변경 감지 또는 24시간 | PersonSnapshot 재생성 |
| Jira 이슈 | 추천 요청 기준 15분 이내 동기화 | 재동기화, 실패하면 `STALE` |
| Calendar | 추천 요청 기준 15분 이내 동기화 | 재동기화 또는 Calendar 보정 제외 |
| Drive 문서 | 선택 파일의 최신 revision 인덱싱 완료 | 재파싱·재인덱싱 전 Task 추출 차단 |
| Identity Mapping | 추천 Snapshot의 `mappingVersion`과 일치 | 관련 Snapshot 재생성 |
| CapacityPolicy | 추천 기간에 유효한 불변 버전 | 정량 부하율 차단 |

추천 요청 직전에 최신성을 다시 확인하고, 요청이 시작된 후에는 동일 요청에서 원천 변경값을 섞지 않는다. 최신 데이터가 도착하면 새로운 Snapshot과 추천 요청을 생성한다.

## 15. Feature Readiness 판정 규칙

### 15.1 전체 판정 순서

```text
1. 원천 연결·권한·필드·형식 검사
2. Identity Mapping 검사
3. Snapshot 정규화·버전·최신성 검사
4. 요청 기능별 필수 데이터 검사
5. 대체 계산 모드와 Fallback 검사
6. SUCCESS / PARTIAL_RESULT / BLOCKED 판정
7. 누락 데이터·제한·가정·신뢰도·사용자 조치 출력
```

### 15.2 주요 판정 규칙

| 조건 | 상태 | 처리 |
|---|---|---|
| 확정 `NewTaskSnapshot`이 없음 | `BLOCKED` | PM 업무 분해 확인 요청 |
| 활성 후보가 0명 | `BLOCKED` | HR·조건·조직 범위 보완 |
| 적격 후보가 1명뿐임 | `PARTIAL_RESULT` | 대안 후보 부족 위험 표시 |
| Jira 전체 조회 실패 | 기본 `BLOCKED` | 업무량 인지 추천 차단, 고객 정책상 허용 시 제한 모드 |
| 일부 직원 Jira 매핑 실패 | `PARTIAL_RESULT` | 해당 직원 Jira 부하 `UNAVAILABLE`, 신뢰도 하향 |
| Skill 요구가 있으나 직원 Skill이 없음 | `PARTIAL_RESULT` 또는 조건부 기능 상태 | 후보를 `UNKNOWN`으로 유지 |
| 개인 근무시간 없음, 정책 있음 | `SUCCESS` 또는 `PARTIAL_RESULT` | CapacityPolicy 적용·가정 표시 |
| 개인 근무시간과 정책 모두 없음 | `BLOCKED` | HOURS 부하율 차단 |
| 잔여 공수 없음, POINTS 데이터 있음 | `PARTIAL_RESULT` | POINTS 모드 전환 |
| 시간·POINTS 모두 없음 | `PARTIAL_RESULT` | PROXY 모드, 퍼센트 표시 금지 |
| 유효 가용용량 0 | `BLOCKED` | 해당 기간 신규 배정 금지 |
| Calendar 미연결, HR 휴가 사용 가능 | `PARTIAL_RESULT` 또는 `SUCCESS` | 회의 보정 제외, 휴가 계산 유지 |
| 문서 미연결, PM이 Task 직접 입력 | `SUCCESS` | `USER_ADDED` 출처로 처리 |
| 문서 기반 Task에 Citation 없음 | `BLOCKED` | 추출 결과 확정 금지 |
| 필수 제약조건 위반 | 데이터 준비도와 별개 | Validation `REJECT` |

### 15.3 데이터 준비도와 추천 검증의 구분

- Feature Readiness는 **계산 가능한가**를 판단한다.
- Validation Agent는 **추천 결과가 조건을 충족하는가**를 판단한다.
- 데이터가 준비되어도 추천 후보가 필수조건을 위반하면 `REJECT`다.
- 데이터가 부족해 조건을 확인할 수 없으면 정책에 따라 `CONDITIONAL_PASS` 또는 해당 기능 `BLOCKED`다.

## 16. 부하 계산 입력과 파생값

### 16.1 계산식

```text
총 근무용량
= 기간 내 근무일 × 일 기준 근무시간 × FTE

유효 가용용량
= 총 근무용량 - 휴가·부재시간 - 고정 비프로젝트 일정시간

현재 할당량
= 해당 기간에 배분된 기존 업무 잔여 공수

잔여 가용시간
= max(0, 유효 가용용량 - 현재 할당량)

현재 부하율
= 현재 할당량 / 유효 가용용량 × 100

배정 후 부하율
= (현재 할당량 + 신규 Task 공수) / 유효 가용용량 × 100
```

### 16.2 WorkloadResult 필드

| 필드 | 타입 | 구분 | 설명 |
|---|---|---|---|
| `personId` | string | 필수 | 계산 대상 직원 |
| `periodStart` | datetime | 필수 | 계산 기간 시작 |
| `periodEnd` | datetime | 필수 | 계산 기간 종료 |
| `calculationMode` | enum | 필수 | `HOURS/POINTS/PROXY` |
| `grossCapacity` | decimal/null | 파생 | 총 근무용량 |
| `absenceDeduction` | decimal/null | 파생 | HR·Calendar 부재 차감 |
| `fixedScheduleDeduction` | decimal/null | 파생 | 고정 비프로젝트 일정 차감 |
| `effectiveCapacity` | decimal/null | 파생 | 유효 가용용량 |
| `currentAllocation` | decimal/null | 파생 | 기존 업무 할당량 |
| `remainingCapacity` | decimal/null | 파생 | 잔여 가용시간 |
| `currentLoadRate` | decimal/null | 파생 | 현재 부하율 |
| `projectedLoadRate` | decimal/null | 파생 | 배정 후 부하율 |
| `conflicts` | array | 파생 | 일정·휴가·마감 충돌 |
| `calculationEvidence` | array | 필수 | 사용한 Snapshot·정책·수식 버전 |
| `confidence` | enum/number | 필수 | 계산 신뢰도 |

유효 가용용량이 0이면 부하율을 계산하지 않고 null로 반환하며 해당 기간의 배정을 `BLOCKED` 처리한다.

## 17. 최소 실행 데이터셋

8월 6일 핵심 시연에서 업무 분배 필수 데이터가 준비되었다고 판단하려면 최소한 다음 데이터가 있어야 한다.

### 17.1 People DB

- 활성 직원 5명 이상
- 직원별 주 소속 조직과 역할
- 관리자 관계 또는 조직 상하위 관계
- 직원별 FTE 또는 적용 가능한 CapacityPolicy
- Skill 완전·부분·없음 사례
- 휴가 중복 사례
- Jira 매핑 성공·실패 사례

### 17.2 Jira

- 대상 프로젝트 1개 이상
- 활성 이슈 10건 이상
- 담당자·상태·최근 갱신 시각
- 시작일·마감일·잔여 공수 사례
- 공수 누락으로 PROXY 또는 대체 모드가 필요한 사례

### 17.3 신규 Task

- PM 확인이 완료된 Task 3건 이상
- 역할·기간·우선순위·공수
- 필수 Skill이 있는 Task와 없는 Task
- 문서 추출 Task와 PM 수정 Task
- 원문 Citation 또는 사용자 입력 근거

### 17.4 기대 결과

- 추천자와 최소 1명의 대안이 생성되는 사례
- 대안이 부족해 `PARTIAL_RESULT`가 되는 사례
- Skill 데이터 부재로 `UNKNOWN`이 되는 사례
- 휴가·기존 업무로 과부하 위험이 발생하는 사례
- 필수조건 위반으로 Validation `REJECT`가 발생하는 사례
