# A. 수행 자격 판단

## 이론적 근거: Person–Job Fit

Person–Job Fit은 업무가 요구하는 조건과 사람이 보유한 능력·자격이 얼마나 일치하는지를 판단하는 이론입니다.

이 단계에서는 “누가 가장 잘하는가”를 판단하지 않습니다.

> 해당 업무를 맡을 수 있는 최소 자격이 있는 사람만 후보로 남깁니다.
> 

## 판단해야 하는 것

- 필요한 직무에 해당하는가? O
- 필수 기술을 보유했는가? O
- 필요한 Job Profile이나 Job Family에 해당하는가? X
- 요구되는 직급·레벨을 충족하는가? O
- 해당 시스템이나 업무에 필요한 권한이 있는가? △
- 특정 조직 소속 조건이 있는가? O
- 프로젝트 기간에 장기 휴직하지 않는가? O
- 고용 형태상 프로젝트 참여가 가능한가? X

## 필요한 업무 데이터

### 기획서

기획서에서 다음 내용을 찾습니다.

- 업무에 필요한 역할
- 요구 기술
- 관련 도메인
- 관련 시스템
- 필수 자격
- 기술적·보안적 제약
- 업무 위험도
- 프로젝트 참여 대상 조직

### Jira 신규 Task

신규 Task를 생성하기 전 내부 업무 모델 또는 Jira 생성 초안에서 다음을 사용합니다.

```
summary
description
issuetype
priority
labels
components
security
issuerestriction
```

- `description`: 업무 내용과 요구조건
- `components`: Backend, Frontend, 결제 API 등 업무 영역
- `labels`: 필요한 기술·도메인
- `security`, `issuerestriction`: 접근 제한 조건

## 판단 방식

이 단계는 점수 방식이 아니라 통과·탈락 방식입니다.

```

AND 필수 Skill 충족
AND 필수 권한 충족

→ 적격 후보

하나라도 필수조건 미충족
→ 후보 제외
```


# B. 업무 적합성 판단

## 이론적 근거 1: Person–Job Fit

수행 자격을 통과한 후보 중에서 업무 요구와 개인의 역량·경험이 얼마나 잘 맞는지 비교합니다.

## 이론적 근거 2: Transactive Memory System

Transactive Memory System은 조직이 “누가 무엇을 알고 있는가”를 활용해 업무를 적절한 사람에게 연결한다는 이론입니다.

단순히 Java Skill을 가진 사람과 회사의 특정 결제 시스템을 실제로 다뤄본 사람은 구분해야 합니다.

## 판단해야 하는 것

- 업무 요구 기술과 보유 Skill이 얼마나 일치하는가? O
- 동일하거나 유사한 업무를 수행했는가? O (유사도 판단)
- 동일 Component 경험이 있는가? X
- 동일 프로젝트나 제품 경험이 있는가? X
- 관련 시스템의 맥락을 알고 있는가? X
- 관련 코드·설계·테스트 업무 경험이 있는가? X
- 고난도 업무를 수행하거나 검토한 경험이 있는가? X
- 주 담당자와 검토자 중 어떤 역할이 적합한가? X

# C. 기존 프로젝트 업무량과 부재 판단

## 이론적 근거: Job Demands–Resources

Job Demands–Resources 이론은 개인에게 요구되는 업무량과 개인이 사용할 수 있는 시간·자원 사이의 균형을 판단합니다.

후보가 업무에 가장 적합하더라도 프로젝트 A에서 이미 많은 업무를 맡고 있다면 프로젝트 B의 담당자로는 적절하지 않을 수 있습니다.

## 판단해야 하는 것

- 후보가 기존 프로젝트에서 맡은 업무는 무엇인가? O
- 각 업무는 얼마나 남아 있는가? O
- 기존 업무의 마감일은 언제인가? O
- 기존 업무가 핵심 경로에 포함되는가? △
- 신규 프로젝트 기간에 휴가가 있는가? O
- 기존 검토·승인 책임이 있는가? X
- 신규 업무를 맡을 수 있는 잔여 시간이 있는가? O

주요 사용 목적:

- `project`: 참여 중인 프로젝트
- `status`, `statusCategory`: 진행 중·완료 여부
- 시작일·`duedate`: 업무가 차지하는 기간
- `timeestimate`: 현재 잔여 공수
- `timeoriginalestimate`: 원래 예상 공수

- `issuelinks`: 핵심 의존 업무 여부

# D. 기간별 배정 가능성 판단

## 이론적 근거: Resource-Constrained Project Scheduling

이 방법론은 업무 간 순서와 사람의 제한된 시간을 함께 고려하여 실제로 실행 가능한 배정안을 만듭니다.

후보의 전체 잔여시간이 충분하더라도 신규 Task가 수행돼야 하는 시점에 시간이 없으면 배정할 수 없습니다.

## 판단해야 하는 것

- 신규 Task를 수행해야 하는 기간은 언제인가? O
- 후보가 해당 기간에 실제 Capacity를 가지고 있는가? X
- 기존 프로젝트 A의 일정과 겹치는가? O
- 동일한 사람이 동시에 두 Task를 수행해야 하는가? O
- 신규 Task의 시작일을 조정할 수 있는가? X
- Task를 분할할 수 있는가? O
- 다른 후보에게 배정하면 병렬 실행이 가능한가? O
- 검토자도 필요한 시점에 가용한가? X

## 필요한 신규 업무 데이터

### 기획서 및 신규 Task 초안

- 업무 시작 가능 시점
- 목표 완료일
- 예상 공수
- 예상 기간
- 선행 업무
- 후행 업무
- 업무 분할 가능 여부
- 연속 작업 필요 여부
- 핵심 경로 포함 여부

# E. 프로젝트 간 우선순위와 충돌 조정

## 이론적 근거: Project Portfolio Management

Project Portfolio Management는 각각의 프로젝트를 독립적으로 보지 않고, 여러 프로젝트의 우선순위와 자원 요구를 함께 조정하는 방법론입니다.

프로젝트 A와 B가 동일한 사람과 기간을 필요로 할 때 어떤 프로젝트를 우선할지 결정해야 합니다.

## 판단해야 하는 것

- A와 B 중 어느 프로젝트가 더 중요한가? O
- 마감일은 고정인가? X
- 일정 변경이 가능한가? X
- 지연 시 사업적 영향은 무엇인가? X
- 계약·법적 마감이 있는가? X
- 특정 인력이 반드시 필요한가? X
- 대체 인력이 있는가? X
- 업무 범위를 줄일 수 있는가? X
- 기존 프로젝트의 담당자 변경이 가능한가? X

## 필요한 프로젝트 데이터

### 기획서

- 신규 프로젝트 B의 우선순위
- 목표 완료일
- 고정 마감 여부
- 지연 시 영향
- 범위 조정 가능 여부
- 핵심 이해관계자
- 필수 인력 조건

### Jira 프로젝트 A

- `priority`
- `duedate`
- `status`
- `timeestimate`
- `issuelinks`
- 프로젝트 또는 Epic 수준의 우선순위
- 핵심 경로 업무
- 현재 담당자

### Jira 프로젝트 B

생성 초안 기준:

- 프로젝트 우선순위
- Epic 우선순위
- 목표 완료일
- 핵심 경로
- 담당 후보

## 충돌 처리

### A 우선

```
A의 기존 배정 유지
→ B의 차순위 후보 탐색
→ B 일정 또는 범위 조정
```

### B 우선

```
A의 변경안과 지연 영향을 계산
→ 자동 변경하지 않음
→ 관리자 승인 요청
```

### 동일 우선순위

```
전체 프로젝트 지연 최소화
핵심 경로 충돌 최소화
대체 후보 활용
```

## 출력

```
프로젝트 간 충돌 목록
권장 우선순위
대체 담당자
일정 변경안
Task 분할안
관리자 승인 필요 여부
```

# F. 업무 집중과 병목 검증

## 이론적 근거 1: Organizational Justice

Organizational Justice는 업무 부담과 기회가 특정 사람에게 부당하게 집중되지 않는지 판단합니다.

## 이론적 근거 2: Transactive Memory System

전문성을 활용하되, 모든 핵심 업무가 한 사람에게만 의존하지 않도록 해야 합니다.

## 판단해야 하는 것

- 특정인에게 전체 프로젝트 공수가 집중됐는가? O
- 동일인이 A와 B의 핵심 경로를 동시에 맡는가? O
- 고난도·고위험 업무가 한 사람에게 몰렸는가? X
- 동일인이 너무 많은 업무의 검토자인가? X
- 특정 Skill 보유자 한 명에게만 의존하는가? O
- 프로젝트 전환이 지나치게 빈번한가? X
- 주 담당자가 없을 때 대신할 사람이 있는가? X

# G. 담당자와 검토자 최종 결정

## 이론적 근거: Human-in-the-loop

시스템에 기록되지 않은 팀 사정과 암묵지가 존재하기 때문에 AI가 자동 확정하지 않고 책임자가 검토·승인해야 합니다.

## 최종 판단 순서

```
1. 필수 자격을 충족했는가?

2. 업무에 필요한 역할·Skill·경험이 적합한가?

3. 기존 프로젝트와 휴가를 제외하고 해당 기간에 Capacity가 있는가?

4. 배정 후 기존 프로젝트 일정이 침해되지 않는가?

5. 신규 프로젝트 핵심 경로를 지킬 수 있는가?

6. 특정 사람에게 핵심 업무와 검토 책임이 집중되지 않는가?

7. 프로젝트 간 우선순위 원칙에 부합하는가?
```

# 최종 판단식

필수조건을 먼저 적용합니다.

```
필수 역할 충족
AND 필수 Skill 충족
AND 권한·자격 충족
AND 프로젝트 기간 참여 가능
→ 적격 후보
```

적격 후보만 비교합니다.

```
담당자 적합성 =
역할·Skill 적합성
+ 유사 업무 경험
+ 프로젝트·시스템 맥락
+ 해당 기간 잔여 Capacity
- 기존 프로젝트 일정 충돌
- 프로젝트 전환 부담
- 핵심 업무 집중 위험
```

최종 배정은 다음 조건을 만족해야 합니다.

```
업무에 적합함
AND 해당 기간에 수행 가능함
AND 기존 프로젝트 약속을 침해하지 않음
AND 신규 프로젝트 일정을 달성할 수 있음
AND 특정인 병목이 과도하지 않음
```

한 문장으로 정리하면 다음과 같습니다.

> **먼저 Person–Job Fit을 통해 업무 수행 자격을 필터링하고, Person–Job Fit과 Transactive Memory System으로 역할·Skill·유사 경험·시스템 맥락을 평가한다. 다음으로 Jira의 기존 업무와 Workday의 휴가·휴직을 이용해 기간별 잔여 Capacity를 계산하고, Resource-Constrained Scheduling으로 실제 배정 가능한 시점을 결정한다. 프로젝트 간 충돌은 Portfolio Management로 조정하고, 업무 집중과 인력 병목은 Organizational Justice 관점에서 검증한 뒤 팀장 승인 후 Jira에 담당자를 반영한다.**
