# AI 기반 프로젝트 업무 추출·분배 모델링 기획 (v2)

> 변경 이력 (v2)
> - 용어 통일: 분해 = **업무 추출 Agent**, 담당자 추천 = **업무 분배 Agent** (잔여 표현 정리)
> - 팀장 확인 게이트 2단계 확정 반영: **추출 → 1차 확인**, **분배·검증 → 2차 확인**
> - 신규 섹션 추가: 데이터 준비 상태 확인(Feature Readiness), Snapshot·Identity Mapping, Task 출처 구분, 업무량 계산 결합(WorkloadResult), 검증 Agent
> - 기준 문서: 기획서 v5 (AI 프로젝트 운영 코파일럿)

## 1. 모델링 목표

본 시스템은 대규모 프로젝트가 팀에 배정되었을 때 프로젝트 관련 문서, 기존 Jira 업무 이력 및 인사 데이터를 종합하여 프로젝트를 실행 가능한 업무 단위로 분해하고 적합한 담당자를 추천하는 것을 목표로 한다.

AI가 생성한 결과는 업무 추출·분배 초안으로 제공하며, 팀장의 검토와 승인을 거쳐 최종 반영한다. 팀장 확인은 두 번 수행한다.

1. **1차 확인 — 업무 추출 직후**: AI가 분해한 업무 구조를 승인·수정·반려한다. 1차 확인 전에는 담당자 추천을 실행하지 않는다.
2. **2차 확인 — 업무 분배·검증 후**: 담당자 추천안과 검증 결과를 보고 최종 승인·수정·반려한다.

프로젝트 진행 중 관련 문서가 수정된 경우에는 재분배 기능을 통해 변경 내용을 다시 분석하고, 영향을 받는 업무와 담당자 배정안을 갱신한다. 재분배 결과도 동일하게 두 번의 확인을 거친다.

## 2. 활용 데이터

### 2.1 프로젝트 관련 비정형 문서

Google Drive에 저장된 다음과 같은 문서를 활용한다.

- 프로젝트 기획서, 요구사항 정의서, 회의록, 프로젝트 계획서
- 기술 설계서, 일정 및 마일스톤 자료, 의사결정 기록, 리스크 및 제약사항 문서

비정형 문서에서는 프로젝트의 목표, 범위, 요구사항, 일정, 제약조건, 완료 조건 및 담당 영역 등의 정보를 추출한다.

### 2.2 Jira 업무 데이터

기존 Jira 업무 데이터에서는 다음 정보를 활용한다.

- 과거 업무 내용 및 유형, 업무별 담당자, 업무 처리 상태 및 결과
- 유사 프로젝트의 업무 추출 사례, 담당자별 업무 수행 이력

신규 업무와 과거 Jira 업무의 유사성을 비교하여 업무 분해 및 담당자 추천의 근거로 활용한다.

### 2.3 인사 데이터

인사 데이터에서는 소속 팀, 직무 및 역할, 보유 기술, 프로젝트 경험, 업무 가능 여부 및 가용성을 활용한다. 이를 통해 각 업무의 요구 역량과 구성원의 역량을 비교하여 담당자 후보를 산정한다.

1차 구현의 인사 데이터는 내부 PostgreSQL People DB에서 제공하며, 추출·분배 로직은 People Repository와 PersonSnapshot을 통해서만 접근한다(기획서 v5 4.2).

## 3. 전체 처리 흐름

```
프로젝트 문서 수집
→ 문서 파싱
→ 의미 기반 청킹
→ 메타데이터 생성
→ 임베딩 및 벡터 색인
→ 프로젝트 정보 클러스터링 및 Project Knowledge Model 구성
→ 분석 시점 Snapshot 고정
→ 데이터 준비 상태 확인 (Feature Readiness)
→ 프로젝트 업무 추출
→ 팀장 업무 확인 (1차: 승인·수정·재추출)
→ 업무량 계산 결과(WorkloadResult) 결합
→ 업무 분배 (담당자 추천)
→ 검증 Agent (PASS / CONDITIONAL_PASS / REJECT)
→ 팀장 최종 확인 (2차: 승인·수정·반려)
```

문서 수정 후 재분배 시에는 다음 흐름을 따른다.

```
수정 문서 재수집
→ 재파싱 및 재청킹
→ 임베딩·메타데이터 갱신
→ 기존 문서 정보와 변경 비교
→ 영향 업무 재추출
→ 팀장 업무 확인 (1차)
→ 담당자 배정 재계산
→ 검증 Agent
→ 팀장 최종 확인 (2차)
```

## 4. 문서 파싱

Google Drive에서 수집한 비정형 문서는 파일 형식에 적합한 파서를 통해 구조화한다. 파서는 문서의 의미를 직접 판단하지 않고, 다음과 같은 물리적 구성요소를 추출한다.

- 문서 제목, 장·절 제목과 제목 단계, 일반 문단
- 글머리표와 번호 목록, 표의 행과 열, 체크박스
- 링크와 강조 표현, 문서 내 텍스트 순서 및 위치

파싱된 결과는 각각 고유한 식별자를 가진 Block으로 구성한다.

```json
{
  "document_id": "doc-001",
  "blocks": [
    { "block_id": "b001", "type": "heading", "level": 1, "text": "회원 인증" },
    { "block_id": "b002", "type": "paragraph", "text": "카카오와 구글 로그인을 지원해야 한다." },
    { "block_id": "b003", "type": "list_item", "text": "기존 이메일 계정과 연결할 수 있어야 한다." }
  ]
}
```

파싱 단계에서 원문의 순서와 구조를 유지하여 이후 AI 분석 결과를 원문과 연결할 수 있도록 한다.

## 5. 문서 및 의미 유형 분류

문서 유형은 회사가 사전에 정의한 카테고리 중에서 LLM이 선택하도록 구성한다. LLM이 새로운 유형을 임의로 생성하지 않도록 제한한다.

- 문서 유형 예시: 기획서, 요구사항 문서, 회의록, 프로젝트 계획서, 기술 설계서, 일정 문서, 의사결정 기록, 리스크 문서, 기타, 분류 불가

하나의 문서 안에는 여러 성격의 내용이 함께 존재할 수 있으므로 문서 전체 유형과 개별 내용의 의미 유형을 별도로 분류한다.

- 청크 수준 의미 유형: 프로젝트 목표, 업무 범위, 요구사항, 결정 사항, 제약조건, 마일스톤, 의존관계, 리스크, 완료 조건, 담당 영역

예를 들어 회의록 안에 새로운 요구사항과 결정 사항이 함께 포함된 경우, 문서 유형은 회의록으로 분류하고 해당 청크에는 요구사항 또는 결정 사항이라는 의미 유형을 별도로 부여한다.

## 6. 청킹 전략

파싱된 문서를 일정한 글자 수로 단순 분할하면 하나의 업무에 포함된 요구사항, 담당 영역, 일정 및 완료 조건이 서로 분리될 수 있다. 따라서 문서의 물리적 구조와 업무상 의미를 함께 고려하여 청킹한다.

### 6.1 물리적 섹션 구성

제목과 문단의 구조를 기준으로 문서의 물리적 섹션을 구성한다. 제목 구조가 명확한 경우에는 동일한 제목 아래에 있는 문단, 목록 및 표를 하나의 물리적 섹션으로 묶는다.

### 6.2 의미적 그룹화

LLM은 파싱된 Block을 분석하여 업무상 의미가 동일한 Block들을 하나의 그룹으로 묶는다. LLM은 원문을 새로 작성하지 않고, 동일한 의미 단위에 속하는 Block 식별자를 그룹화한다.

```json
{
  "topic": "인증 API 개발",
  "block_ids": ["b010", "b011", "b012"],
  "semantic_types": ["요구사항", "담당 영역", "마일스톤"]
}
```

### 6.3 Parent·Child 청크 구성

의미적으로 그룹화된 내용을 Parent 청크로 구성한다.

- Parent 청크: 하나의 요구사항이나 프로젝트 계획 요소에 필요한 전체 문맥
- Child 청크: 벡터 검색을 위한 세부 검색 단위

검색은 작은 Child 청크를 대상으로 수행하고, 실제 AI 분석에서는 해당 Child가 속한 Parent 청크의 전체 문맥을 활용한다. 이 구조를 통해 검색 정확도를 확보하면서도 하나의 업무와 관련된 조건들이 분리되어 해석되는 문제를 줄인다.

## 7. 메타데이터 생성

문서와 청크에는 검색, 의미 분류, 원문 추적 및 변경 감지를 위한 메타데이터를 부여한다.

### 7.1 문서 메타데이터

프로젝트 식별자, 문서 식별자, 문서명, 문서 유형, 문서 상태, 문서 버전, 생성일 및 수정일, 원본 링크, 접근 권한, 콘텐츠 해시

### 7.2 청크 메타데이터

Parent 및 Child 청크 식별자, 원본 Block 식별자, 문서 내 섹션 경로, 의미 유형, 업무 영역, 기능, 관련 역할 및 기술, 담당자 또는 담당 조직 언급, 일정 및 마감일, 이전 버전 청크 식별자, 청크 콘텐츠 해시

### 7.3 업무 영역과 기능 추출

업무 영역(Business Scope)은 회사가 정의한 상위 업무 영역 중에서 LLM이 선택하도록 구성한다. (예: 회원, 상품, 주문, 결제, 배송, 정산, 인프라, 보안)

Feature는 Parent 청크의 원문에서 LLM이 추출한다. 같은 기능이 문서마다 다르게 표현된 경우 기존 기능명과 비교하여 표준 명칭으로 정규화한다.

```json
{
  "business_scope": "회원",
  "feature": "소셜 로그인",
  "sub_features": ["카카오 로그인", "구글 로그인", "기존 계정 연결"],
  "source_block_ids": ["b002", "b003"]
}
```

메타데이터로 추출된 값은 원본 Block을 근거로 연결하며, 근거가 부족한 경우에는 미분류 또는 빈 값으로 처리한다.

## 8. 임베딩 및 벡터 검색

검색용 Child 청크에는 원문과 함께 프로젝트 및 의미 정보를 추가한 후 임베딩을 생성한다. 원문과 임베딩용 텍스트는 별도로 관리한다.

- 원문: 결과의 근거를 확인하기 위한 용도
- 임베딩용 텍스트: 프로젝트와 업무 문맥을 포함한 벡터 검색 용도

Agent는 프로젝트 식별자와 접근 권한을 기준으로 검색 범위를 제한하고, 벡터 유사도와 메타데이터를 함께 활용한다. 벡터 검색은 다음 목적으로 사용한다.

- 여러 문서에 분산된 유사 요구사항 탐색
- 같은 기능을 다르게 표현한 문서 검색
- 요구사항과 관련된 기술·일정·제약조건 탐색
- 신규 업무와 유사한 과거 Jira 업무 검색

## 9. 프로젝트 정보 클러스터링

프로젝트 관련 내용은 여러 문서에 반복되거나 서로 다른 표현으로 분산될 수 있다. 이러한 내용을 문서별로 각각 업무로 변환하면 동일하거나 유사한 업무가 중복 생성될 수 있다. 따라서 업무 추출 전에 Parent 청크를 업무 영역, 기능, 의미적 유사성 등을 기준으로 그룹화한다.

클러스터링을 통해 다음 결과를 얻는다.

- 여러 문서에 분산된 동일 요구사항 통합
- 중복 업무 생성 방지
- 기능별 관련 조건과 결정 사항 통합
- 프로젝트를 업무 영역별로 분해하기 위한 기준 생성

```json
{
  "cluster": "소셜 로그인",
  "business_scope": "회원",
  "source_parent_ids": ["p-requirement-01", "p-meeting-03", "p-design-02"]
}
```

업무 추출 Agent는 개별 문서를 직접 업무로 변환하지 않고, 통합된 클러스터를 기준으로 업무를 분해한다.

## 10. 데이터 준비 상태 확인 (Feature Readiness)

업무 추출·분배는 항상 성공하는 경로가 아니다. Snapshot 생성 후 해당 기능이 요구하는 데이터 계약을 검사하고, 결과를 세 가지 상태로 판정한다(기획서 v5 10장).

| 상태 | 의미 | 처리 |
|---|---|---|
| SUCCESS | 필수 데이터 충족 | 전체 기능 실행 |
| PARTIAL_RESULT | 선택 데이터·일부 매핑 누락 | 제한·가정·낮은 신뢰도를 표시하고 가능한 범위만 실행 |
| BLOCKED | 필수 데이터 누락 | 해당 기능 실행 중단, 누락 항목과 해결 방법 안내 |

이 모델링과 직접 관련된 판정 예시는 다음과 같다.

- 문서 본문이 비어 있거나 접근 권한이 없으면 해당 문서는 BLOCKED로 제외한다.
- 후보자의 필수 Skill 데이터가 없으면 부적합이 아니라 **UNKNOWN**으로 처리하고 조건부 후보로 유지한다.
- Jira 계정 매핑에 실패한 인원은 기존 업무량을 알 수 없으므로, 해당 인원을 제외하지 않되 PARTIAL_RESULT 근거와 낮은 신뢰도를 기록한다.

모든 추출·분배 결과에는 missingData, limitations, assumptions, confidence를 포함한다.

## 11. 분석 시점 Snapshot 고정과 Identity Mapping

### 11.1 Snapshot

추출·분배 실행 직전에 입력 데이터를 불변 Snapshot으로 고정한다. Snapshot 생성 후 Feature Readiness를 검사하고 필수 입력이 준비된 기능만 실행한다(기획서 v5 11장).

- 대상: PersonSnapshot(인사), ExistingTaskSnapshot(Jira), DocumentIndex(문서·청크), 회사 정책, 팀장 추가 요구사항
- 분석이 시작되면 입력 Snapshot은 변경하지 않는다. 원천 데이터가 바뀌어도 기존 결과는 당시 snapshotAsOf를 유지한다.
- 모든 결과에 사용한 Snapshot 식별자·정책 버전·계산 버전을 기록하여 재현과 감사가 가능하게 한다.
- 재분배 시에는 새 Snapshot과 새 결과를 생성한다.

### 11.2 Identity Mapping

인사 데이터·Jira·Google 계정은 내부 personId를 기준으로 연결한다(기획서 v5 6장).

- 매핑 우선순위: 확정된 IdentityLink → HR employee_id와 사전 등록 외부 ID → 검증된 회사 이메일 완전 일치 → 복합 일치 후보 → 수동 확인
- 이름 단독 일치나 LLM 추정만으로 자동 확정하지 않는다.
- 매핑 실패는 전체 흐름을 중단하지 않고 부분 결과(PARTIAL_RESULT)로 처리한다. 단, 정책상 Jira 업무량이 필수인 기능은 BLOCKED다.

## 12. 업무 추출 Agent

업무 추출 Agent는 프로젝트 문서에서 생성된 Parent 청크와 클러스터링 결과를 Project Knowledge Model로 통합한 뒤 프로젝트 전체를 실행 가능한 업무 구조로 분해한다. Vector/RAG 검색은 유사 표현·과거 Jira 업무·추가 근거를 찾는 보조 입력으로만 사용한다.

업무 구조는 다음 계층으로 구성한다.

```
프로젝트 → 업무 영역 → 업무 패키지 → 실행 가능한 세부 업무
```

세부 업무는 다음 조건을 기준으로 분리한다.

- 담당 역할이 다른 경우, 서로 다른 산출물이 필요한 경우
- 완료 여부를 독립적으로 판단할 수 있는 경우, 선행·후행 의존관계가 있는 경우
- 서로 다른 기술 역량이 필요한 경우, 별도의 검토 또는 테스트가 필요한 경우

업무 추출 결과에는 업무명 및 설명, 상위 업무 영역, 필요 역할, 필요 기술, 선행·후행 업무, 완료 조건, 일정 및 우선순위, 원문 Parent 및 Block 근거가 포함된다. 모든 업무는 하나 이상의 원문 Parent 청크와 연결하여 생성 근거를 추적할 수 있도록 한다.

### 12.1 Task 출처 구분

추출된 모든 업무에는 출처 유형을 부여한다(기획서 v5 9.1).

| 유형 | 의미 | 처리 |
|---|---|---|
| EXTRACTED | 문서에 명시된 업무를 구조화 | 원문 근거 필수 |
| GENERATED | 상위 요구를 실행 가능한 하위 업무로 분해 | 상위 요구 근거와 생성 이유 필수 |
| AI_SUGGESTED_MISSING_TASK | 문서에 없지만 의존성·품질상 필요하다고 AI가 제안 | 자동 확정 금지, 1차 확인에서 별도 배지로 표시하고 팀장 승인 필수 |
| USER_ADDED | 팀장이 직접 추가 | 추가 사용자와 사유 기록 |

근거 없는 내용은 사실로 추출하지 않으며, 제안 업무 또는 가정(assumption)으로 분리한다.

## 13. 팀장 업무 확인 (1차 게이트)

업무 추출 직후 팀장이 분해 결과를 확인한다. **1차 확인 전에는 담당자 추천을 실행하지 않는다.** 잘못 분해된 업무를 기준으로 추천하는 오류를 차단하기 위한 단계다.

| 조치 | 처리 |
|---|---|
| 승인 | 초안을 불변 확정 Task(NewTaskSnapshot)로 고정 |
| 수정 | 업무명·역할·공수·일정·우선순위를 수정하고 이력 저장 후 확정 |
| 반려 | 문서 또는 분해 조건을 보완한 뒤 재추출 |

## 14. 업무량 계산 결합 (WorkloadResult)

담당자의 가용성과 부하는 LLM이 판단하지 않는다. 규칙·수식 기반 **업무량 계산 서비스**가 결정론적으로 계산하고, 업무 분배 Agent는 그 출력(WorkloadResult)을 입력으로만 사용한다(기획서 v5 13.1).

```
총 근무용량   = 기간 내 근무일 × 일 기준 근무시간 × FTE
유효 가용용량 = 총 근무용량 − 휴가·부재시간 − 고정 비프로젝트 일정시간
현재 할당량   = 해당 기간에 배분된 기존 업무 잔여시간
잔여 가용시간 = max(0, 유효 가용용량 − 현재 할당량)
현재 부하율   = 현재 할당량 / 유효 가용용량 × 100
배정 후 부하율 = (현재 할당량 + 신규 배정시간) / 유효 가용용량 × 100
```

- WorkloadResult에는 총 근무용량, 유효 가용용량, 잔여 가용시간, 현재·배정 후 부하율, 일정 충돌, 계산 근거가 포함된다.
- 유효 가용용량이 0이면 부하율을 계산하지 않고 해당 기간 배정을 BLOCKED 처리한다.
- 시간과 Story Point는 직접 합산하지 않으며, 계산 방식별 Readiness를 먼저 판정한다.
- LLM의 역할은 계산 결과를 팀장이 이해할 수 있는 설명으로 변환하는 것까지다.

## 15. 업무 분배 Agent

업무 분배 Agent는 **확정 Task(1차 확인 완료)** 와 **WorkloadResult** 를 입력으로 받아, 기존 Jira 업무 이력과 인사 데이터를 비교하여 담당자를 추천한다.

담당자 추천 시 다음 항목을 고려한다.

- 직무 및 역할 적합도, 업무 요구 기술과 보유 기술의 일치도
- 과거 유사 Jira 업무 수행 경험, 동일 프로젝트 또는 시스템 경험
- 현재 업무 가용성과 부하율 (WorkloadResult 기반)
- 프로젝트 전체 업무의 편중도

업무별 최고 점수의 담당자만 개별적으로 선택하면 특정 인력에게 업무가 집중될 수 있으므로, 프로젝트 전체 업무와 전체 후보자를 함께 고려해 배정안을 생성한다.

배정 결과에는 추천 담당자, 대체 담당자 후보, 추천 점수, 직무·기술 적합 근거, 과거 유사 업무 수행 근거, 예상 부하율, 누락 데이터·가정·신뢰도를 포함한다.

## 16. 검증 Agent

업무 분배 초안은 팀장에게 전달되기 전에 검증 Agent를 통과해야 한다. 검증은 규칙 기반 체크리스트로 수행하며 LLM에 최종 판정을 맡기지 않는다(기획서 v5 14.3).

필수 검사 항목: 역할·필수 Skill·책임수준, 조직 제한과 권한, 휴가·휴직 중복, 일정 충돌과 최대 부하율, 추천 근거 존재 여부, 원본 출처 접근 가능 여부, 데이터 누락·가정·신뢰도.

| 결과 | 의미 | 후속 |
|---|---|---|
| PASS | 필수조건 충족, 근거 충분 | 팀장 2차 확인으로 전달 |
| CONDITIONAL_PASS | UNKNOWN 또는 선택 데이터 누락 | 제한 사항을 표시한 채 팀장 2차 확인으로 전달 |
| REJECT | 필수조건 위반 또는 근거 없음 | 팀장 승인 대상에서 제외, 데이터·조건 수정 후 재분배 |

REJECT 결과는 Jira에 반영하지 않는다.

## 17. 팀장 최종 확인 (2차 게이트)

검증을 통과한 추천안에 대해 팀장이 최종 결정한다.

| 조치 | 처리 |
|---|---|
| 승인 | DecisionRecord 저장 (Jira 반영은 후속 단계) |
| 수정 | 담당자·기간·공수 변경 후 검증 Agent로 직접 재전달하여 재검증 |
| 반려 | 사유에 따라 회귀: 분해 오류 → 업무 추출, 조건 변경 → 업무 분배, 데이터 오류 → 데이터 수집 |

## 18. 재분배 처리

프로젝트 관련 문서가 수정되고 재분배가 실행되면 변경된 문서를 다시 파싱·청킹하고 임베딩 및 메타데이터를 갱신한다. 재분배 시에도 새 Snapshot을 생성한다.

문서 및 Parent 청크의 버전과 콘텐츠 해시를 비교하여 신규 추가·수정·삭제·변경 없음을 식별하고, 변경된 Parent 청크가 기존 업무의 근거로 사용되었는지 확인하여 영향을 받는 업무를 식별한다.

업무 변경은 업무 추가, 업무 수정, 업무 삭제, 업무 분할, 업무 병합, 변경 없음으로 구분한다. 재추출된 업무는 팀장 1차 확인을 다시 거친다.

변경된 요구사항으로 인해 업무의 역할이나 필요 기술이 달라진 경우에는 담당자 적합도를 다시 계산한다. 기존 업무를 그대로 수행할 수 있는 경우에는 업무 연속성을 고려하여 기존 담당자를 유지하는 방향으로 추천한다.

재분배 결과도 새로운 초안으로 생성하며, 검증 Agent와 팀장 2차 확인을 거쳐 최종 반영한다.

## 19. 모델링 원칙

1. 문서를 고정 길이로 단순 분할하지 않고 구조와 의미를 함께 고려한다.
2. 검색은 Child 청크를 대상으로 하고 실제 분석은 Parent 문맥을 활용한다.
3. 프로젝트 문서를 개별적으로 업무화하지 않고 클러스터링을 통해 중복 내용을 통합한다.
4. 업무 영역은 사전 정의된 분류체계를 사용하고 기능은 원문에서 추출해 정규화한다.
5. 모든 업무와 담당자 추천은 원본 문서 또는 기존 데이터의 근거와 연결한다.
6. 시간·가용성·부하율·필수조건은 규칙과 수식으로 계산하며 LLM에 최종 판정을 맡기지 않는다 (Bounded LLM).
7. 데이터가 부족해도 가능한 범위의 결과를 제공하고, 누락·제한·가정·신뢰도를 함께 표시한다 (SUCCESS / PARTIAL_RESULT / BLOCKED).
8. 추출·분배 입력은 불변 Snapshot으로 고정하여 재현과 감사가 가능하게 한다.
9. 문서 변경 시 전체 결과를 무조건 재생성하지 않고 변경 영향을 중심으로 재분배한다.
10. AI는 추출·분배 초안을 생성하며, 업무 추출 직후(1차)와 최종 반영 전(2차) 두 번의 팀장 확인을 거친다.
