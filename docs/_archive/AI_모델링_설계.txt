AI 기반 프로젝트 업무 추출·배정 모델링 기획
1. 모델링 목표
본 시스템은 대규모 프로젝트가 팀에 배정되었을 때 프로젝트 관련 문서, 기존 Jira 업무 이력 및 인사 데이터를 종합하여 프로젝트를 실행 가능한 업무 단위로 분배하고 적합한 담당자를 추천하는 것을 목표로 한다.
AI가 생성한 결과는 업무 추출·배정 초안으로 제공하며, 팀장의 검토와 승인을 거쳐 최종 반영한다.
프로젝트 진행 중 관련 문서가 수정된 경우에는 재분배 기능을 통해 변경 내용을 다시 분석하고, 영향을 받는 업무와 담당자 배정안을 갱신한다.

4. 문서 파싱
Google Drive에서 수집한 비정형 문서는 파일 형식에 적합한 파서를 통해 구조화한다.
파서는 문서의 의미를 직접 판단하지 않고, 다음과 같은 물리적 구성요소를 추출한다.
문서 제목
장·절 제목과 제목 단계
일반 문단
글머리표와 번호 목록
표의 행과 열
체크박스
링크와 강조 표현
문서 내 텍스트 순서 및 위치
파싱된 결과는 각각 고유한 식별자를 가진 Block으로 구성한다.
예시는 다음과 같다.
{
  "document_id": "doc-001",
  "blocks": [
    {
      "block_id": "b001",
      "type": "heading",
      "level": 1,
      "text": "회원 인증"
    },
    {
      "block_id": "b002",
      "type": "paragraph",
      "text": "카카오와 구글 로그인을 지원해야 한다."
    },
    {
      "block_id": "b003",
      "type": "list_item",
      "text": "기존 이메일 계정과 연결할 수 있어야 한다."
    }
  ]
}
파싱 단계에서 원문의 순서와 구조를 유지하여 이후 AI 분석 결과를 원문과 연결할 수 있도록 한다.

5. 문서 및 의미 유형 분류
문서 유형은 회사가 사전에 정의한 카테고리 중에서 LLM이 선택하도록 구성한다. LLM이 새로운 유형을 임의로 생성하지 않도록 제한한다.
문서 유형의 예시는 다음과 같다.
기획서
요구사항 문서
회의록
프로젝트 계획서
기술 설계서
일정 문서
의사결정 기록
리스크 문서
기타
분류 불가
하나의 문서 안에는 여러 성격의 내용이 함께 존재할 수 있으므로 문서 전체 유형과 개별 내용의 의미 유형을 별도로 분류한다.
청크 수준의 의미 유형은 다음과 같이 구성한다.
프로젝트 목표
업무 범위
요구사항
결정 사항
제약조건
마일스톤
의존관계
리스크
완료 조건
담당 영역
예를 들어 회의록 안에 새로운 요구사항과 결정 사항이 함께 포함된 경우, 문서 유형은 회의록으로 분류하고 해당 청크에는 요구사항 또는 결정 사항이라는 의미 유형을 별도로 부여한다.

6. 청킹 전략
파싱된 문서를 일정한 글자 수로 단순 분할하면 하나의 업무에 포함된 요구사항, 담당 영역, 일정 및 완료 조건이 서로 분리될 수 있다.
따라서 문서의 물리적 구조와 업무상 의미를 함께 고려하여 청킹한다.
6.1 물리적 섹션 구성
제목과 문단의 구조를 기준으로 문서의 물리적 섹션을 구성한다.
제목
├─ 상위 섹션
│  ├─ 하위 섹션
│  ├─ 문단
│  └─ 표
└─ 상위 섹션
제목 구조가 명확한 경우에는 동일한 제목 아래에 있는 문단, 목록 및 표를 하나의 물리적 섹션으로 묶는다.
6.2 의미적 그룹화
LLM은 파싱된 Block을 분석하여 업무상 의미가 동일한 Block들을 하나의 그룹으로 묶는다.
예를 들어 다음 내용은 서로 다른 문단으로 작성되어 있어도 하나의 의미 단위로 구성한다.
인증 API를 개발한다.
플랫폼팀이 담당한다.
9월 30일까지 완료한다.
LLM은 원문을 새로 작성하지 않고, 동일한 의미 단위에 속하는 Block 식별자를 그룹화한다.
{
  "topic": "인증 API 개발",
  "block_ids": ["b010", "b011", "b012"],
  "semantic_types": [
    "요구사항",
    "담당 영역",
    "마일스톤"
  ]
}
6.3 Parent·Child 청크 구성
의미적으로 그룹화된 내용을 Parent 청크로 구성한다.
Parent 청크: 하나의 요구사항이나 프로젝트 계획 요소에 필요한 전체 문맥
Child 청크: 벡터 검색을 위한 세부 검색 단위
검색은 작은 Child 청크를 대상으로 수행하고, 실제 AI 분석에서는 해당 Child가 속한 Parent 청크의 전체 문맥을 활용한다.
이 구조를 통해 검색 정확도를 확보하면서도 하나의 업무와 관련된 조건들이 분리되어 해석되는 문제를 줄인다.

7.3 업무 영역과 기능 추출
업무 영역인 Business Scope는 회사가 정의한 상위 업무 영역 중에서 LLM이 선택하도록 구성한다.
업무 영역의 예시는 다음과 같다.
회원
상품
주문
결제
배송
정산
인프라
보안
Feature는 Parent 청크의 원문에서 LLM이 추출한다. 같은 기능이 문서마다 다르게 표현된 경우 기존 기능명과 비교하여 표준 명칭으로 정규화한다.
예시는 다음과 같다.
{
  "business_scope": "회원",
  "feature": "소셜 로그인",
  "sub_features": [
    "카카오 로그인",
    "구글 로그인",
    "기존 계정 연결"
  ],
  "source_block_ids": ["b002", "b003"]
}
메타데이터로 추출된 값은 원본 Block을 근거로 연결하며, 근거가 부족한 경우에는 미분류 또는 빈 값으로 처리한다.

8. 임베딩 및 벡터 검색
검색용 Child 청크에는 원문과 함께 프로젝트 및 의미 정보를 추가한 후 임베딩을 생성한다.
예시는 다음과 같다.
프로젝트: 쇼핑몰 구축
문서 유형: 요구사항 정의서
업무 영역: 회원
기능: 소셜 로그인
의미 유형: 요구사항, 완료 조건

카카오와 구글 로그인을 지원하며 기존 이메일 계정과 연결할 수 있어야 한다.
원문과 임베딩용 텍스트는 별도로 관리한다.
원문: 결과의 근거를 확인하기 위한 용도
임베딩용 텍스트: 프로젝트와 업무 문맥을 포함한 벡터 검색 용도
Agent는 프로젝트 식별자와 접근 권한을 기준으로 검색 범위를 제한하고, 벡터 유사도와 메타데이터를 함께 활용한다.
벡터 검색은 다음 목적으로 사용한다.
여러 문서에 분산된 유사 요구사항 탐색
같은 기능을 다르게 표현한 문서 검색
요구사항과 관련된 기술·일정·제약조건 탐색
신규 업무와 유사한 과거 Jira 업무 검색

9. 프로젝트 정보 클러스터링
프로젝트 관련 내용은 여러 문서에 반복되거나 서로 다른 표현으로 분산될 수 있다. 이러한 내용을 문서별로 각각 업무로 변환하면 동일하거나 유사한 업무가 중복 생성될 수 있다.
따라서 프로젝트 분배 전에 Parent 청크를 업무 영역, 기능, 의미적 유사성 등을 기준으로 그룹화한다.
클러스터링 대상은 다음과 같다.
동일 프로젝트에 포함된 Parent 청크
여러 문서에 반복된 동일·유사 요구사항
같은 기능과 관련된 결정 사항, 제약조건 및 완료 조건
하나의 업무 영역에 속하는 관련 문서 내용
클러스터링을 통해 다음 결과를 얻는다.
여러 문서에 분산된 동일 요구사항 통합
중복 업무 생성 방지
기능별 관련 조건과 결정 사항 통합
프로젝트를 업무 영역별로 분해하기 위한 기준 생성
예를 들어 요구사항 문서, 회의록 및 기술 문서에 소셜 로그인 관련 내용이 각각 존재한다면 이를 하나의 기능 클러스터로 통합한다.
{
  "cluster": "소셜 로그인",
  "business_scope": "회원",
  "source_parent_ids": [
    "p-requirement-01",
    "p-meeting-03",
    "p-design-02"
  ]
}
업무 추출 Agent는 개별 문서를 직접 업무로 변환하지 않고, 통합된 클러스터를 기준으로 업무를 분해한다.

10. 업무 추출 Agent
업무 추출 Agent는 프로젝트 문서에서 생성된 Parent 청크와 클러스터링 결과를 이용하여 프로젝트 전체를 실행 가능한 업무 구조로 분해한다.
업무 구조는 다음 계층으로 구성한다.
프로젝트
→ 업무 영역
→ 업무 패키지
→ 실행 가능한 세부 업무
세부 업무는 다음 조건을 기준으로 분리한다.
담당 역할이 다른 경우
서로 다른 산출물이 필요한 경우
완료 여부를 독립적으로 판단할 수 있는 경우
선행·후행 의존관계가 있는 경우
서로 다른 기술 역량이 필요한 경우
별도의 검토 또는 테스트가 필요한 경우
업무 추출 결과에는 다음 정보가 포함된다.
업무명 및 설명
상위 업무 영역
필요 역할
필요 기술
선행·후행 업무
완료 조건
일정 및 우선순위
원문 Parent 및 Block 근거
모든 업무는 하나 이상의 원문 Parent 청크와 연결하여 생성 근거를 추적할 수 있도록 한다.

11. 업무 분배 Agent
업무 분배 Agent는 업무 추출 Agent가 생성한 업무별 요구 역할 및 기술을 기존 Jira 업무 이력과 인사 데이터에 비교하여 담당자를 추천한다.
담당자 추천 시 다음 항목을 고려한다.
직무 및 역할 적합도
업무 요구 기술과 보유 기술의 일치도
과거 유사 Jira 업무 수행 경험
동일 프로젝트 또는 시스템 경험
현재 업무 가용성
프로젝트 전체 업무의 편중도
업무별 최고 점수의 담당자만 개별적으로 선택하면 특정 인력에게 업무가 집중될 수 있으므로, 프로젝트 전체 업무와 전체 후보자를 함께 고려해 배정안을 생성한다.
배정 결과에는 다음 정보를 포함한다.
추천 담당자
대체 담당자 후보
추천 점수
직무·기술 적합 근거
과거 유사 업무 수행 근거
Agent는 업무 추출와 담당자 배정 초안을 생성하며, 최종 반영 전 팀장의 검토 및 승인을 받는다.

12. 재분배 처리
프로젝트 관련 문서가 수정되고 재분배가 실행되면 변경된 문서를 다시 파싱·청킹하고 임베딩 및 메타데이터를 갱신한다.
문서 및 Parent 청크의 버전과 콘텐츠 해시를 비교하여 다음 변경을 식별한다.
신규 내용 추가
기존 내용 수정
기존 내용 삭제
변경 없음
변경된 Parent 청크가 기존에 생성된 업무의 근거로 사용되었는지 확인하여 영향을 받는 업무를 식별한다.
업무 변경은 다음과 같이 구분할 수 있다.
업무 추가
업무 수정
업무 삭제
업무 분할
업무 병합
변경 없음
변경된 요구사항으로 인해 업무의 역할이나 필요 기술이 달라진 경우에는 담당자 적합도를 다시 계산한다. 기존 업무를 그대로 수행할 수 있는 경우에는 업무 연속성을 고려하여 기존 담당자를 유지하는 방향으로 추천한다.
재분배 결과도 새로운 초안으로 생성하며, 팀장의 검토와 승인을 거쳐 최종 반영한다.

13. 모델링 원칙
본 모델링은 다음 원칙을 따른다.
문서를 고정 길이로 단순 분할하지 않고 구조와 의미를 함께 고려한다.
검색은 Child 청크를 대상으로 하고 실제 분석은 Parent 문맥을 활용한다.
프로젝트 문서를 개별적으로 업무화하지 않고 클러스터링을 통해 중복 내용을 통합한다.
업무 영역은 사전 정의된 분류체계를 사용하고 기능은 원문에서 추출해 정규화한다.
모든 업무와 담당자 추천은 원본 문서 또는 기존 데이터의 근거와 연결한다.
문서 변경 시 전체 결과를 무조건 재생성하지 않고 변경 영향을 중심으로 재분배한다.
AI는 분배·배정 초안을 생성하며 최종 결정은 팀장 승인을 통해 이루어진다.
