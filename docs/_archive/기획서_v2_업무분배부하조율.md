# AI 업무 분배·부하 조율 멀티에이전트 플랫폼 — 기획서 (v2)

> **방향 전환 요약**
> 기존 기획("메일·회의·보고서를 통합하는 취합형 그룹웨어")은 멘토링(7/15)·팀 회의(7/16)를 거쳐 폐기했다.
> 여러 기능을 나열하던 구조가 "그룹웨어 하나 더"로 보였고, "왜 새로 만드는가"에 답하지 못했다.
> **본 기획서는 문제를 "업무 분배의 불공정성·불투명성" 하나로 좁히고, 나머지 모든 기능(메일·메신저·보고서·캘린더·회의록·워크스페이스)을 그 문제를 풀기 위한 "데이터 수집 계층"으로 재배치한다.**

---

## 1. 프로젝트 개요

### 프로젝트명
**AI 업무 분배·부하 조율 멀티에이전트 플랫폼**

### 한 줄 소개
기존 업무 활동(회의록·기획서·메일·캘린더·업무 로그)에서 **새로운 보고 없이** 데이터를 자동 수집·구조화하여, 구성원의 **업무량·역할·가용량을 객관적으로 파악**하고, 이를 근거로 **공정한 업무 배분과 과부하 감지·재배분**을 제안하는 다중 에이전트 시스템이다.

### 핵심 한 문장 (발표용)
> "누가 얼마나 바쁜지 아무도 정확히 모른다"는 문제를, **기존 업무 흔적에서 부하를 자동 산출**하는 멀티에이전트로 해결한다. 우리는 그룹웨어를 만드는 것이 아니라, **업무 분배의 판단 근거를 만든다.**

---

## 2. 문제 정의

프로젝트 협업에서 구성원의 **업무량과 역할을 객관적으로 파악하기 어려워**, 업무가 일부 인원에게 집중되거나 중복·누락되는 문제가 반복된다. 이는 개인 역량 문제가 아니라 **구조적 문제**다.

- 업무를 누구에게 어떻게 배분할지 **기준이 모호**하다.
- **역할과 책임(R&R)**이 명확하지 않다.
- 업무가 **일부 숙련자에게 집중**된다.
- **업무량을 객관적으로 파악**하기 어렵다.
- 그 결과 갈등, 생산성 저하, 번아웃, 이직이 발생한다.

### 근거 (통계)

| 지표 | 수치 |
| --- | ---: |
| 업무 분담이 가장 애매한 회사 문제라고 응답 | 73.4% |
| 업무 분배가 공정하다고 느끼는 사람 | 7.9% |
| 업무 분배가 불공정하다고 인식 | 약 40% |
| 역할 명확성이 높은 조직의 일정 준수율 | 2.5배 |
| 역할 불명확 시 생산성 손실 | 15~25% |
| 업무 조율에 쓰는 시간 (Asana) | 업무시간의 약 60% |
| 중복 업무 | 연 209시간 |
| 과도한 업무로 프로젝트 지연 경험 | 88% |

> ※ 통계와 실무 사례(LinkedIn·커뮤니티 등)는 발표 시 구분해 제시한다. 해외 자료 의존 수치는 출처를 명기한다.

### 문제의 핵심

> **"업무를 얼마나 가지고 있는지"보다, "누가 어떤 책임을 지고 있고 얼마나 더 수행 가능한지"를 알기 어렵다는 것**이 핵심이다. 리더는 실제 업무량을 보지 못하고, 이를 파악하려 **불필요한 보고 업무만 늘어난다.**

## 3. 솔루션 개요

### 핵심 아이디어
**새로운 보고를 추가하지 않는다.** 대신 이미 존재하는 업무 활동에서 데이터를 자동으로 수집·구조화해 **업무량·역할·가용량을 객관 수치화**하고, 그 위에서 **배분을 제안**하며, 진행 중 **과부하를 감지해 재배분을 제안**한다. 모든 최종 결정은 사람(팀장)이 한다.

### 세 가지 설계 원칙 (문제정의 인사이트 반영)
1. **공정 = 균등이 아니다.** 업무량·난이도·중요도·숙련도·가용시간을 함께 고려한다.
2. **초기 배정보다 지속적 재조정이 중요하다.** 진행 중 부하 변화를 감지해 재배분을 제안한다.
3. **파악을 위해 새 보고를 늘리지 않는다.** 기존 문서·회의록·업무 로그를 재활용해 자동 구조화한다.

## 4. 기존 기능의 재정의 — "데이터 수집 계층"

**핵심 전환점.** 메일·메신저·보고서·캘린더·회의록·워크스페이스는 **독립 서비스가 아니라, 업무 분배·부하 조율 엔진에 입력을 공급하는 보조 데이터 계층**이다. 각 기능은 "무엇을 위한 데이터를 공급하는가"로만 존재 이유를 갖는다.

| 데이터 소스 (보조 기능) | 공급하는 입력 | 엔진에서의 역할 |
| --- | --- | --- |
| **회의록 / 기획서** | 해야 할 일(Task) 원천 | 업무 자동 추출의 입력 (STT·문서 파싱 → 태스크) |
| **메일 / 메신저** | 업무 맥락·진행상황·액션아이템 | 진행률 갱신, 신규 업무 발생 감지 |
| **캘린더** | 빈 시간·마감 일정 | 가용시간(여유) 산출 |
| **일일 보고서 / 업무 로그** | 개인별 진행률·업무량·블로킹 여부 | **부하 판단의 1차 입력** (누가 지금 얼마나 진행 중인가) |
| **주간 취합 보고서** | 팀 단위 진행 현황 종합 | 부하 추세·재배분 판단 입력 (팀 취합 보고서) |
| **워크스페이스** | 팀·프로젝트·역할·권한 경계 | 누가 어느 프로젝트에 속하는지, 데이터 접근 범위 정의 |

> **보고서는 이 시스템에서 가장 직접적인 부하 데이터원**이다. 일일 보고서는 개인 진행률·업무량을, 주간 취합 보고서는 팀 단위 추세를 공급한다. 단, "부하 파악을 위해 새 보고를 늘리지 않는다"(3절 원칙)는 전제에 따라, 보고서는 **기존 업무 활동·업무 로그에서 자동 구조화**하는 것을 기본으로 하고 수기 입력은 최소화한다.
>
> 특히 **주간 취합 보고서는 시스템이 진행 데이터를 자동으로 취합해 생성하는 경량 산출물**이다(원래 기획의 '팀 취합 보고서' 기능을 흡수). 즉 보고서는 입력이자 동시에 부산물이며, 생성된 종합 결과가 다시 부하 추세 판단에 재사용된다. → 자세한 부가 기능 배치는 12절 참조.

> 이 재정의 덕분에 "기능이 많다"는 인상 대신 **"하나의 문제를 풀기 위해 여러 소스를 엮는다"**는 인상을 준다. 기능 나열이 아니라 데이터 파이프라인이다.

## 5. 핵심 MVP 시나리오 (와우 포인트)

> 여러 기능을 설명하지 않는다. **이 단일 업무 흐름 하나**를 처음부터 끝까지 완성하는 것이 MVP다.

1. **프로젝트 생성** — 팀장이 프로젝트명 입력, 기획서/요구사항 문서 업로드, 팀원 선택.
2. **업무 자동 추출** — AI가 기획서·회의록을 분석해 Task를 자동 생성 (예: 시장조사, 경쟁사 분석, UI 설계, API 개발, 테스트, 발표자료). 각 태스크에 중요도·난이도·예상 소요·마감을 부여.
3. **구성원 업무 현황 분석** — 각 팀원의 현재 업무량·진행 중 업무·예상 소요·마감·가용시간·역할·역량을 기존 데이터로 자동 파악. → **"현재 누가 얼마나 바쁜지 모르겠다" 문제 해결.**
4. **업무 배정 추천** — 업무량·가용시간·역할·기술 역량·난이도·우선순위를 종합해 태스크별 담당자를 **근거와 함께** 추천 (Explainable).
5. **팀장 검토·확정 (Human-in-the-loop)** — 추천 확인, 담당자 변경, 업무 추가/삭제 후 확정.
6. **진행 모니터링** — 업무 완료·일정 지연·신규 업무·업무량 변화를 지속 감지.
7. **과부하 감지** — 예: "김철수 업무 8개·예상 52시간·긴급 4개 → 적정 초과."
8. **재배분 추천** — 예: "시장조사 → 박민수 이동 추천 (현재 부하 25%, 동일 경험, 일정 영향 없음)." 팀장 승인 시 재배정.

## 6. 멀티에이전트 아키텍처

### 6.1 전체 구조 (Scatter-Gather)

```
사용자(팀장) → 프로젝트 생성
      │
      ▼
[중앙 오케스트레이터]  의도 파악 → 워크플로우 매핑 (MCP 기반)
      │
      ├─▶ [업무 추출 에이전트] 문서/회의록 → N개 태스크 (추출 스킬 사용)
      │
      ├─▶ (분산/Scatter) [부하판단 에이전트 ×N]  팀원 1명씩 독립 판단
      │        · 현재 업무량 조회  · 직무 역량 매칭  · 캘린더 가용 검증
      │
      ├─▶ (수집/Gather) [업무 분배 에이전트]  부하 데이터 + 태스크 → 배정안 수립
      │
      ▼
[팀장 확정 게이트]  승인 → DB 등록·알림 / 수정 → 재조율(루프백)
      │
      ▼
마이페이지·DB 등록 + 팀원별 대시보드
```

- **하단 공유 계층:** 공유 스킬(교정·요약·추출·검색RAG·번역) · 모델(공용 LLM + 보안 sLLM RunPod) · 데이터(PostgreSQL + pgvector).

### 6.2 워크플로우 vs 스킬 분리 (멘토 지침 반영)

- **워크플로우는 도메인별로 나눈다** — 프로젝트 배분 워크플로우, 회의 처리 워크플로우, 보고서 워크플로우 등 각자 독립적 처리 순서를 가진다.
- **스킬(요약·추출·교정·검색·번역)은 공유한다** — 같은 스킬을 호출하되 **목적(purpose)과 출력 형식(output format)만 파라미터로 다르게** 전달한다. (예: 요약 스킬 — 회의용은 "액션아이템 필수 포함", 보고서용은 "경영진 대상·핵심지표 포함".)
- 결과: 동일 로직 중복 제거, 수정 지점 단일화.

### 6.3 에이전트 분리 기준 (근거 명시)

무엇을 에이전트로 나눌지는 다음 세 기준으로 판단한다.

| 기준 | 의미 | 본 프로젝트 적용 |
| --- | --- | --- |
| **① 목표가 다른가** | 추구하는 품질 기준 자체가 다른가 | 업무 추출(정확한 태스크 분해) vs 부하 판단(객관적 여력 산출)은 목표가 다르므로 분리 |
| **② 도구·데이터 권한이 다른가** | 접근 권한이 분리돼야 하는가 | 부하판단 에이전트는 팀원별 사적 컨텍스트(개인 업무량·캘린더)만 접근 → 팀원 수만큼 독립 인스턴스 |
| **③ 독립 판단이 필요한가** | 다음 행동을 스스로 선택하는 지점인가 | 배분/재배분은 상황에 따라 결정이 달라지는 판단 지점 → 에이전트, 단순 실행(발송·저장)은 툴 |

> 이 기준에 따라 **판단이 필요한 지점만 에이전트**로 두고, 단순 실행은 툴로 처리한다. "다중 에이전트"는 라벨이 아니라, **부하판단 N개의 독립 판단 + 조율(Scatter-Gather)**이라는 실제 협력 구조로 성립한다.

### 6.4 에이전트 목록 (판단) / 스킬 (실행)

- **에이전트(판단):** 오케스트레이터 · 업무 추출 · 부하판단(×N) · 업무 분배(조율)
- **스킬(실행, 판단 없음):** 추출 · 요약 · 교정 · 검색(RAG) · 번역

## 7. 부하 조율 점수화 모델 (핵심 차별·깊이)

> 멘토 지적: "조율 아이디어는 좋으나 로직이 추상적 — 어떻게 점수화할지 구체화 필요." 이 절이 프로젝트의 깊이다.

부하 판단은 **설명 가능한(Explainable) 수치 모델**로 정의한다.

**① 개인 가용 여력 (Available Capacity)**

```
가용여력 = 주간 가용시간(캘린더 빈 시간)
         − Σ( 진행중 태스크 예상 잔여시간 × 난이도 가중 )
```

**② 태스크 부하 (Task Load)**

```
태스크부하 = 예상 소요시간 × 난이도 가중 × 마감 임박 가중
```

**③ 배정 적합도 (Assignment Fit)**

```
적합도 = w1·(가용여력 충분도) + w2·(역량 적합도) + w3·(일정 영향 최소)
```

- **과부하 임계:** 주간 부하율 > 100%(가용여력 < 0) 또는 긴급 태스크 집중 시 경고.
- **추천 근거 제시:** "현재 부하 40%, 동일 업무 경험, 일정 영향 없음"처럼 **각 점수를 근거로 노출**한다 (Explainable AI).
- **가중치(w) 튜닝**과 예상 소요시간 산정은 도메인 데이터(디지털마케팅 프로젝트 로그)로 보정한다.

## 8. 모델 전략

- **공용 LLM (Commercial API):** 오케스트레이션·추출·판단 등 범용 추론.
- **보안 sLLM (RunPod 온프레미스, 파인튜닝):** **작업(task) 단위 파인튜닝**으로 방향을 잡는다. 범용 SFT로 외부 공개 데이터만으로 성능 우위를 내긴 어렵다는 멘토 조언 반영.
  - 대상: 일관된 요약·태스크 추출 포맷, 배정 근거 문장 생성 등 **정형 출력**.
  - 어필 포인트: **"GPT보다 저렴"**, **"항상 일관된 포맷으로 출력하도록 학습"** — 비용·일관성 모두 차별점.
- **하드웨어:** RunPod. 격식/포맷 파인튜닝은 7~8B급 QLoRA(RTX 4090~A6000)로 충분. (지원 예산 약 $500 범위 내 여유 — 유휴 시 pod 정지 원칙.)

---

## 9. 범용 vs 도메인 방향

- **지향:** 범용 플랫폼 (어느 조직이든 업무 분배·부하 조율에 적용 가능).
- **MVP 적용·검증:** **디지털마케팅 도메인(인크로스)** 데이터로 한정해 구현·검증한다.
  - 정형 데이터: 표준 업무 프로세스(사전 정의) · 후속 비정형 데이터: 기획서·보고서(자동 수집).
- **근거 조사:** 산업 리포트(맥킨지·가트너 등)로 실제 워크플로우를 확보하고, 상용 플랫폼(Copilot·Slack·뤼이도·먼데이닷컴)을 조사해 차별점을 명확히 한다.

---

## 10. 평가 방법

결과에 "정답"이 없는 업무이므로 **정답 검증이 아니라 형식·합리성 검증**으로 평가한다.

- **형식 검증:** 출력 규칙 준수, 필수 항목 포함(날짜·담당자·일정 등 원문 일치), 요약·추출에 원문 근거 연결 여부.
- **배분 합리성 검증:** 과부하 임계 초과자에게 신규 업무를 몰지 않는가, 재배분 추천이 부하·역량·일정 근거와 일치하는가(시뮬레이션 시나리오로 검증).
- **비교군:** (A) 수동 배분, (B) 단순 균등 배분, (C) 제안 모델 — 부하 편차·과부하 발생률 비교.

## 11. 시장 조사 & 차별점

| 제품 | 성격 | 우리와의 차이 |
| --- | --- | --- |
| **뤼이도(riido.io)** | AI 프로젝트 관리, 팀원 상태 확인 후 업무 분배 | 상태 확인 중심 — **부하의 정량 산출·과부하 재배분 근거**가 약함 |
| **먼데이닷컴** | AI 태스크 관리 플랫폼 | 태스크 관리 도구 — **기존 활동에서 부하를 자동 수집·수치화**하는 축이 없음 |
| **Copilot / Slack / 기존 그룹웨어** | 요약·일정 추출·연동 | 개별 기능 제공 — **"객관적 부하 산출 → 공정 배분 → 재조율"의 통합 판단 흐름**이 없음 |

> **차별점 한 줄:** 남들이 "정보를 요약·연동"할 때, 우리는 **"부하를 객관적으로 산출해 배분을 판단"**한다. NCS는 제외했고, 역량은 업무량·가용시간과 함께 종합 판단하는 **한 요소**로만 사용한다.

## 12. MVP 범위

### 반드시 구현 (핵심)
- 업무 추출 에이전트 (기획서·회의록 → 태스크)
- 부하판단 에이전트(×N) + 업무 분배 에이전트 (Scatter-Gather)
- 부하 점수화 모델 + Explainable 추천 근거
- 과부하 감지 → 재배분 추천
- 팀장 확정 게이트 (Human-in-the-loop) + 팀원 대시보드
- 공유 스킬(추출·요약) + 데이터 수집 파이프라인(회의록·캘린더·업무 로그)

### 부가 기능 (원래 기획 흡수 — 핵심을 보조하는 범위에서만)

> 원래 기획의 기능은 **삭제하지 않고 전부 유지**하되, 독립 목적이 아니라 **핵심(업무 분배·부하 조율)을 위한 데이터 공급 또는 경량 부산물**로만 배치한다.

**(1) 데이터 수집 (입력)**
- **회의 요약** (STT → 결정사항·액션아이템 추출): 업무 추출의 원천
- **메일·메신저**: 업무 맥락·진행상황·액션아이템 감지
- **캘린더 / 일정 추출·등록**: 가용시간(여유) 산출
- **일일 보고서·업무 로그**: 업무량·진행률
- **워크스페이스**: 팀·프로젝트·역할·권한 경계

**(2) 경량 산출물 (출력)**
- **주간 취합 보고서 자동 생성**: 진행 현황 종합 — 부하 추세 대시보드의 부산물 (원래 '팀 취합' 기능 흡수)
- **개인 보고서 작성·교정 지원**: 진행 기록을 구조화해 입력 품질을 높임

**(3) 공유 스킬 기반 편의**
- **메일 문구 교정·요약·번역**: 공유 스킬(교정·요약·번역) 재사용
- **공지·FAQ 검색 / 개인 메모**: 검색 스킬(RAG) 재사용
- **리마인드 / 제출 관리**: 진행 모니터링의 보조 알림

**(4) 플랫폼 기반**
- **4단계 권한 모델, 관리자 설정(보고서 양식 등록), 마이페이지**: 운영 기반 기능

> **배치 원칙:** 부가 기능은 *핵심 흐름에 데이터를 넣거나, 핵심 산출물의 부산물*일 때만 존재한다. 발표에서는 개별 나열하지 않고 **"핵심을 둘러싼 데이터 수집·부산물 계층"으로 한 장에 압축**한다 (기능 나열 인상 방지).

### 제외 / 시뮬레이션
- NCS 기반 매칭 (완전 제외)
- Production 성숙도 요소(장기 메모리·툴 재시도·Observability) — 데모 범위 밖, 발표 시 "Production 확장 로드맵"으로만 언급.
- 실제 사내 메일 서버·조직도 연동 (mock 데이터 대체)

## 13. 일정 (요약)

| 주차 | 핵심 목표 |
| --- | --- |
| 1주 | 문제 정의·방향 확정, 데이터 수집 계층 설계, 대표 시나리오 I/O 정의 |

## 부록 A. 발표 서사 (문제 → 화면 → 아키텍처)

1. **문제:** "누가 얼마나 바쁜지 아무도 정확히 모른다" (통계 3~4개).
2. **화면 하나:** 팀장이 기획서를 올리자 태스크가 뽑히고, 팀원별 부하가 수치로 뜨고, 과부하자 대신 여유 있는 사람에게 재배분이 추천되는 장면.
3. **그게 되는 이유(아키텍처):** 기존 활동에서 데이터를 수집 → 부하판단 N개 병렬 → 조율 → 팀장 승인. (1~2장으로 압축.)
