## 1. Workday/HR 데이터를 어떻게 처리할지

- 데이터를 엔지니어링해서 인위적으로 붙이는 방식이면 실제 데이터 형태를 사람들이 자세히 볼 거라 유사 데이터가 의미 없을 수 있음. 커넥터로 워크스페이스 데이터를 가져오는 방식이면 내부 데이터 형태를 몰라도 되고, "어떤 데이터든 접근해서 쓸 수 있다"는 것 자체가 어필 포인트가 됨
- 커넥터 하나를 제대로 만드는 것도 해야 할 일이 많음. 하나를 잘 설계했다는 걸 잘 어필하는 것만으로도 충분히 좋을 수 있음
- 지금 연동하려는 것들(Google Calendar/Drive, Jira) 중 HR을 빼면 이미 다뤄본 것들이라 러닝커브가 낮은데, HR 하나만 새로 만들면 그것 때문에만 새로운 러닝커브가 생김. 배보다 배꼽이 더 클 수 있음. 이 시간에 차라리 노션·슬랙 같은 실제 서비스를 붙이는 게 나을 수 있음
- HR만 가진 고유 정보(직급, 조직, 직무, 매니저)는 구현이 어려우니 "다른 데 있다고 가정"하는 방식도 나쁘지 않음. 이 시간을 다른 데 쓰는 게 더 명확한 학습이 될 것 같음
- 완전히 의미 없는 작업은 아님. 커넥터를 하나씩 분석·구현해보면 고려해야 할 부분들이 자연스럽게 맞춰질 수 있음. 다만 지금은 이 부분을 뒤로 미루고, 빠르게 보여줄 수 있는 부분부터 중간 점검에 내놓는 걸 우선함

## 2. HR 데이터 vs 다른 시스템 데이터의 역할 구분

- "역량·과거 프로젝트 이력처럼 HR에만 있는 정보가 빠지면 배정 로직이 빈약해지지 않냐"고 문제 제기했을 때, 그 말이 절반은 맞다고 답하며 아래처럼 정리함
    - Workday(HR): 소속·직급·직무·매니저·역량
    - Jira: 현재 맡은 티켓과 마감일
    - Notion/Confluence: 프로젝트별 역할·담당
    - Bitbucket: 리뷰 이력·실제 코드 기여
    - Slack: 협업 주제
    - 이런 시스템들이 오히려 더 밀도 높은 데이터를 갖고 있을 수 있음
    - HR은 "그 사람의 기본 프로필", Jira/Git류는 "현재 상태와 실제 역량의 증거"로 성격이 구분됨
- HR 데이터가 있으면 배정 논리가 더 탄탄해질 수 있음은 맞음. 다만 그것을 위해 지금 새로 만드는 건 들이는 노력 대비 얻는 논리적 이득이 크지 않음
- 대안: "People DB" — HR 제품을 그대로 쓰지 않고 이름·팀·직무·기본 스킬 정도만 담은 간단한 DB를 만들어서 "HR 시스템이 없어서 이렇게 DB로 대체했다"고 설명. 나머지 역량 정보는 업무 커넥터(Jira 등)에서 가져오면 시간도 아끼고 논리적으로 설명 가능함
- 애초에 커넥터 방식을 택한 이유: 팀이 데이터 자체를 검증할 수 있는 입장이 아니라서, 이미 많은 사람이 쓰고 있고 데이터가 실제로 쌓여 있는 외부 시스템에 연결하는 방식을 택함. HR은 그 4~5개 연동 대상 중 하나일 뿐이라 하나 빠진다고 전체 전제가 무너지는 건 아님. 다만 이런 근거를 다 빼고 "그냥 가짜 데이터로 채운 것"이 되면 곤란함
- 그래도 찝찝하면 무료 HR 플랫폼을 그대로 연동하는 방법도 있음. 다만 그 경우에도 원래 커넥터를 만들려던 이유(권한·인증 같은 복잡성을 실제로 겪어보는 것)를 놓치지 않도록 주의 필요

## 3. 목업을 만든다면 어느 수준까지 해야 하는가

- 목업을 만들 경우 진짜 커넥터에서 중요한 요소: 인증 토큰, 요청 파라미터(리밋/오프셋/페이지네이션), 응답 스키마, 필터링 방법, 오류 코드
- 이 정도까지 고려해서 구현하면 의미가 있음. 단순히 값을 주고받는 수준이면 "커넥터를 만들었다"고 보기 어려움. 구체적으로는 토큰 발급 시점·저장 위치, 페이지네이션 방식, 오류 코드별 처리(401/403/404/425 등), 메타데이터(updatedDate/createdDate/version)까지 고민해보는 게 공부가 많이 됨
- "이렇게까지 안 하면 그냥 DB랑 다를 게 없지 않냐"는 질문에는, 어설프게 CRUD만 만들 거면 처음부터 안 하는 게 낫고 할 거면 확실하게 만들어야 함("할 거면 확실하게, 안 할 거면 아예 목업으로 가라")
- 이건 개인 의견이고, 이 정도 깊이는 중간발표·시험발표에서는 별로 안 물어볼 것 같음. 면접에서 커넥터를 얘기할 때는 오히려 단순 API CRUD + OpenAPI(Swagger) 문서 + Bearer 토큰 인증 + 검색 필터 정도만 만들어도 큰 차이는 없음. 개념 자체를 이해하고 설명할 수 있는 건 그래도 도움이 됨
- 결론: 시간이 되면 "커넥터 구현 시 고려해야 할 5가지"(직무 조회, 조직 조회, 필터링, 페이지네이션, 인증 방식) 정도는 공부 삼아 다뤄볼 만함. 다만 이것 때문에 전체 진행 속도가 늦어지는 건 우려됨

## 4. 비정형 데이터 처리 레이어

- 비정형 데이터 처리 레이어 필요함
- 파일을 다운로드하면 S3에 저장, 청킹할 때 이미지 있으면 OCR, 벡터 DB에 저장, 메타데이터 필요, 이 페이지의 주제가 뭔지 LLM이 구조화할 수 있게 추출
- 벡터 DB에서는 메타데이터가 가장 중요함. 이 메타데이터를 어떻게 저장할지도 중요함
- Citation(데이터 신뢰 추적성)은 사람을 위한 것
    - 메타데이터에 없는 내용으로 이야기하지 말라고 강제할 수 있음
    - 논문 citation처럼, "safety 관점에서 ~한 이유로 citation을 썼다"고 설명하면 좋음
- 메타데이터 예시

```json
{"documentId":"DOC001","fileName":"2026_AI_Project.pdf","department":"AI Platform","owner":"PM 김철수","createdAt":"2026-07-01","modifiedAt":"2026-07-20","version":"v3","security":"Internal","project":"AI Assistant","page":5,"chunk":18}
```

- 처리 파이프라인: Google Drive 문서 → 파일 다운로드 → PDF/DOCX 파싱 → 텍스트 청킹 → Metadata 부착 → Embedding → Vector DB → 관련 문서 검색 → LLM 구조화 추출 → NewTaskDraft
- DB는 일단 3개로 구성
    - PostgreSQL: 사람, 스킬, 업무, 일정, 매핑, 추천 결과
    - ChromaDB 또는 pgvector: 문서 청크와 임베딩
    - Object Storage: 원본 PDF/문서

## 5. 데이터 최신성·동기화·Snapshot

- 데이터 최신성 문제: Workday는 어제 데이터, Jira는 한 달 전 데이터일 수 있음. refresh/cache/TTL 필요
- 주기적인 refresh 필요. 데이터마다 갱신 속도가 다름(직무는 최신인데 역량은 오래된 것일 수 있음). 이것도 snapshot으로 할지 refresh token으로 들고 올지 고민 포인트
- 프로젝트를 생성한 순간에 최신 데이터로 업데이트된 상태에서 프로젝트를 생성한다고 가정할 수 있음
- last sync data 개념 필요, 주기적 동기화 + 수동 새로고침
- 일일 보고서 기준으로 하루 한 번씩 동기화하는 방식은 사람이 많으면 비용이 커질 수 있음
- 모든 직원이 일일 업무를 작성해야 하는 건지, 막무가내로 일일 snapshot을 하는 게 맞는지는 의문
- 기능 자체가 비동기적 기능임. 하루 일괄 실행이 아니라 사람의 요청이 기준
- snapshot은 추가적으로 생성해야 하지만 주기적인 건 아닐 수 있음. 프로젝트가 추가되는 시기가 중요함
- 업무 분배 요청할 때마다 비동기가 시작되는 구조인지, 매번 요청하는 게 아니라 매일 고정으로 도는 구조인지 확인 필요
- 오프라인 상에서 팀장에게 프로젝트가 할당되는 시점을 기준으로 생각함
- 비정형 데이터가 많아지면 청킹·메타데이터·인덱싱에 시간이 오래 걸릴 텐데, 매번 그때마다 다시 돈다면 시간이 얼마나 걸릴지 확인 필요
- 초기에는 전체 인덱싱, 이후에는 변경된 부분만 증분 인덱싱하는 방식 필요
- 문서끼리 엮여 있는 부분도 고려 필요

## 6. 사용자 편의성 및 문서 인덱싱

- 회사마다 다른 양식 중에서 어떤 내용을 잘 가져와서 잘 인덱싱될 수 있냐가 중요함
- 비정형 데이터 청킹 전략, 파싱 필요
- 구글 드라이브 내부 문서의 메타데이터와 우리 쪽 메타데이터를 최초에 합쳐서 저장해야 하는지 확인 필요
- 메타데이터 파싱은 일단 생각 안 하기로 함
- 전제 없이 범용적인 사용자가 들어왔을 때: 메타데이터를 사용자가 직접 정할 수도 있고, 자동이 안 되면 라디오박스로 날짜·주제 등을 선택해서 생성할 수도 있고, 기본적인 메타데이터만 추출해도 무방함
- 폴더 선택 권한, 메타데이터 추출 권한도 사용자에게 줌. 고객이 범용적으로 선택할 수 있게 가되, 팔란티어처럼 온톨로지를 자동으로 해주는 부분은 지금 하지 않음
- 다만 선택지를 주는 부분 자체는 편리해야 함
- 선택지 예시: 어떤 폴더를 메인으로 쓸 것인가, 어떤 폴더를 포함할지(기획서/요구사항정의서/개인 문서)
- 프로젝트 기획서·회의록은 사용하고, 일일 보고서 같은 건 제외할 수 있게
- 프로젝트 기획서에는 신규 태스크가 있고, 일일 보고서에는 무엇이 있다는 식으로 이 폴더에 뭐가 있는지 고객이 명시할 수 있게 해주는 방법도 있음

## 7. 5번 레이어 — Feature Readiness Check

- 이게 맞는지 검증하는 게 힘들 수 있음
- 업무 분배는 기획서, 태스크, 추천으로 구성됨. 구글 드라이브 커넥터를 생각해보면 문제 정의를 document understanding으로 가면 좋음
- 예시: 업무 분배가 하나의 use case이고 구글 회사라고 하면, 드라이브에서 PDF를 뽑고 DOCX·PDF를 공통으로 처리하는 식으로 검증되어야 함
- 왜 success인지에 대한 기준이 없음
- 업무 분배가 맞는지 아닌지 확인하려면 업무 분배에 필요한 데이터가 있는지가 중요함(업무 태스크, 현재 업무량 존재 여부, 스키마)
- 업무량 없이 추천하는 경우도 있을 수 있음
- 데이터 품질 검사보다는 필수 데이터가 있는지 확인하는 방향이 좋음
- success factor는 고객이 설정하는 것. 우리가 정하면 말이 안 됨
- 5번 레이어 이름을 Feature Readiness Check로 수정

## 8. 6번 레이어 — 오케스트레이션/추출 로직

- 재추출 요청할 때는 description 피드백을 가지고 재요청
- 데이터 자체가 잘못됐을 때 업무 추출 agent로 가는 게 맞는지에 대해: 위에서 적합성을 체크했으니 업무 추출 agent로 가는 게 맞음. 데이터가 잘못됐다면 다시 실행해달라는 메시지가 있으면 좋음
- 결과적인 일관성이 있어야 함. 다만 현재 어떤 기준에서 일관성이 있어야 하는지 틀이 없다는 게 고민 포인트
- 이건 부서·사람마다 다른 암묵지임. 이것도 입력받아서 만들어야 함
- "말이 되는데 정렬만 바꾸고 싶다", "말이 안 된다"처럼 사용자가 쉽게 적어서 힌트를 줄 수 있게 만들어야 함
- 오케스트레이터는 언제든 호출할 수 있다는 의미. 화살표가 없어도 됨
- router는 쿼리가 매번 달라질 때 실행하는 개념인데, 지금 병렬로 그려져 있는 건 잘못됨. 병렬이면 router라고 쓰면 안 됨
- 업무 부하 수치 계산식을 파이썬 함수처럼 만들어놓은 건 잘함. agent에게 너무 많은 역할을 부여하면 안 됨
- 오케스트레이터는 순서만 관리함. 호출만 함
- ReAct 구조가 아니라 Fallback 구조여야 함
- 기획서에 있는 내용을 추출하는 건지, 없는 걸 LLM이 생성하는 건지 기준이 필요함(생성 기준의 문제)

## 9. 플랫폼이 처리해야 할 기술 (13개)

1. **Connector / API Integration** — Workday, Jira, Google Drive, Calendar처럼 서로 다른 시스템을 붙이는 기술
2. **Canonical Data Model 설계** — 서로 다른 데이터를 공통 스키마로 바꾸는 능력
3. **Identity Resolution** — 동일 인물을 여러 시스템에서 하나로 합치는 기술
4. **Document Parsing과 Information Extraction** — 기획서, 회의록, 프로젝트 문서에서 목표·산출물·제약조건을 구조화
5. **Retrieval Engineering** — 과거 유사 프로젝트와 업무 사례를 찾는 기술
6. **Task Graph Engineering**
7. **Resource Allocation / Optimization** — 업무를 누구에게 배정할지 계산하는 핵심 엔지니어링
    
    ex) 
    
    maximize:   
    
    skillFit + experienceFit + fairness   
    
    subject to:   
    
    capacity <= 40시간   
    
    휴가일 배정 금지   
    
    필수 Skill 미충족자 제외   
    
    동일 기간 최대 프로젝트 3개
    
8. **Policy Engine** — 팀장의 암묵지를 규칙과 정책으로 실행하는 기술
9. **Capacity와 Scheduling Engine** — 단순 현재 Task 개수로 부하를 계산하면 안 됨
10. **Evidence와 Citation Engineering** — 추천 결과마다 근거를 연결해야 함
11. **Validation / Evaluation Engineering** — 정답이 하나가 아니므로 평가 설계가 중요함
12. **Workflow Orchestration** — Agent 여러 개를 그냥 연결하는 게 아니라 상태와 재처리를 관리해야 함
13. **Security와 Access Control (중요)** — HR 정보와 직원 평가 데이터가 들어가므로 반드시 필요함
    - 예: RBAC, Row-level security, Document ACL, PII masking, Audit log, Secret management, Tenant isolation, 최소 권한 접근, 데이터 보존 정책

- 이 중 몇 개는 사용자가 암묵지로 정해줄 수 있는 부분임 (예: Google Compass)
- 13번(보안)은 특히 중요하니 한번 더 공부해볼 것을 권함
- 문서 파싱·청킹·OCR을 잘 다듬어두면 이후 과정이 자동으로 처리될 수 있음

## 10. 7번 레이어 — 운영/대시보드

- 운영 로그는 실제 개발자가 뭘 보는지를 기준으로 작성하면 좋음
- 관리자 대시보드는 KPI를 보는 것, accept/reject를 보는 데 필요함
- AI가 추천했던 assignment history, 수정이 어떻게 됐는지, 최종 수정 이력을 볼 수 있어야 함 → 업무 분배 결과를 대시보드에 넣고 추적 가능하게 만들어야 함
- 감사 로그(audit log)에는 근거(evidence)·신뢰도(confidence) 버전까지 남기면 좋음

    # 3. Workday와 HR 데이터 처리 방향
    
    ## 3.1 Workday 목업을 본격 구현하는 것은 우선순위에서 제외
    
    팀에서는 Workday API 문서를 바탕으로 FastAPI 목업 서버를 만들고 실제 커넥터처럼 사용하는 방안을 제안했습니다.
    
    멘토님의 판단은 다음과 같습니다.
    
    - 단순 CRUD 서버라면 일반 DB와 차이가 거의 없음
    - 진짜 커넥터로 만들려면 아래 요소까지 고려해야 함
        - 인증·권한
        - 요청 파라미터
        - 페이지네이션
        - 응답 스키마
        - 필터링
        - 오류 코드와 재시도
        - 데이터 버전과 수정 시각
    - 이 수준까지 구현하면 학습 가치는 있지만, 현재 일정에서는 전체 프로젝트보다 일이 커질 수 있음
    - 중간 발표 전에는 실제로 보여줄 수 있는 핵심 흐름을 먼저 완성하는 것이 좋음
    
    따라서 Workday 목업은 “완전 폐기”라기보다 **후순위 또는 확장 항목**에 가깝습니다.
    
    ## 3.2 현실적인 대안: People DB
    
    HR 시스템을 직접 재현하는 대신 다음과 같은 최소 People DB를 둘 수 있습니다.
    
    | 구분 | 저장할 내용 |
    | --- | --- |
    | 기본 프로필 | 이름, 이메일, 팀, 직무 |
    | 조직 정보 | 관리자, 소속 조직, 재직 상태 |
    | 최소 역량 정보 | 기본 스킬, 숙련도 |
    | 식별자 매핑 | Google·Jira·GitHub 계정 연결 |
    
    나머지 정보는 실제 업무 시스템에서 확보합니다.
    
    - Jira: 현재 업무, 담당 티켓, 상태, 마감일
    - GitHub/Bitbucket: 커밋·리뷰·코드 기여 이력
    - Notion/Confluence/Drive: 프로젝트 역할과 산출물
    - Calendar: 일정, 회의, 휴가·출장
    - Slack 등: 협업 주제와 업무 맥락
    
    정리하면 다음과 같습니다.
    
    > HR 데이터는 기본 프로필이고, Jira·Git·문서 데이터는 현재 상태와 실제 역량의 증거다.

    # 4. 비정형 문서 처리 레이어
    
    이번 멘토링에서 가장 중요하게 강조된 기술 영역입니다.
    
    ## 4.1 문서 처리 흐름
    
    1. 사용자가 Google Drive의 대상 폴더 선택
    2. 문서 원본 다운로드 및 Object Storage 저장
    3. PDF·DOCX·PPTX·Google Docs 파싱
    4. 스캔·이미지가 포함된 경우 OCR
    5. 문서를 의미 단위로 청킹
    6. 원본 메타데이터와 시스템 메타데이터 결합
    7. 임베딩 후 Vector DB 저장
    8. 관련 문서와 과거 사례 검색
    9. LLM이 태스크·목표·산출물·제약조건을 구조화
    10. 각 결과에 원문 근거 연결
    
    ## 4.2 메타데이터 예시
    
    ```json
    {
      "document_id": "DOC001",
      "file_name": "2026_AI_Project.pdf",
      "source": "google_drive",
      "project": "AI Assistant",
      "department": "AI Platform",
      "owner": "PM 김철수",
      "created_at": "2026-07-01",
      "modified_at": "2026-07-20",
      "security": "internal",
      "page": 5,
      "chunk": 18
    }
    ```
    
    초기에는 모든 메타데이터를 자동 추출하려 하지 않고 다음 정도만 우선 구현하는 것이 권장됐습니다.
    
    - 파일 ID·이름·형식
    - 원본 시스템
    - 생성·수정 시각
    - 작성자 또는 소유자
    - 폴더·프로젝트
    - 페이지·청크 위치
    - 접근 권한
    - 문서 주제 정도
    
    Google Drive는 파일 ID, 부모 폴더, MIME 유형, 생성·수정 시각, 버전, 소유자 및 일부 권한 정보를 제공하므로 이 기본 메타데이터는 API에서 확보할 수 있습니다. Google Drive Files 리소스
    
    ## 4.3 Citation의 목적
    
    Citation은 단순히 출처 링크를 보여주는 기능이 아닙니다.
    
    - 추천 결과의 근거 확인
    - 환각 여부 검토
    - “어떤 문서의 어느 부분을 사용했는가” 추적
    - 권한 없는 문서를 근거로 사용하지 않았는지 감사
    - 사용자가 추천을 승인하거나 반려할 판단 자료 제공
    
    따라서 `source_id`, `page_number`, `chunk_id`, `description` 등을 결과와 연결해야 합니다.
    
    ## 4.4 멘토링 중 언급된 제품명 보정

    # 5. 모든 문서를 볼 것인가, 필요한 문서만 볼 것인가
    
    회의에서는 처음에 “전체 문서를 인덱싱해야 한다”는 방향이 제시됐지만, 최종적으로는 **사용자가 범위를 선택하는 구조**로 정리됐습니다.
    
    가능한 선택지는 다음과 같습니다.
    
    - 전체 Drive 인덱싱
    - 특정 Shared Drive만 선택
    - 특정 프로젝트 폴더만 선택
    - 기획서·요구사항·회의록 폴더만 포함
    - 개인 문서·임시 파일·일일 보고서 제외
    - 각 폴더가 어떤 역할인지 사용자가 설명
        - 기획서: 신규 태스크와 목표
        - 회의록: 결정사항과 변경사항
        - 일일 보고서: 진행 상태와 업무 경험
    
    이렇게 하면 특정 회사의 문서 구조를 미리 가정하지 않으면서도 비용과 처리량을 통제할 수 있습니다.

    # 6. 데이터 최신성·Snapshot·증분 인덱싱
    
    ## 6.1 기본 원칙
    
    데이터마다 변화 속도가 다릅니다.
    
    - Calendar·Jira: 자주 변경
    - Drive 문서: 수정 시 변경
    - 직무·조직: 상대적으로 변경 빈도가 낮음
    - 스킬·프로젝트 경험: 갱신 기준이 불명확할 수 있음
    
    따라서 모든 데이터를 매일 일괄 수집하는 방식보다 다음 구조가 적절합니다.
    
    1. 커넥터 최초 연결 시 전체 수집·인덱싱
    2. 이후 생성·수정·삭제된 항목만 증분 반영
    3. 업무 분배 요청 시 최신 동기화 여부 확인
    4. 필요하면 사용자가 수동 새로고침
    5. 추천 시점의 데이터를 Snapshot으로 고정
    6. `last_synced_at`, `source_updated_at`, `snapshot_as_of`를 함께 기록
    
    Google Drive는 시작 토큰을 저장한 뒤 `changes.list`로 변경사항을 순서대로 가져오고, 마지막 토큰을 다음 동기화에 재사용할 수 있으므로 증분 인덱싱 구현이 가능합니다. Google Drive 변경사항 추적
    
    ## 6.2 용어상 보정
    
    노션에는 “Snapshot으로 할지 refresh token으로 가져올지”라는 표현이 있었지만 두 개념은 비교 대상이 아닙니다.
    
    - Refresh token: 만료된 API access token을 다시 발급받기 위한 인증 수단
    - Snapshot: 특정 시점의 사람·업무·가용성 데이터를 고정해 저장한 결과
    - Change token/page token: 변경된 Drive 파일을 추적하기 위한 동기화 기준
    
    Refresh token은 데이터 최신성을 보장하지 않습니다. Google OAuth 2.0 문서

    # 7. Feature Readiness Check
    
    기존 아키텍처의 “데이터가 정확한지 검증” 단계는 범위가 지나치게 넓고 정답을 정의하기 어려웠습니다.
    
    따라서 5번 레이어는 **Feature Readiness Check**로 변경합니다.
    
    검사 대상은 “추천 결과가 정답인가?”가 아니라 다음입니다.
    
    - 직원 기본 정보가 있는가
    - 프로젝트 목표가 있는가
    - 추출하거나 생성할 업무 정보가 있는가
    - 업무별 필요 스킬을 판단할 정보가 있는가
    - 현재 업무량 또는 가용시간 정보가 있는가
    - 고객이 필수로 지정한 필드가 있는가
    - 근거 문서에 접근할 권한이 있는가
    
    결과는 다음처럼 나눌 수 있습니다.
    
    - Ready: 추천 실행 가능
    - Conditional: 일부 정보가 없지만 제한된 추천 가능
    - Not Ready: 필수 정보가 없어 실행 중단
    - User Input Required: 사용자 보완 필요
    
    중요한 점은 **성공 조건을 플랫폼이 일방적으로 정하지 않고 고객이 설정할 수 있게 하는 것**입니다.

    # 8. 에이전트와 오케스트레이션 구조
    
    ## 8.1 Router와 병렬 실행
    
    기존 그림에서는 Router 뒤에 업무 부하 분석과 업무 추출이 병렬로 연결되어 있었습니다.
    
    하지만 Router는 입력이나 조건에 따라 “어느 경로로 보낼지” 선택할 때 사용합니다. 두 작업을 항상 병렬 실행한다면 Router가 아니라 단순 실행 계획 또는 오케스트레이션입니다.
    
    따라서 다음 중 하나로 명확히 해야 합니다.
    
    - 조건에 따라 하나를 선택한다 → Router
    - 두 작업을 동시에 수행한다 → Parallel execution
    - 정해진 순서로 수행한다 → Workflow orchestration
    
    ## 8.2 오케스트레이터의 책임
    
    오케스트레이터는 직접 분석하거나 판단하는 에이전트가 아니라 다음을 담당합니다.
    
    - 단계 호출
    - 실행 순서 관리
    - 상태 저장
    - 실패·재시도 관리
    - 사용자 승인 대기
    - 재처리 경로 결정
    
    ## 8.3 ReAct가 아니라 Fallback
    
    검증 실패 후 이전 단계로 돌아가는 흐름은 ReAct 구조라기보다 Fallback 또는 Retry flow에 가깝습니다.
    
    - ReAct: 모델이 추론과 행동을 반복하며 다음 도구를 선택
    - Fallback: 특정 단계 실패 시 정해진 대체 경로로 이동
    - Retry: 같은 단계에 수정된 입력을 넣어 재실행
    
    이번 시스템에서는 Fallback·Retry·Human-in-the-loop가 더 정확한 표현입니다.

    # 9. 업무 추출과 생성의 경계
    
    회의 중 아직 완전히 확정되지 않은 중요한 질문입니다.
    
    > 기획서에 적힌 업무를 그대로 추출하는가, 아니면 목표를 보고 새로운 Epic과 Task를 생성하는가?
    > 
    
    두 기능은 평가 기준도 다릅니다.
    
    | 기능 | 의미 | 평가 기준 |
    | --- | --- | --- |
    | 추출 | 문서에 존재하는 업무를 구조화 | 원문 충실도·누락률 |
    | 생성 | 목표·제약을 바탕으로 필요한 업무 설계 | 완전성·실행 가능성·근거 |
    | 보완 | 추출된 업무에 누락 후보 제안 | 기존 내용과의 구분·승인 여부 |
    
    따라서 화면과 데이터 모델에서도 `extracted`, `generated`, `user_added` 등의 출처를 구분하는 것이 좋습니다.

    # 10. 업무 분배 로직과 팀장의 암묵지
    
    팀에서는 팀장 인터뷰가 없어 분배 원칙을 이론과 프롬프트로 구성하려 했습니다. 멘토님은 프롬프트 자체보다 다음이 더 중요한 AI 엔지니어링이라고 설명했습니다.
    
    1. Connector/API Integration
    2. Canonical Data Model
    3. Identity Resolution
    4. Document Parsing·Information Extraction
    5. Retrieval Engineering
    6. Task Graph Engineering
    7. Resource Allocation·Optimization
    8. Policy Engine
    9. Capacity·Scheduling Engine
    10. Evidence·Citation Engineering
    11. Validation·Evaluation Engineering
    12. Workflow Orchestration
    13. Security·Access Control
    
    암묵지는 다음 방법으로 반영할 수 있습니다.
    
    - 사용자가 중요 요소의 가중치 선택
    - 팀·부서별 정책 입력
    - 과거 프로젝트와 배정 이력 검색
    - 승인·수정·반려 결과를 피드백 데이터로 저장
    - “내용은 맞지만 우선순위만 변경”과 “업무 정의 자체가 틀림”을 구분
    - 팀장 유형을 억지로 고정된 페르소나로 만들기보다 실제 과거 사례를 우선 활용
    
    초기 데이터가 없다면 시연용 가상 사례를 만들 수 있지만, 실제 도입 논리는 “기존 Jira·Drive·Notion 등에 이미 축적된 과거 업무 기록을 사용한다”로 설명하는 편이 자연스럽습니다.

    # 11. 결과 일관성 확보 방법
    
    LLM 출력은 완전히 동일하게 고정하기 어렵기 때문에 일관성을 여러 층에서 확보합니다.
    
    - Structured Output으로 스키마 고정
    - 숫자 계산은 LLM이 아닌 일반 함수로 처리
    - 용어 사전과 동의어 매핑
    - 승인된 결과는 다시 생성하지 않고 Snapshot 재사용
    - 프롬프트·모델·정책 버전 기록
    - 동일 입력에 대한 핵심 태스크 누락 여부 평가
    - 추천 후보의 점수 차이가 작으면 “동급 후보”로 표시
    - 여러 번 생성 후 공통 결과를 채택하는 방법은 비용 때문에 선택적으로 사용
    
    여기서 중요한 일관성은 문장 표현이 똑같은지가 아니라 다음입니다.
    
    - 핵심 업무가 반복해서 유지되는가
    - 필수 제약조건을 위반하지 않는가
    - 같은 근거에서 전혀 모순된 결론이 나오지 않는가
    - 변경됐다면 그 이유를 설명할 수 있는가

    # 12. 사람의 승인과 결과 처리
    
    최종 배정 판단은 AI가 아니라 PM·팀장이 수행합니다.
    
    가능한 사용자 행동은 다음과 같습니다.
    
    - 승인: Jira 프로젝트·이슈 생성 또는 배정 확정
    - 수정: 사람이 일부를 수정한 뒤 제약조건 재검증
    - 반려: 사유를 입력하고 재분석 또는 종료
    - 조건부 승인: 일부 데이터가 부족하지만 사람이 책임지고 진행
    
    추천 결과에는 다음을 함께 보여줘야 합니다.
    
    - 추천된 담당자
    - 적합도와 추천 이유
    - 현재 부하와 가용시간
    - 충족·미충족 스킬
    - 적용된 정책과 제약조건
    - 근거 문서
    - 데이터 기준 시점
    - 검증 경고
    - 대체 후보
