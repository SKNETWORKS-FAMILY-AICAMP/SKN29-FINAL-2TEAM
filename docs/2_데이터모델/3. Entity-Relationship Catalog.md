# Canonical Entity-Relationship Catalog

> 기준 문서: `기획서_v5_AI프로젝트운영코파일럿.md`  
> 기준 시각화: Figma Page 1~3  
> 목적: Google Drive, People DB, Jira에서 들어오는 서로 다른 데이터를 하나의 분석 모델로 통일하고, 업무 추출·업무량 계산·담당자 추천·검증 결과의 계보를 보존한다.

---

## 0. 설계 기준

1. 인력 기준 데이터는 별도 Workday 호환 API가 아니라 내부 PostgreSQL **People DB**로 준비한다.
2. Jira는 기존 프로젝트의 업무량·일정·이력 조회에 사용한다.
3. Google Drive 문서는 전체 범위를 구조화해 **Project Knowledge Model**을 만든다.
4. Vector DB와 RAG는 유사 표현 통합, 과거 Jira 검색, 추가 근거 조회를 보조한다. 신규 프로젝트의 전체 Task 범위를 Vector Top-K 검색만으로 만들지 않는다.
5. 별도 Graph DB는 사용하지 않는다. 관계는 PostgreSQL의 PK·FK·조인 테이블로 표현한다.
6. 분석 실행 순서는 `원천 검증 → Canonical 정규화 → Identity Mapping → AnalysisSnapshot → Feature Readiness → Agent·계산 실행`으로 고정한다.
7. 계산값은 규칙·수식 기반 코드가 생성한다. Agent는 계산 결과를 비교·조합하고 설명한다.
8. 모든 추천은 원문 근거, 누락 데이터, 제한 사항, 가정, 신뢰도, 검증 상태를 함께 가진다.
9. 문서 처리 계층의 **물리 테이블명 기준(SoT)** 은 「비정형 문서 파싱·청킹 공통 스키마 설계」로 하고, 본 카탈로그의 Entity명은 논리 뷰다(2.1 매핑표 참조).

## 1. 공통 규칙

### 1.1 공통 식별자

| 항목 | 규칙 |
|---|---|
| 내부 PK | UUID |
| 외부 ID | 원본 문자열을 별도 필드로 보존 |
| 시간 | UTC 저장, 화면에서 프로젝트 또는 사용자 시간대로 변환 |
| 버전 | 원본 버전·내용 해시·수정 시각을 함께 보존 |
| 삭제 | 분석 계보가 필요한 객체는 물리 삭제보다 `is_active`, `deleted_at` 사용 |
| Tenant | 모든 주요 테이블에 `tenant_id` 적용 |
| 프로젝트 범위 | 문서·지식·Task·분석 결과는 `project_id`로 격리 |

### 1.2 공통 출처 유형

| 코드 | 의미 |
|---|---|
| EXTRACTED | 원문에서 직접 추출 |
| GENERATED | 원문 근거를 조합해 생성 |
| AI_SUGGESTED_MISSING_TASK | AI가 보완 제안 |
| USER_ADDED | PM이 직접 추가 |

### 1.3 데이터 준비 상태

| 상태 | 의미 | 처리 |
|---|---|---|
| SUCCESS | 필수 데이터가 충분함 | 전체 기능 실행 |
| PARTIAL_RESULT | 선택 데이터 또는 일부 매핑 누락 | 제한·가정과 함께 실행 |
| BLOCKED | 필수 데이터 또는 조건부 필수 데이터 누락 | 해당 기능 실행 중단 |

## 2. 프로젝트·문서·지식 모델

### 2.1 원천 및 파싱

| Entity | 업무적 의미 | 핵심 필드 |

| DOCUMENT_SYNC_STATE | 파일별 증분 동기화 상태 | `sync_state_id`, `document_id`, `change_type`, `sync_status`, `checkpoint_token`, `retry_count`, `last_synced_at` |
| DOCUMENT_BLOCK | 제목·문단·표·목록 등 의미 보존 원문 단위 | `block_id`, `document_id`, `block_type`, `page`, `section_path`, `content`, `content_hash`, `block_order` |

| 본 카탈로그(논리) | 물리 테이블(SoT) | 수집 보고서 표기 |

| DOCUMENT | `source_document` + `document_revision` | `document_index/chunks` 원천부 |
| DOCUMENT_SYNC_STATE | `document_index`(indexStatus·isActiveRevision) | `processing_runs` |
| DOCUMENT_BLOCK | `content_block_manifest` | document_index/chunks 블록부 |

| (근거) | `citation_ref` | — |

원천 수집 레지스트리는 세 문서 공통으로 `source_registry`를 사용한다.

Snapshot은 Graph DB 대체재가 아니다. 분석 중 원천 데이터가 바뀌어도 동일한 입력·결과를 다시 설명하기 위한 **재현성과 감사 단위**다.

### 5.1 업무량 계산

```text
총 근무용량 = 기간 내 근무일 × 일 기준 근무시간 × FTE
유효 가용용량 = 총 근무용량 - 휴가·부재 - 정책상 제외시간
현재 할당량 = Jira 기간 중 기존 잔여 공수
예상 할당량 = 현재 할당량 + 신규 배정 공수
예상 부하율 = 예상 할당량 / 유효 가용용량
잔여 가용시간 = 유효 가용용량 - 예상 할당량
```

- 유효 가용용량이 0이면 시간 기반 계산은 `BLOCKED`다.
- 예상 부하율이 100%를 초과하면 과부하 위험으로 표시한다.
- 시간 데이터가 없고 Story Point·Sprint·Velocity가 충분한 경우에만 Sprint 방식으로 대체한다.

### 5.2 후보 처리 경계

1. 규칙 기반 후보 필터가 재직·역할·책임수준·조직·권한 Hard Constraint를 적용한다.
2. 점수 코드가 Skill·역할·유사 경험 지표를 계산한다.
3. 업무량 계산 모듈이 기간별 가용성과 부하율을 계산한다.
4. 위험 분석 코드가 충돌·집중·병목을 계산한다.
5. 업무 분배 Agent가 위 결과를 비교해 주 후보·대안 후보·근거를 구성한다.
6. 검증 Agent가 체크리스트를 실행한다.

Skill 데이터가 없으면 불충족으로 단정하지 않고 `UNKNOWN` 조건부 후보로 처리한다.

## 7. Identity Mapping

매칭 우선순위:

1. 관리자가 확정한 `manual_override` 매핑
2. 동일 외부 계정 ID
3. 정규화한 회사 이메일 완전 일치
4. 사번 또는 별도 관리 ID 일치
5. 이름+조직+직무 조합 후보
6. 미확정 상태로 PM 또는 관리자 확인

동명이인·다중 후보·충돌은 자동 병합하지 않는다. Jira `accountId` 매핑 실패는 전체 실행 오류가 아니라 해당 직원의 Jira 업무량을 확인할 수 없다는 `PARTIAL_RESULT` 근거로 남긴다. 단, 고객 정책이 Jira 업무량을 필수로 설정한 기능에서는 해당 후보를 제외하거나 `BLOCKED`할 수 있다.
