# 전체 데이터 명세 — 21문답

> 기술 구성을 빠르게 이해하기 위한 쉬운 설명이다. 상세 필드와 관계는 `3. Entity-Relationship Catalog.md`, 순서는 `4. Data Flow.md`, 물리 관계는 `6. ERD.md`를 따른다.

---

## 1. 이 시스템이 해결하는 문제는 무엇인가?

신규 프로젝트 문서를 읽어 실행할 업무를 만들고, 현재 인력의 역할·Skill·일정·업무량을 함께 비교해 PM에게 담당자와 대안 후보를 추천한다.

AI가 최종 배정하지 않는다. PM이 승인·수정·반려한다.

---

## 2. 데이터는 어디에서 들어오는가?

세 곳이 핵심이다.

- Google Drive: 신규 프로젝트 기획서·요구사항·명세·회의록
- People DB: 직원·조직·역할·Skill·근무시간·휴가
- Jira: 기존 프로젝트의 담당자·상태·일정·잔여 공수

Google Calendar는 후속 선택 데이터다.

---

## 3. People DB는 Workday API인가?

아니다. 1차 구현에서는 내부 PostgreSQL에 People DB 테이블과 샘플 데이터를 만든다. Workday에서 일반적으로 필요한 직원·조직·직무·Skill·근무·부재 개념을 참고하되, 실제 Workday API와 같다고 표현하지 않는다.

테이블명에도 `MOCK`을 넣지 않는다. 목업 여부는 실행 환경과 데이터셋 설명으로 구분한다.

---

## 4. Google Drive 파일은 어떻게 읽는가?

```text
프로젝트 폴더 선택
→ 파일 목록·권한·수정 시각 확인
→ 원본 다운로드
→ PDF/DOCX 파싱
→ DocumentBlock 생성
→ 검색용 ChildChunk 생성
→ 의미 유형 추출
```

8월 6일 시연에서는 PDF와 DOCX가 우선이다.

---

## 5. 문서를 왜 Block과 Chunk로 두 번 나누는가?

Block은 제목·문단·표 같은 원문의 구조를 보존한다. Chunk는 긴 Block을 의미 검색하기 좋은 길이로 나눈다.

- 정보 추출과 원문 근거: `DOCUMENT_BLOCK`
- 임베딩과 검색: `CHILD_CHUNK`

Chunk만 남기면 문서 전체 구조와 페이지·섹션 계보를 잃기 쉽다.

---

## 6. Vector DB가 전체 분석의 중심인가?

아니다. 전체 프로젝트 업무를 만들 때는 활성 DocumentBlock 전체를 의미 유형별로 구조화한다.

Vector DB는 다음에 보조로 쓴다.

- 서로 다르게 표현된 같은 요구사항 찾기
- 과거 유사 Jira 업무 찾기
- 추천 근거를 추가로 조회하기
- PM의 근거 질문에 답하기

Top-K 검색만으로 프로젝트 전체 Task를 만들지 않는다.

---

## 7. Project Knowledge Model은 무엇인가?

여러 문서에 흩어진 내용을 프로젝트 단위로 합친 구조다.

- 목표
- 범위
- 요구사항
- 결정
- 제약
- 일정·마일스톤
- 의존관계
- 리스크
- 완료 조건
- 담당 역할
- 원문 근거

중복 표현을 합치고 상충하는 내용과 최신 버전을 표시한다. 업무 추출 Agent는 검색 청크 몇 개가 아니라 이 모델을 기준으로 Task를 만든다.

---

## 8. 새 파일이 폴더에 추가되면 어떻게 아는가?

초기 연결 때 파일 목록과 Drive Changes API 시작 토큰을 저장한다. 이후 Changes API를 조회해 신규·수정·삭제·권한 상실을 확인한다.

- 신규: 새 문서·Block·Chunk 생성
- 수정: 새 버전과 변경 Block 생성
- 삭제: 기존 계보를 남기고 비활성화
- 권한 상실: 검색·분석 대상에서 제외

---

## 9. Jira 데이터는 어떻게 통일하는가?

회사마다 이슈 필드가 달라서 최초 연결 때 `/field`, `/project/search`, `/users/search`, `/board`를 조회한다. 그 결과로 Tenant별 Field Mapping을 만든 뒤 Jira 이슈를 `EXISTING_TASK`로 변환한다.

---

## 10. Jira의 비정형 설명과 댓글은 어떻게 하는가?

업무량 계산에는 구조화된 `EXISTING_TASK` 필드를 사용한다. 설명·댓글을 의미 검색에 쓸 때는 Jira 이슈를 가상 `DOCUMENT`로 보고 `DOCUMENT_BLOCK`과 `VECTOR_INDEX`를 만든다.

즉 계산용 데이터와 검색용 텍스트를 분리한다.

---

## 11. 사용자 매핑은 왜 필요한가?

People DB의 직원 ID와 Jira `accountId`, Google 계정 ID는 서로 다르다. `IDENTITY_LINK`가 이들을 한 사람으로 연결한다.

우선순위:

1. 관리자가 확정한 매핑
2. 외부 ID
3. 회사 이메일
4. 사번
5. 이름+조직+직무 후보
6. 관리자 확인

동명이인은 자동 병합하지 않는다.

---

## 12. Jira accountId 매핑에 실패하면 전체가 멈추는가?

기본적으로 멈추지 않는다. 해당 직원의 Jira 업무량을 확인할 수 없다는 `PARTIAL_RESULT`를 표시한다.

단, 고객이 “Jira 업무량 확인 필수” 정책을 켠 기능에서는 그 후보를 제외하거나 `BLOCKED`할 수 있다.

---

## 13. Snapshot은 왜 필요한가?

추천을 실행하는 동안 Jira 이슈나 휴가 정보가 바뀔 수 있다. 원천을 매번 실시간 조회하면 같은 추천을 다시 설명하기 어렵다.

Snapshot은 “이 추천이 어떤 시점의 문서·인력·업무·정책으로 만들어졌는지”를 고정한다. Graph DB가 없어도 재현성과 감사 때문에 필요하다.

---

## 14. Feature Readiness는 언제 검사하는가?

원천 검증과 Canonical 변환을 끝내고 Snapshot을 만든 다음 검사한다.

```text
원천 검증
→ Canonical 정규화
→ Identity Mapping
→ AnalysisSnapshot
→ Feature Readiness
→ Agent·계산 실행
```

원천 레코드를 바로 보고 추천 가능 여부를 결정하지 않는다.

---

## 15. SUCCESS, PARTIAL_RESULT, BLOCKED는 어떻게 다른가?

- `SUCCESS`: 필요한 데이터가 충분하다.
- `PARTIAL_RESULT`: 실행은 가능하지만 선택 데이터·매핑·일부 업무량이 빠졌다.
- `BLOCKED`: 필수 데이터가 없어 해당 기능을 실행할 수 없다.

모든 결과에는 누락 데이터, 제한 사항, 가정, 신뢰도, 실행 가능한 기능을 표시한다.

---

## 16. 업무량은 누가 계산하는가?

Agent가 아니라 규칙·수식 기반 계산 모듈이 계산한다.

```text
총 근무용량 = 기준 근무시간 × FTE
유효 가용용량 = 총 근무용량 - 휴가·부재 - 정책상 제외시간
예상 할당량 = Jira 기존 잔여 공수 + 신규 배정 공수
예상 부하율 = 예상 할당량 / 유효 가용용량
```

유효 가용용량이 0이면 시간 기반 계산은 `BLOCKED`다.

---

## 17. 업무 분배 Agent는 무엇을 하는가?

코드가 만든 적격 후보·적합도·부하율·위험 지표를 비교해 프로젝트 전체 관점의 배분안을 구성한다.

출력:

- 추천 담당자
- 최소 한 명의 대안 후보
- 추천 근거
- 적합도와 업무 부하
- 충돌·집중·병목 위험
- 누락 데이터·가정·신뢰도

계산값을 임의로 만들거나 바꾸지 않는다.

---

## 18. Skill 데이터가 없으면 부적합인가?

아니다. Skill이 없다는 사실과 Skill 데이터가 없다는 사실은 다르다.

Skill 데이터가 없으면 `UNKNOWN` 조건부 후보로 처리하고 PM에게 확인을 요청한다. 명시적으로 필수 Skill 불충족이 확인된 경우에만 제외 또는 REJECT한다.

---

## 19. 검증 Agent는 무엇을 검사하는가?

체크리스트를 실행한다.

- 역할·필수 Skill·책임수준
- 조직 제한
- 권한
- 휴가·휴직 중복
- 일정 충돌
- 과부하
- 추천 근거
- 누락 데이터와 신뢰도

`PASS`와 `CONDITIONAL_PASS`는 PM이 검토할 수 있다. `REJECT`는 Jira로 보낼 수 없다.

---

## 20. PM이 수정하면 처음부터 다시 실행하는가?

수정 이유에 따라 돌아가는 위치가 다르다.

- Task 분해 오류: 업무 추출
- 인력·조건 변경: 업무 분배
- 데이터 누락·오류: Connector/Data
- 검증 조건만 변경: 검증 Agent
- 후보·일정·공수 수정: 검증 Agent로 직접 재검증
